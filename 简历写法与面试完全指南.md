**Q1：你的秒杀系统怎么防止超卖的？（80%必问）**

> 答题思路：分层讲，每层说清职责

```
三层防御：

第一层 —— 动态令牌桶限流（流量入口）：
  根据Redis库存动态调整限流速率，库存1000时只放1500 QPS进来
  把无效请求在入口就挡掉，Redis和DB不做无效计算

第二层 —— Redis Lua脚本原子扣库存（核心）：
  check库存>0 + check用户限购 + DECR库存，三步合一原子执行
  Redis单线程，不存在并发问题，这层彻底防超卖

第三层 —— DB CAS兜底（安全网）：
  UPDATE ... SET STOCK=STOCK-1 WHERE STOCK>0
  affected=0代表没库存了，防止Redis与DB不一致时的极端超卖
  
设计原则：每层各司其职，去掉任何一层都有风险。
```

**Q2：Redis和DB库存不一致怎么处理？（70%必问）**

```
两种场景处理不同：

场景A（Redis=0，DB>0）：
  Kafka消费失败回滚了DB库存，但Redis还没补回来
  → 不直接返回"已售罄"，返回"活动火爆，稍后重试"
  → 用户刷新后Redis回滚已完成，可以继续抢
  → 这就是我设计的"智能库存判断"，提升了库存利用率到100%

场景B（Redis>0，DB=0）：
  正常不应出现，DB CAS兜底：UPDATE WHERE STOCK>0 → affected=0 → 失败
  回滚Redis库存，下次请求同样失败

定期对账：
  每5分钟Job对比Redis和DB库存，差值超过阈值告警，人工介入
```

**Q3：Redisson锁和DB CAS都用了，是不是过度设计？（刁难题！）**

```
不冗余，两层职责完全不同：

Redisson锁的作用：
  针对"同一个码被并发请求"（用户双击、网络重试打到不同实例）
  让同一个码同时只有1个线程进入业务逻辑
  作用：减少DB CAS冲突率，保护DB压力

DB CAS的作用：
  针对"锁失效的极端情况"（Redis宕机、锁实现BUG）
  数据库行锁天然防并发，不依赖外部组件
  作用：最终数据正确性的保证

如果只有锁没有CAS：Redis宕机时重复兑换
如果只有CAS没有锁：所有并发请求打到DB，冲突率高性能差

两层是"快失败（锁）+ 终极兜底（CAS）"的组合，不是冗余。
```

**Q4：HMAC签名能防所有攻击吗？（考察诚实度）**

```
诚实回答，分层说：

HMAC能防的：
✅ 暴力枚举/猜码（签名4位共167万种，配合IP限速，破解需2799小时）
✅ 伪造新码（无私钥无法生成合法签名）

HMAC不能防的：
❌ 重放真实码 → 这是DB CAS的职责
❌ 内部人员泄露 → 属于管理问题
❌ 密钥泄露后批量伪造 → 需要密钥版本轮换机制（这是我规划中的P1需求）

结论：系统是分层设计，格式合法性靠HMAC，重复使用靠DB CAS，
      两层职责不同，HMAC只负责它该负责的，不需要万能。
```

**Q5：为什么不用SELECT FOR UPDATE，而用CAS？**

```
SELECT FOR UPDATE是悲观锁：
  先SELECT加行锁 → 判断状态 → UPDATE → COMMIT解锁
  高并发下后续请求全部排队等待，串行化，性能差
  异常时可能死锁（锁未释放）

CAS是乐观锁：
  UPDATE ... WHERE STATUS=0（一条SQL，数据库原子执行）
  affected=1成功，affected=0失败，直接返回，不阻塞其他请求
  无等待、无死锁、无额外锁开销

兑换码场景STATUS只有0→1一次跳转，
CAS冲突率极低，几乎所有请求第一次就成功，
所以CAS是最优选择。
```

**Q6：Kafka消费失败怎么办？（分布式系统必问）**

```
三级重试+死信：

第1次失败 → 立即重试（应对网络抖动）
第2次失败 → 延迟5秒重试（应对DB短暂不可用）
第3次失败 → 进入死信Topic（seckill-order-dlt）

死信队列处理：
  1. 持久化到DB（不丢失）
  2. 发送告警通知
  3. 人工补偿或定时重试

保证幂等（防止重复消费发双份券）：
  消费前先查DB订单状态
  status=1（已成功）→ 直接跳过
  没有这个幂等检查，重试会导致重复发券
```

**Q7：动态限流怎么实现的？（加分题）**

```
背景：固定限流浪费资源
  库存100张时，还允许10000 QPS进来
  9900个请求做无效Lua计算，Redis白白消耗

实现（DynamicRateLimiterManager）：
  1. 每5秒定时任务读取Redis库存
  2. 新速率 = 剩余库存 × 1.5（留缓冲防止回滚时速率过低）
  3. 调用 rateLimiter.setRate(newRate) 动态调整

效果：
  库存10000 → 放15000 QPS
  库存100   → 放150 QPS
  库存0     → 放20 QPS（只允许智能判断逻辑通过）
  
  Redis无效Lua执行↓87.5%，大幅降低Redis计算压力
```

**Q8：支付系统怎么保证幂等性？（90%必问）**

```
三层防护：

第一层 —— Redis分布式锁（毫秒级拦截）：
  前端生成UUID作为幂等Token，请求进来先在Redis抢锁
  同一Token的并发请求，只有第一个能进入业务
  适合：高并发瞬间的并发请求去重

第二层 —— DB唯一索引（持久化保证）：
  IDEMPOTENCY_KEY字段加UNIQUE约束
  即使第一层失效，DB写入时唯一约束保证不会有两条相同记录
  适合：跨实例、跨时间段的幂等

第三层 —— 乐观锁+终态检查（状态机保护）：
  处理回调时先检查当前状态：SUCCESS/FAILED → 直接返回，不再处理
  UPDATE ... WHERE STATUS='PENDING' AND VERSION=?
  终态不可逆，防止重复回调把成功状态改成失败

三层各有适用场景，缺任何一层都有漏洞。
```

**Q9：TIMEOUT死锁场景怎么解决的？（PaymentSys核心亮点）**

```
问题背景：
  客户在银行端看到"支付成功"，系统显示"TIMEOUT"
  银行钱已转出，但订单失败，财务没有入账记录
  → 死账：客户告投诉，公司追不到款

根本原因：
  系统在银行发送回调确认前就把交易标记TIMEOUT
  + 没有主动查询机制，只靠银行推送回调

解决方案（双管齐下）：

方案1 —— 对账补偿Job（主动查询）：
  定时Job扫描所有TIMEOUT/PENDING状态超过X分钟的订单
  主动调用银行查询接口：queryPaymentStatus(transactionId)
  银行返回SUCCESS → 更新���地为SUCCESS，触发发货/发券
  银行返回FAILED → 更新为FAILED，释放库存

方案2 —— 状态机终态保护：
  SUCCESS和FAILED是终态，不可逆转
  回调回来说SUCCESS，但本地已经是SUCCESS → 直接返回，不重复处理
  防止因为状态乱序回调把成功变成失败

效果：消灭了所有"有钱没单"死账，支付最终一致性100%达成。
```

**Q10：乐观锁和悲观锁在支付场景怎么选？（考察理解深度）**

```
支付系统的特点：
  同一笔订单的并发回调很少（银行通常只发1-3次）
  但一旦并发冲突后果严重（重复扣款/重复退款）

选乐观锁（VERSION）：
  普通回调处理：UPDATE ... WHERE VERSION=? AND STATUS='PENDING'
  冲突概率低，无锁等待，性能好
  冲突时：放弃本次处理（对方已处理成功），返回200告诉银行已收到

选悲观锁（SELECT FOR UPDATE）：
  退款场景：金额大、操作不可逆
  UPDATE PAYMENT_TRANSACTION ... FOR UPDATE → 确保串行处理退款

原则：
  读多写少 + 冲突概率低 → 乐观锁
  写操作不可逆 + 金额大 → 悲观锁做额外保护
  两种锁结合用，不是非此即彼
```

**Q12：支付系统的回调为什么需要Redis去重，DB唯一索引不够吗？**

```
DB唯一索引的问题：
  银行1秒内发送3次相同回调（重试策略）
  第1个请求进入：SELECT查状态 → 判断PENDING → 准备UPDATE
  第2个请求进入：SELECT查状态 → 判断PENDING（第1个还没提交）
  第2个请求也开始UPDATE
  → 两个请求都通过了状态检查，可能导致重复处理
  （这是DB唯一索引防不住的并发窗口问题）

Redis去重的作用：
  回调进来，先 SET callback:txnId NX EX 60（60秒内去重）
  第1个回调：SET成功 → 继续处理
  第2个回调：SET失败 → 直接返回200（已在处理中）
  → 毫秒级拦截，完全不进入DB层

两者组合：
  Redis：毫秒级，防止并发窗口问题
  DB唯一索引：持久化兜底，防止Redis宕机时的漏网之鱼
```

### 刁难题1：你说的HMAC防枚举，如果攻击者控制了10万台肉鸡呢？

```
承认限制，说出应对：

最终防线：即使破解了签名格式，每个有效码也只能兑换一次（DB CAS）
攻击者只能拿到已使用的码，没有收益，攻击成本远大于收益

这是安全设计的核心思想：让攻击的ROI为负数。
```

### 刁难题2：你的Lua脚本库存扣到-1怎么办？

```
Lua脚本里有严格检查，不会扣到-1：

local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock <= 0 then
    return -1  -- 无库存，直接返回失败
end
-- 只有stock > 0才执行DECR
redis.call('DECR', KEYS[1])
return 1

如果因为BUG真的到了-1：
  DECR后值 < 0 → 额外检查，立即INCR回来+告警
  DB CAS兜底：UPDATE WHERE STOCK>0，affected=0也会阻止
  对账Job：发现Redis库存 < 0，立即告警重置
```

### 刁难题3：K8s扩容30秒，这30秒内积压的Kafka消息怎么办？

```
这30秒Kafka消息只是积压，不会丢失：

Kafka消息持久化在磁盘：
  消费者Group的offset记录了消费到哪里
  新Pod启动后，从上次offset继续消费
  没有处理的消息全部在Kafka里等待，不会丢

对用户的影响：
  抢购成功后，用户看到"处理中"状态
  30-60秒后（新Pod消费完积压）状态变成"成功"
  用户端轮询订单状态，最终看到结果

关键前提：
  消费者要保证幂等（重复消费不会重复发券）
  所以积压期间即使有重复消费也安全

这就是Kafka异步化的价值：削峰时短暂延迟是可以接受的，数据不丢。
```

### 刁难题4：支付系统VERSION乐观锁，如果并发100个回调冲突，只有1个成功，其他99个怎么办？

```
其他99个回调直接返回200给银行：

处理逻辑：
  UPDATE WHERE VERSION=? → affected=0 → 说明有人先处理了
  查询当前状态：已经是SUCCESS → 说明处理成功了
  直接返回成功响应给银行（银行不会再重试）

为什么不重试：
  银行回调的目的是"通知支付状态"
  如果已经有人处理成功了，目的已达到，不需要再处理
  重试只会浪费资源，不增加价值

如果查询到状态还是PENDING：
  说明第一个请求正在处理中（时间窗口）
  此时可以短暂等待后重试一次（最多1次）
  或者直接返回200，依靠对账Job补偿

关键点：99个"失败"不是真的失败，是"让位给第一个成功者"。
```

### 刁难题5：你的系统怎么保证对账绝对准确？

```
诚实回答：没有绝对准确，说对账的边界和补偿。

对账流程：
  1. 每天凌晨定时Job：从DB捞TIMEOUT/PENDING状态订单
  2. 批量调用银行查询接口
  3. 比对：本地状态 vs 银行状态
     - 银行SUCCESS，本地TIMEOUT → 补偿为SUCCESS
     - 银行FAILED，本地PENDING → 更新为FAILED
     - 银行无记录 → 说明银行没有收到请求，可以重试或关单

对账可能的漏洞：
  银行查询接口也可能超时 → 记录为"待核查"，下次继续
  数据量大时对账Job可能跑很慢 → 分批处理，每批200条
  极端情况：银行数据也不准 → 人工对账+财务确认（最后防线）

核心思想：系统保证T+1天之内所有订单状态最终一致，
          无法保证实时一致，但保证最终一致。
```

## 六、面试结尾总结（背下来！）

### CouponSys 30秒总结：

> 这个系统我认为有三个核心亮点：
> 第一，**分层防超卖**：从动态限流到Lua原子扣减到Kafka落库到DB CAS，每层各司其职，即使某层失效下层兜底；
> 第二，**HMAC本地验签**：服务端不查DB就能拦截99%的伪造码，让攻击流量在入口就消失；
> 第三，**精细化边界处理**：比如Kafka回滚后的"幽灵售罄"问题，分布式锁和CAS的微妙职责分工——这些细节体现了对分布式系统边界场景的深入理解。

### PaymentSys 30秒总结：

> 这个系统我认为有三个核心价值：
> 第一，**解决了真实的业务痛点**：TIMEOUT死锁是真实生产问题，通过主动对账+终态机彻底消灭了"有钱没单"死账；
> 第二，**三层幂等防护**：Redis锁+DB索引+乐观锁，将重复扣款概率从5%降至0.001%；
> 第三，**设计的扩展性**：策略模式路由器让新增支付渠道从3天降到半天，体现了对扩展性和开闭原则的实践。
