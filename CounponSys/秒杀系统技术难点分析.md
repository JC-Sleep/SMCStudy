## 二、技术难点与解决方案

### 难点1: 高并发下的库存超卖

**问题**: 10万人同时抢500张券，如何保证不超卖?

**错误示范**:
```java
// ❌ 这样会超卖!!!
int stock = redis.get("stock");
if (stock > 0) {
    redis.decr("stock");  // 多个线程同时执行，都能进来
    // 发券...
}
```

**正确方案**: Lua脚本原子操作
```lua
-- ✅ 原子操作，不会超卖
local stock = redis.call('GET', KEYS[1])
if tonumber(stock) <= 0 then return -1 end
redis.call('DECR', KEYS[1])
return 1
```

---

### 难点2: 重复抢购

**问题**: 用户快速点击多次，如何防止重复抢购?

**解决方案**: 三重保障
1. **前端**: 点击后按钮禁用3秒
2. **Redis**: Lua脚本中检查用户已抢次数
3. **数据库**: 唯一索引 `UK_USER_ACTIVITY(user_id, activity_id)`

```lua
-- Lua脚本中检查
local grabbed = redis.call('GET', userKey)
if tonumber(grabbed or 0) >= limit then return -2 end
redis.call('INCR', userKey)
```

---

### 难点3: Redis与数据库数据一致性

**问题**: Redis扣了库存，但数据库写入失败怎么办?

**解决方案**: 最终一致性
```
Redis扣库存(快) → 返回用户成功 → MQ异步 → 数据库扣库存(慢)
                                        ↓
                                   如果失败
                                        ↓
                              回滚Redis库存 + 标记订单失败
```

**代码实现**:
```java
try {
    // 1. 扣积分
    pointsService.deductPoints(...);
    // 2. 发券
    couponService.grantCoupon(...);
    // 3. 扣数据库库存
    activityMapper.deductStock(...);
    // 4. 更新订单成功
    order.setStatus(1);
} catch (Exception e) {
    // 回滚Redis库存
    redisTemplate.opsForValue().increment(stockKey);
    // 回滚积分
    pointsService.rollbackPoints(...);
    // 标记订单失败
    order.setStatus(2);
    order.setFailReason(e.getMessage());
}
```

### 难点4: 消息堆积

**问题**: 10万QPS，Kafka消费者处理不过来怎么办?

**解决方案**:
1. **增加分区**: Topic设置10个分区
2. **增加消费者**: 10个消费者实例并行消费
3. **批量处理**: 每次拉取500条消息批量处理
4. **限流保护**: 消费端也加限流，保护数据库

```yaml
spring:
  kafka:
    consumer:
      max-poll-records: 500  # 批量拉取
    listener:
      concurrency: 10        # 10个并发消费者
```

### 难点5: 缓存穿透/击穿

**问题**: 活动不存在或缓存过期，大量请求打到数据库

**解决方案**:
1. **缓存空值**: 活动不存在也缓存(设置短过期时间)
2. **预热**: 活动开始前提前加载到Redis
3. **互斥锁**: 缓存失效时，只允许一个请求查库

```java
// 缓存击穿防护
public SeckillActivity getActivity(Long id) {
    // 1. 查缓存
    SeckillActivity activity = redis.get(key);
    if (activity != null) return activity;
    
    // 2. 加锁，只有一个请求能查库
    RLock lock = redisson.getLock("lock:activity:" + id);
    try {
        lock.lock();
        // 双重检查
        activity = redis.get(key);
        if (activity != null) return activity;
        
        // 查库并缓存
        activity = db.selectById(id);
        redis.set(key, activity, 1, TimeUnit.HOURS);
        return activity;
    } finally {
        lock.unlock();
    }
}
```

### 难点6: 热点Key

**问题**: 所有人抢同一个活动，`seckill:stock:1001`成为热点Key

**解决方案**: 库存分片
```
原来: seckill:stock:1001 = 500

分片后:
seckill:stock:1001:0 = 100
seckill:stock:1001:1 = 100
seckill:stock:1001:2 = 100
seckill:stock:1001:3 = 100
seckill:stock:1001:4 = 100

用户请求时: hash(userId) % 5 → 决定去哪个分片扣
```
