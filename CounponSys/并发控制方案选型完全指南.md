# 并发控制方案选型完全指南

> 2026-04-21 | 从本系统真实案例出发，彻底搞清楚：
> **synchronized / ReentrantLock / Redis Lua / Redisson分布式锁 / DB CAS**
> 各自用在哪里、为什么、不用会怎样

---

## 一、先建立一个判断框架

```
面对并发问题，先问三个问题：

Q1：操作是在单台JVM内，还是多台服务器？
    单台JVM  → 可以考虑 synchronized / ReentrantLock
    多台服务器 → 只能考虑 Redis锁 / DB锁（JVM锁对其他实例无效）

Q2：需要保护的是"内存数据"还是"DB数据"还是"Redis数据"？
    保护内存数据（本地缓存、计数器）→ JVM锁
    保护Redis数据（库存、限速计数）→ Lua脚本（Redis单线程原子）
    保护DB数据（订单状态、码状态）→ DB CAS / 悲观锁

Q3：操作是"简单的状态检查+修改"，还是"复杂的多步业务"？
    简单状态跳转（0→1）→ CAS（一条SQL搞定）
    复杂多步（查库存+扣积分+发券）→ 分布式锁（锁住整个流程）
```

---

## 二、五种方案一览表

| 方案 | 作用范围 | 适用场景 | 关键特点 |
|------|---------|---------|---------|
| `synchronized` | 单JVM内 | 多线程共享本地对象 | 简单，但出了JVM就完全失效 |
| `ReentrantLock` | 单JVM内 | 需要tryLock/超时的本地锁 | 比synchronized灵活，仍单机 |
| **Redis Lua脚本** | Redis单实例 | 原子操作Redis数据（库存扣减、限速计数） | Redis单线程保证原子，无锁等待 |
| **Redisson分布式锁** | 跨JVM/跨服务器 | 保护完整业务流程，防止并发重入 | 锁住整个代码块，适合复杂流程 |
| **DB CAS** | 数据库行级 | 简单状态机跳转（只改一个字段） | 无需外部依赖，DB行锁天然原子 |

## 三、核心区别：用时序图说清楚

### 场景1：synchronized — 只管单台JVM内

```
┌─────── 单台服务器 ──────┐
│                        │
│  线程A ──→ synchronized │
│           [锁住方法]    │
│  线程B ──→ 等待...      │
│                        │
└────────────────────────┘

问题：另一台服务器的线程C完全不受影响！
```

**时序图（本系统有3个实例，synchronized完全失效）：**

```
用户甲                 实例A                    用户乙                 实例B
  │                     │                         │                     │
  │── 兑换码CP-001 ──→  │                         │                     │
  │                     │ synchronized lock ✅    │                     │
  │                     │ 查询STATUS=0            │                     │
  │                     │                         │── 兑换码CP-001 ──→  │
  │                     │                         │                     │ synchronized lock ✅
  │                     │                         │                     │ 查询STATUS=0 (!)
  │                     │ 更新STATUS=1            │                     │
  │                     │ 发券 ✅                 │                     │ 更新STATUS=1 ← 重复！
  │←── 兑换成功 ────────│                         │                     │ 发券 ✅ ← 重复！
  │                     │                         │←── 兑换成功 ────────│
  │                     │                         │                     │

结果：一张码被兑换两次！synchronized对另一台实例无效！
```

**结论：有多台服务器时，synchronized根本不能用于分布式场景。**

---

### 场景2：Redisson分布式锁 — 跨JVM互斥整个流程

```
┌── 实例A ──┐     ┌── Redis ──┐     ┌── 实例B ──┐
│           │     │           │     │           │
│ 线程A     │──→  │ SET lock  │     │ 线程B     │
│ tryLock   │     │ NX EX 5s  │     │ tryLock   │
│           │     │ ← 成功    │     │           │
│ 执行业务  │     │           │     │           │──→ SET lock NX → 失败（已存在）
│ ...       │     │           │     │ 等待/放弃 │
│ unlock    │──→  │ DEL lock  │     │           │
│           │     │           │     │           │──→ SET lock NX → 成功
│           │     │           │     │ 执行业务  │
└───────────┘     └───────────┘     └───────────┘
```

**时序图（分布式锁保护兑换码完整流程）：**

```
用户甲                 实例A           Redis(锁)        实例B                用户乙
  │                     │                │               │                     │
  │── 兑换CP-001 ──→   │                │               │                     │
  │                     │── tryLock ──→ │               │                     │
  │                     │← 获取成功 ───  │               │                     │
  │                     │ 查DB STATUS=0  │               │  兑换CP-001 ←───── │
  │                     │                │               │── tryLock ────────→ │
  │                     │                │               │← 等待(已被A持有) ── │
  │                     │ UPDATE STATUS  │               │  等待...            │
  │                     │ = 1 (CAS成功)  │               │  等待...            │
  │                     │── unlock ────→ │               │                     │
  │←── 兑换成功 ────────│                │               │← 获取成功 ─────────  │
  │                     │                │               │ 查DB STATUS=1!      │
  │                     │                │               │ 返回"已被使用"      │
  │                     │                │               │─→ 兑换失败 ────────→│

结果：跨实例互斥，码只被兑换一次 ✅
```

**分布式锁的价值**：把"查DB + 业务判断 + 更新DB"这一整段逻辑保护起来，同一时刻全局只有1个线程执行。

### 场景3：DB CAS — 不需要锁，一条SQL搞定状态跳转

```
三个请求同时到达（假设分布式锁没有，直接执行CAS）：

请求A：UPDATE T_COUPON_REDEEM_CODE SET STATUS=1 WHERE CODE='CP-001' AND STATUS=0
请求B：UPDATE T_COUPON_REDEEM_CODE SET STATUS=1 WHERE CODE='CP-001' AND STATUS=0
请求C：UPDATE T_COUPON_REDEEM_CODE SET STATUS=1 WHERE CODE='CP-001' AND STATUS=0

Oracle内部执行（行锁串行）：
  → 请求A先到：STATUS=0 ✅ 更新成功，affected=1，STATUS变为1
  → 请求B：STATUS已经是1了，WHERE STATUS=0 不满足，affected=0 ❌
  → 请求C：同B，affected=0 ❌

结果：无论多少并发，只有1个请求能成功 ✅
```

**时序图（CAS为什么能防重复兑换）：**

```
请求A      请求B      请求C          Oracle DB
  │          │          │               │
  │          │          │               │
  ├─ UPDATE WHERE STATUS=0 ────────────→│
  │          │          │               │ [行锁] 串行执行
  │          │          │               │ 检查: STATUS=0? ✅
  │          │          │               │ 执行UPDATE → STATUS=1
  │←─ affected=1 成功 ──────────────────│
  │          │          │               │
  │          ├─ UPDATE WHERE STATUS=0 ─→│
  │          │          │               │ 检查: STATUS=0? ❌ (已是1)
  │          │←─ affected=0 失败 ────── │
  │          │          │               │
  │          │          ├─ UPDATE WHERE STATUS=0 ─→│
  │          │          │               │ 检查: STATUS=0? ❌ (已是1)
  │          │          │←─ affected=0 失败 ───────│

只有A成功 ✅ B和C都失败 ❌ — 这就是CAS
```

**CAS的局限**：只能保证"最终数据正确"，但在CAS之前的"查询+业务判断"阶段没有保护，高并发下会有很多请求都进到CAS这一步，造成大量数据库压力。

### 场景4：Redis Lua脚本 — 为什么秒杀库存不用CAS？

这是最关键的问题。先看如果秒杀用DB CAS会怎样：

**❌ 错误方案：秒杀用DB CAS**

```
1万人同时抢购，每人都执行：
UPDATE T_SECKILL_ACTIVITY SET STOCK=STOCK-1 WHERE ACTIVITY_ID=1 AND STOCK>0

Oracle DB：
  - 1万个请求同时到达
  - DB行锁把1万个请求串行化
  - 每次UPDATE都需要加行锁 → 等待 → 释放
  - 1万次串行 × 每次5ms = 50秒 ← 用户全在等！
  - DB连接池耗尽，其他业务全部超时
  - 结果：系统宕机 ❌
```

**✅ 正确方案：秒杀用Redis Lua脚本**

```
Redis单线程模型：
  所有命令排队执行，Lua脚本是一个不可中断的原子操作

秒杀Lua脚本：
  1. GET stock → 检查库存
  2. GET userKey → 检查用户限购
  3. DECR stock → 扣减库存
  4. INCR userKey → 记录用户
  以上4步在Redis内部原子执行，不可中断
```

**时序图（Redis Lua vs DB CAS 在秒杀场景的区别）：**

```
=== 方案对比：1万人同时抢购，库存500 ===

【DB CAS方案】
1万个请求       Oracle DB行锁
    │               │
    ├──→ 排队等待行锁 │
    ├──→ 等...       │
    ├──→ 等...       │  ← 1万人都在等DB行锁
    │               │    DB连接池：20个连接 × 1万请求 = 崩溃
    │               │
    [结果：DB宕机，秒杀失败]

【Redis Lua方案】
1万个请求         Redis(单线程)         Kafka            Oracle DB
    │                   │                 │                 │
    ├──→ EVAL Lua ──→  │                 │                 │
    │   (纳秒级执行)     │                 │                 │
    ├──→ EVAL Lua ──→  │ [Redis单线程]    │                 │
    │                   │ 排队但极快       │                 │
    ├──→ EVAL Lua ──→  │                 │                 │
    │                   │                 │                 │
    ← 结果1：成功(库存-1) │                 │                 │
    ← 结果2：成功(库存-1) │                 │                 │
    ← 结果3：失败(已售罄) │                 │                 │
    │ ...               │                 │                 │
    500个成功            │                 │                 │
    9500个立即失败        │                 │                 │
    │                   │                 │                 │
    成功的500个           │──→ 发Kafka消息 → │                 │
                        │                 │── 异步落库 ────→ │
                        │                 │  (慢慢写DB，      │
                        │                 │   不影响前端)    │

[结果：1万请求在Redis层毫秒内处理完，DB只接到500条写入，压力↓95%]
```

**为什么Lua脚本能做到"原子"？**

```
Redis 是单线程的：
  命令1 → 命令2 → 命令3 → ...  （严格串行）

Lua脚本相当于把多个命令"打包"成一个：
  [GET stock + GET userKey + DECR stock + INCR userKey]
  ↓
  作为一个整体执行，中间不插入任何其他命令

对比 Redis MULTI/EXEC（事务）：
  MULTI
  GET stock    ← 入队，不立即执行
  DECR stock   ← 入队
  EXEC         ← 一起执行

  但MULTI/EXEC是乐观锁（WATCH），在高并发下EXEC可能因WATCH失败整个回滚
  Lua脚本：执行期间其他客户端的命令全部等待，真正的原子
```

### 4.1 秒杀抢券：为什么用 Lua + 分布式锁 + Kafka，而不是CAS？

```
秒杀核心业务需要做的事：
  ① 检查活动是否在进行中
  ② 检查用户是否已抢过（防重复）
  ③ 扣减库存
  ④ 创建订单（写DB）
  ⑤ 扣积分（可能调外部服务）
  ⑥ 发券

步骤①②③必须原子执行 → Redis Lua脚本（内存操作，毫秒级）
步骤④⑤⑥异步处理    → Kafka（解耦，DB只接有效请求）

如果改成DB CAS：
  UPDATE STOCK WHERE STOCK>0 AND ACTIVITY_ID=?
  问题：1万人同时执行这条SQL，DB崩溃（见上面时序图）
  DB CAS只适合低并发的状态跳转，不适合高并发计数扣减
```

### 4.2 兑换码兑换：为什么用 分布式锁 + CAS，而不是Lua？

```
兑换码兑换需要做的事：
  ① HMAC验签（CPU计算）
  ② 查Redis限速（简单INCR）
  ③ 查DB获取码详情
  ④ 检查码状态（是否已用/过期）
  ⑤ 检查同批次限额
  ⑥ CAS更新DB：STATUS 0→1
  ⑦ 写GrantTask（补偿任务）
  ⑧ 发券

为什么不用Lua？
  Lua只能操作Redis，但兑换码状态在Oracle DB
  你不可能在一个Lua脚本里同时操作Redis和DB
  → Lua只负责Redis部分（限速计数），DB部分用CAS

为什么要加分布式锁？
  场景：用户双击提交，同一个码同时打到实例A和实例B
  没有锁：两个请求都读到STATUS=0，都进入CAS
         CAS会让一个失败，但失败的那个会加FAIL_COUNT
         正常用户被误加了失败计数（虽然不严重，但逻辑不优雅）
  有了锁：同一码同时只有1个请求进入，另一个等待/放弃
         完全避免并发CAS冲突
```

**兑换码完整时序图：**

```
用户               实例A           Redis(锁+限速)      Oracle DB
  │                  │                  │                  │
  │─ 兑换CP-001 ────→│                  │                  │
  │                  │                  │                  │
  │                  │ 1.HMAC验签(本地)  │                  │
  │                  │                  │                  │
  │                  │── 2.IP限速 ─────→│                  │
  │                  │← INCR通过        │                  │
  │                  │                  │                  │
  │                  │── 3.用户限速 ────→│                  │
  │                  │← INCR通过        │                  │
  │                  │                  │                  │
  │                  │── 4.tryLock ────→│                  │
  │                  │← 获取锁成功       │                  │
  │                  │                  │                  │
  │                  │── 5.查码信息 ─────────────────────→ │
  │                  │← STATUS=0，未过期 ──────────────────│
  │                  │                  │                  │
  │                  │── 6.CAS更新 ──────────────────────→ │
  │                  │   WHERE STATUS=0  │                  │ 行锁保证原子
  │                  │← affected=1成功 ───────────────────│
  │                  │                  │                  │
  │                  │── 7.写GrantTask ───────────────────→│
  │                  │                  │                  │
  │                  │── 8.unlock ─────→│                  │
  │                  │                  │                  │
  │←── 兑换成功 ─────│                  │                  │

[如果另一个请求同时发来]

用户乙             实例B           Redis(锁)
  │                  │                │
  │─ 兑换CP-001 ────→│                │
  │                  │── tryLock ────→│
  │                  │← 失败(A持有锁)  │
  │                  │                │
  │←── 请稍后重试 ───│                │
```

### 4.3 为什么兑换码CAS可以，秒杀库存不用CAS？

这是最关键的问题，用一个对比表说清楚：

| 对比维度 | 兑换码 CAS | 秒杀库存 CAS |
|---------|-----------|------------|
| **并发量** | 低（每个码只有1个用户兑换） | 极高（1万人同时抢） |
| **DB压力** | 低（1万个码，每个码1次CAS） | 极高（1万人同时UPDATE同一行） |
| **行锁竞争** | 基本无竞争（每个码独立的行） | 极度竞争（1万人抢同一行） |
| **操作复杂度** | 简单（只改STATUS） | 复杂（扣库存+检查用户+记录用户） |
| **失败处理** | 直接返回失败即可 | 需要立即告知用户是否抢到 |

```
兑换码：
  有10万张码，用户甲兑换CP-001，用户乙兑换CP-002
  → 两个CAS操作的是不同的行 → 完全不冲突 → CAS完全没问题

秒杀：
  1万人都要修改 ACTIVITY_ID=1 这同一行的 STOCK 字段
  → 1万个请求排队等同一行的行锁
  → Oracle DB挂了
```

## 五、如果用错了会怎样？

### 错误1：秒杀用 synchronized

```java
// ❌ 错误写法
public synchronized void grabCoupon(Long userId) {
    // 查库存
    int stock = db.getStock(activityId);
    if (stock <= 0) throw new Exception("已售罄");
    // 扣库存
    db.decreaseStock(activityId);
    // 发券
    grantCoupon(userId);
}
```

**问题**：

```
实例A - 线程1：synchronized拿到锁，执行中...
实例A - 线程2：等待锁...
实例B - 线程1：synchronized拿到锁（实例B自己的锁！），执行中...
实例C - 线程1：synchronized拿到锁（实例C自己的锁！），执行中...

三个实例同时执行，库存=3，三个线程各查到stock=3，各发一次券
实际只有3份库存，却发了9份券！超卖！
```

// ❌ 高并发下错误

### 错误3：兑换码只用CAS，不用分布式锁

```java
// ⚠️ 只有CAS，没有分布式锁
public Long redeem(String code, String userId) {
    CouponRedeemCode rc = db.findByCode(code);
    // 没有锁，两个请求同时到这里
    if (rc.getStatus() != 0) throw ...;
    int affected = db.casUpdate(code); // CAS
    if (affected == 0) {
        db.incrementFailCount(code); // ← 问题在这里
        throw new Exception("失败");
    }
    return grantCoupon(userId);
}
```

**问题**：

```
用户甲双击提交，同时打到实例A和实例B：
  请求1(实例A)：查STATUS=0 → CAS成功 → 发券 ✅
  请求2(实例B)：查STATUS=0 → CAS失败(affected=0) → FAIL_COUNT+1

正常用户被误加了失败计数！
虽然CAS保证了数据正确，但FAIL_COUNT被错误累加
如果这个用户频繁网络重试，FAIL_COUNT可能达到5，码被锁定！

加了分布式锁后：
  请求1拿到锁，请求2等待
  请求1完成后释放锁
  请求2拿到锁，查STATUS=1(已使用)，直接返回"已被使用"，不加FAIL_COUNT ✅
```

### 错误4：兑换码用Lua脚本管状态

```lua
-- ❌ 概念性错误（无法在Lua里操作Oracle DB）
local status = redis.call('GET', 'code:status:CP-001')  -- 错误：状态在DB，不在Redis
if status == '0' then
    redis.call('SET', 'code:status:CP-001', '1')
    return 1
end
return 0
```

**问题**：

```
Redis是缓存，Oracle才是数据源：
  - 如果Redis宕机，所有码的状态丢失
  - Redis和DB的状态会不一致（DB里是0，Redis里是1）
  - 重启Redis后，所有码状态回到初始值，可以被无限兑换！

正确方式：
  码状态永远在Oracle DB（可靠持久化）
  Redis只做辅助（限速计数、分布式锁），不存核心业务数据
```

## 六、总决策树

```
面对一个并发问题，按照以下决策树选择方案：

开始
  │
  ▼
是否跨多台服务器？
  │
  ├─ 否（单台） ──→ 简单用 synchronized / ReentrantLock
  │                 （本地缓存、单机计数器、本地队列）
  │
  └─ 是（多台） ──→ 继续↓
                    │
                    ▼
                  操作的数据在哪里？
                    │
                    ├─ 在 Redis ──→ 是简单原子操作？
                    │               │
                    │               ├─ 是（计数、扣减、检查+修改）
                    │               │   └─ Redis Lua脚本
                    │               │      （库存扣减、限速计数）
                    │               │
                    │               └─ 否（需要读Redis后做复杂判断）
                    │                   └─ Redisson分布式锁 + 普通Redis操作
                    │
                    ├─ 在 DB ───→ 并发量高吗？
                    │               │
                    │               ├─ 低并发（兑换码、订单状态）
                    │               │   └─ DB CAS（一条WHERE条件SQL）
                    │               │      + 分布式锁（减少CAS冲突和误加FAIL_COUNT）
                    │               │
                    │               └─ 高并发（秒杀库存、积分扣减）
                    │                   └─ Redis Lua预扣（先扣Redis）
                    │                      + Kafka异步落库（再写DB）
                    │                      DB CAS做最终兜底
                    │
                    └─ 跨DB + Redis ──→ 分布式锁 + 本地消息表（最终一致性）
                                        （兑换码发券：CAS码状态 + 写GrantTask）
```

## 七、本系统实际使用汇总

| 模块 | 使用方案 | 原因 |
|------|---------|------|
| **秒杀库存扣减** | Redis Lua脚本 | 1万人同时操作同一个数字，Lua保证原子+无DB压力 |
| **秒杀用户限购检查** | Redis Lua脚本（同上） | 与库存扣减合并在一个Lua里，原子执行 |
| **秒杀异步落库** | Kafka + DB | Lua成功后发消息，消费者慢慢写DB，DB只接有效请求 |
| **兑换码并发保护** | Redisson分布式锁 | 防双击/网络重试，保护查DB→判断→CAS整个流程 |
| **兑换码状态跳转** | DB CAS（WHERE STATUS=0） | 简单0→1跳转，Oracle行锁天然保证，无需额外依赖 |
| **兑换码过期Job** | DB批量UPDATE | 低并发定时任务，直接操作DB即可 |
| **IP/用户限速** | Redis INCR + TTL | 简单计数，Redis原子INCR，多实例共享 |
| **发券幂等** | Redis KV缓存 | requestId → couponId，5分钟TTL |
| **GrantTask重试** | DB轮询 + 指数退避 | 简单可靠，不需要复杂机制 |
| **本地缓存** | Caffeine（单机） | 券模板读多写少，每个实例独立缓存，不需要分布式 |

## 八、一句话总结记忆法

```
synchronized / ReentrantLock：
  → "单台机器内部的锁，出了JVM就没用"

Redis Lua脚本：
  → "Redis数据的原子操作，用于高并发计数/扣减，不等待不排队"

Redisson分布式锁：
  → "跨服务器的互斥锁，保护整个业务流程，同一时刻全局只有一个人执行"

DB CAS（WHERE条件更新）：
  → "简单状态跳转的终极保证，不依赖外部组件，DB行锁天然原子"
  → "适合低并发状态机，不适合高并发同行竞争"

Kafka异步落库：
  → "先抢（Redis），后写（DB），把高峰流量转移到DB能处理的节奏"
```
