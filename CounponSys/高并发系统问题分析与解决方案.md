# 高并发系统问题分析与解决方案

## 一、分布式锁方案对比

### 1.1 行业主流方案对比

| 方案 | 推荐度 | 优点 | 缺点 | 适用场景 |
|------|-------|------|------|---------|
| **Redisson** | ⭐⭐⭐⭐⭐ | 功能完善、自动续期、看门狗机制 | 依赖Redis | **强烈推荐** |
| Redis SETNX | ⭐⭐⭐ | 简单易用 | 无自动续期、需手动处理 | 简单场景 |
| ZooKeeper | ⭐⭐⭐⭐ | 强一致性 | 性能较低 | 对一致性要求极高 |
| MySQL | ⭐⭐ | 无需额外中间件 | 性能差 | 不推荐高并发 |
| etcd | ⭐⭐⭐⭐ | 强一致性、云原生 | 学习成本高 | K8s环境 |

### 1.2 为什么推荐Redisson？

```
┌─────────────────────────────────────────────────────────────────┐
│  问题: 普通Redis锁的缺陷                                          │
└─────────────────────────────────────────────────────────────────┘

场景: 用户A获取锁，执行业务，但业务执行时间 > 锁过期时间

时序图:
┌──────┐     ┌──────┐     ┌──────┐
│用户A │     │Redis │     │用户B │
└──┬───┘     └──┬───┘     └──┬───┘
   │            │            │
   │ SETNX lock │            │
   │───────────>│            │
   │  成功(TTL=5s)           │
   │<───────────│            │
   │            │            │
   │ 执行业务...│            │
   │ (耗时8秒)  │            │
   │            │            │
   │    ┌───────┴───────┐    │
   │    │ 5秒后锁自动过期│    │
   │    └───────┬───────┘    │
   │            │            │
   │            │ SETNX lock │
   │            │<───────────│
   │            │  成功!!!   │  <-- 用户B也拿到锁了!!!
   │            │───────────>│
   │            │            │
   │ 执行完毕   │ 执行业务...|
   │ DEL lock   │            │
   │───────────>│            │
   │    删除了B的锁!!!       │  <-- 灾难! 删错锁了!
   │            │            │

结果: 两个用户同时执行"加锁"代码段，数据错乱!
```

### 1.3 Redisson如何解决？

```
┌─────────────────────────────────────────────────────────────────┐
│  Redisson的看门狗(WatchDog)机制                                  │
└─────────────────────────────────────────────────────────────────┘

时序图:
┌──────┐     ┌──────────┐     ┌──────┐
│用户A │     │Redisson  │     │Redis │
└──┬───┘     └────┬─────┘     └──┬───┘
   │              │              │
   │   lock()     │              │
   │─────────────>│  SETNX lock  │
   │              │─────────────>│
   │              │   成功(30s)  │
   │              │<─────────────│
   │              │              │
   │   执行业务...│              │
   │   (耗时8秒)  │              │
   │              │              │
   │      ┌───────┴───────┐     │
   │      │ 看门狗线程     │     │
   │      │ 每10秒检查一次│     │
   │      │ 如果业务未完成│     │
   │      │ 自动续期到30秒│     │
   │      └───────┬───────┘     │
   │              │  EXPIRE 30s │
   │              │─────────────>│ <-- 自动续期!
   │              │<─────────────│
   │              │              │
   │   业务完成   │              │
   │   unlock()   │              │
   │─────────────>│  DEL lock   │
   │              │─────────────>│
   │              │  只删自己的锁(通过UUID)
   │              │              │

优点:
1. 自动续期，不会提前过期
2. UUID标识，不会误删别人的锁
3. 可重入锁，同一线程可多次获取
```

### 1.4 Redisson代码示例

```java
// 本项目已使用的方式
@Autowired
private RedissonClient redissonClient;

public void claimCoupon(String userId, Long templateId) {
    String lockKey = "coupon:claim:" + userId + ":" + templateId;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        // tryLock(等待时间, 锁持有时间, 时间单位)
        // 如果不指定leaseTime，则使用看门狗自动续期
        if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
            // 业务逻辑...
        } else {
            throw new CouponException("操作频繁，请稍后重试");
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        // 只有当前线程持有锁时才释放
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

### 2.2 限流算法对比

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **令牌桶** | 以固定速率放令牌，请求需获取令牌 | 允许突发流量 | 实现稍复杂 | **推荐** |
| 漏桶 | 请求进入桶，以固定速率流出 | 流量平滑 | 不支持突发 | 需要平滑流量 |
| 计数器 | 固定时间窗口内计数 | 简单 | 临界问题 | 简单场景 |
| 滑动窗口 | 滑动时间窗口计数 | 解决临界问题 | 内存占用高 | 精确限流 |

### 2.4 本项目限流配置详解

```java
/**
 * 限流配置详解
 */
@Configuration
public class RateLimiterConfig {

    // 从配置文件读取容量，默认10000
    @Value("${seckill.rate-limit.capacity:10000}")
    private int capacity;

    /**
     * 全局限流器
     * 
     * RateLimiter.create(10000) 表示:
     * - 每秒生成10000个令牌
     * - 初始桶是满的(可突发)
     * - 请求必须获取到令牌才能执行
     */
    @Bean
    public RateLimiter seckillRateLimiter() {
        return RateLimiter.create(capacity);
    }
}
```

```java
/**
 * 在Controller中使用
 */
@PostMapping("/grab")
public ApiResponse<SeckillGrabResult> grabCoupon(@RequestBody SeckillGrabRequest request) {
    
    // tryAcquire(100, TimeUnit.MILLISECONDS)
    // 尝试在100毫秒内获取令牌
    // - 成功: 返回true，继续执行
    // - 失败: 返回false，拒绝请求
    
    if (!seckillRateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
        // 获取不到令牌，说明请求太多了
        throw SeckillException.systemBusy(); // 返回"系统繁忙"
    }
    
    // 获取到令牌，正常处理
    return seckillService.grabCoupon(request);
}
```

### 2.5 限流的问题与解决方案

#### 问题1: 单机限流 vs 分布式限流

```
┌─────────────────────────────────────────────────────────────────┐
│  问题: Guava RateLimiter是单机限流                               │
└─────────────────────────────────────────────────────────────────┘

                    10万请求
                        │
         ┌──────────────┼──────────────┐
         │              │              │
         ▼              ▼              ▼
    ┌─────────┐    ┌─────────┐    ┌─────────┐
    │ 服务器1  │    │ 服务器2  │    │ 服务器3  │
    │ 限流1万 │    │ 限流1万 │    │ 限流1万 │
    └─────────┘    └─────────┘    └─────────┘
         │              │              │
         ▼              ▼              ▼
    放行1万        放行1万        放行1万
    
    结果: 实际放行了3万请求! 限流失效!
```

**解决方案**: Redis + Lua 分布式限流

```java
/**
 * 分布式限流器
 */
@Component
public class DistributedRateLimiter {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    // Lua脚本: 滑动窗口限流
    private static final String LUA_SCRIPT = 
        "local key = KEYS[1] " +
        "local limit = tonumber(ARGV[1]) " +
        "local window = tonumber(ARGV[2]) " +
        "local current = tonumber(redis.call('GET', key) or '0') " +
        "if current + 1 > limit then " +
        "    return 0 " +
        "else " +
        "    redis.call('INCR', key) " +
        "    redis.call('EXPIRE', key, window) " +
        "    return 1 " +
        "end";

    /**
     * 尝试获取令牌
     * @param key 限流key
     * @param limit 限制次数
     * @param windowSeconds 时间窗口(秒)
     */
    public boolean tryAcquire(String key, int limit, int windowSeconds) {
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(LUA_SCRIPT, Long.class),
            Collections.singletonList(key),
            limit, windowSeconds
        );
        return result != null && result == 1;
    }
}
```

#### 问题2: 限流粒度

| 粒度 | Key示例 | 场景 |
|------|--------|------|
| 全局 | `seckill:limit:global` | 保护整个系统 |
| 接口 | `seckill:limit:api:grab` | 保护单个接口 |
| 用户 | `seckill:limit:user:U123` | 防止单用户刷接口 |
| IP | `seckill:limit:ip:1.2.3.4` | 防止恶意IP |
| 活动 | `seckill:limit:activity:1001` | 保护单个活动 |

**最佳实践: 多层限流**

```java
@PostMapping("/grab")
public ApiResponse<SeckillGrabResult> grabCoupon(@RequestBody SeckillGrabRequest request) {
    
    String userId = request.getUserId();
    String ip = getClientIp();
    
    // 第1层: 全局限流(1万/秒)
    if (!globalRateLimiter.tryAcquire()) {
        throw SeckillException.systemBusy();
    }
    
    // 第2层: IP限流(100/秒) - 防CC攻击
    if (!distributedLimiter.tryAcquire("limit:ip:" + ip, 100, 1)) {
        throw SeckillException.rateLimitExceeded();
    }
    
    // 第3层: 用户限流(10/秒) - 防刷
    if (!distributedLimiter.tryAcquire("limit:user:" + userId, 10, 1)) {
        throw SeckillException.rateLimitExceeded();
    }
    
    return seckillService.grabCoupon(request);
}
```


#### 解决方案: Lua脚本原子操作

```lua
-- 原子检查+扣减，不会超卖
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock <= 0 then 
    return -1  -- 库存不足，直接拒绝
end
redis.call('DECR', KEYS[1])
return 1  -- 扣减成功
```

### 3.4 问题3: 缓存击穿 🟡

#### 时序分析

```
┌─────────────────────────────────────────────────────────────────┐
│  问题场景: 热点Key过期，大量请求打到数据库                          │
```

```java
/**
 * 解决缓存击穿
 */
public SeckillActivity getActivity(Long activityId) {
    String cacheKey = "seckill:activity:" + activityId;
    
    // 1. 查缓存
    SeckillActivity activity = (SeckillActivity) redisTemplate.opsForValue().get(cacheKey);
    if (activity != null) {
        return activity;
    }
    
    // 2. 缓存不存在，加锁查库
    String lockKey = "lock:activity:" + activityId;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        // 尝试获取锁，只有一个请求能进入
        if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
            // 双重检查
            activity = (SeckillActivity) redisTemplate.opsForValue().get(cacheKey);
            if (activity != null) {
                return activity;
            }
            
            // 查数据库
            activity = activityMapper.selectById(activityId);
            
            // 写入缓存
            if (activity != null) {
                redisTemplate.opsForValue().set(cacheKey, activity, 1, TimeUnit.HOURS);
            } else {
                // 缓存空值，防止缓存穿透
                redisTemplate.opsForValue().set(cacheKey, "NULL", 5, TimeUnit.MINUTES);
            }
            
            return activity;
        } else {
            // 没获取到锁，等待后重试
            Thread.sleep(100);
            return getActivity(activityId);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        throw new RuntimeException(e);
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

```java
/**
 * 解决缓存雪崩: 过期时间加随机值
 */
public void cacheActivity(SeckillActivity activity) {
    String cacheKey = "seckill:activity:" + activity.getId();
    
    // 基础过期时间: 1小时
    long baseExpire = 3600;
    
    // 加上随机值: 0-600秒(0-10分钟)
    long randomExpire = ThreadLocalRandom.current().nextLong(600);
    
    // 最终过期时间: 1小时 + 随机0-10分钟
    long finalExpire = baseExpire + randomExpire;
    
    redisTemplate.opsForValue().set(cacheKey, activity, finalExpire, TimeUnit.SECONDS);
}
```

```java
/**
 * 延迟双删策略
 */
public void updateStock(Long activityId, int newStock) {
    String cacheKey = "seckill:stock:" + activityId;
    
    // 1. 先删除缓存
    redisTemplate.delete(cacheKey);
    
    // 2. 更新数据库
    activityMapper.updateStock(activityId, newStock);
    
    // 3. 延迟再删一次(异步)
    // 延迟时间 = 读操作时间 + 几百毫秒
    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(500); // 延迟500ms
            redisTemplate.delete(cacheKey);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    });
}
```
