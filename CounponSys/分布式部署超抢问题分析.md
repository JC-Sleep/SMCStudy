## 二、超抢是怎么发生的？

### 2.1 问题1: 单机限流器失效

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Guava RateLimiter 是单机限流器，每个JVM进程独立计数                              │
└─────────────────────────────────────────────────────────────────────────────────┘

时序图:

时间    │  服务器1         │  服务器2         │  服务器3        │ 总放行数
────────┼─────────────────┼─────────────────┼────────────────┼──────────
        │  令牌桶=1万      │  令牌桶=1万      │  令牌桶=1万     │
        │                 │                 │                │
t=0     │  收到3.3万请求   │  收到3.3万请求   │  收到3.4万请求  │
        │  放行1万 ✓      │  放行1万 ✓      │  放行1万 ✓     │  3万!!!
        │  拒绝2.3万      │  拒绝2.3万      │  拒绝2.4万     │
        │                 │                 │                │
────────┼─────────────────┼─────────────────┼────────────────┼──────────

结果: 限流器设置1万/秒，但实际放行了3万请求!!!

原因: 每台服务器的Guava RateLimiter是独立的，互不知道对方的状态
```

### 3.2 Redis分布式限流

```java
/**
 * 分布式限流 - 所有服务器共享同一个计数器
 */
@Component
public class DistributedRateLimiter {

    // Lua脚本: 滑动窗口限流
    private static final String SCRIPT = 
        "local key = KEYS[1] " +
        "local limit = tonumber(ARGV[1]) " +  -- 限制次数
        "local window = tonumber(ARGV[2]) " + -- 时间窗口
        "local current = tonumber(redis.call('GET', key) or '0') " +
        "if current >= limit then " +
        "    return 0 " +  -- 超过限制，拒绝
        "end " +
        "redis.call('INCR', key) " +
        "if current == 0 then " +
        "    redis.call('EXPIRE', key, window) " +
        "end " +
        "return 1";  -- 放行

    public boolean tryAcquire(String key, int limit, int windowSeconds) {
        Long result = redisTemplate.execute(script, 
            Collections.singletonList(key), limit, windowSeconds);
        return result != null && result == 1;
    }
}
```

### 3.3 Redis Lua脚本原子扣库存

```lua
-- seckill_grab.lua
-- 原子操作：检查库存 + 检查限购 + 扣减库存 + 记录用户
-- 整个脚本在Redis中原子执行，不会被其他命令打断!!!

local stockKey = KEYS[1]      -- 库存Key
local userKey = KEYS[2]       -- 用户已抢Key
local limitPerUser = tonumber(ARGV[1])  -- 每人限购
local grabCount = tonumber(ARGV[2])     -- 本次抢购数量

-- ① 检查库存
local stock = tonumber(redis.call('GET', stockKey) or '0')
if stock < grabCount then
    return -1  -- 库存不足
end

-- ② 检查用户限购
local userGrabbed = tonumber(redis.call('GET', userKey) or '0')
if userGrabbed + grabCount > limitPerUser then
    return -2  -- 超过限购
end

-- ③ 扣减库存 (原子操作!!!)
redis.call('DECRBY', stockKey, grabCount)

-- ④ 记录用户已抢数量
redis.call('INCRBY', userKey, grabCount)
redis.call('EXPIRE', userKey, 86400)  -- 24小时过期

return 1  -- 成功
```

## 五、本项目已实现的防超抢机制

### 5.1 代码位置

```

│  本项目的防超抢实现                                                              │
└─────────────────────────────────────────────────────────────────────────────────┘

   文件: util/DistributedRateLimiter.java
   作用: 所有服务器共享Redis计数器

   文件: resources/lua/seckill_grab.lua
   作用: 原子检查+扣减，防止超卖

   文件: controller/SeckillController.java
   作用: 本地限流 + IP限流 + 用户限流
```

### 5.2 SeckillController代码

```java
@PostMapping("/grab")
public ApiResponse<SeckillGrabResult> grabCoupon(
        @RequestBody SeckillGrabRequest request,
        HttpServletRequest httpRequest) {
    
    String userId = request.getUserId();
    String clientIp = getClientIp(httpRequest);
    
    // ━━━━━━━━━━ 第1层: 本地限流 (Guava) ━━━━━━━━━━
    // 作用: 快速拦截，减少Redis压力
    // 问题: 分布式环境下不精确，但能挡住大部分请求
    if (!seckillRateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
        throw SeckillException.systemBusy();
    }
    
    // ━━━━━━━━━━ 第2层: 分布式IP限流 (Redis) ━━━━━━━━━━
    // 作用: 防止单个IP刷接口
    if (!distributedRateLimiter.tryAcquire("limit:ip:" + clientIp, 100, 1)) {
        throw SeckillException.rateLimitExceeded();
    }
    
    // ━━━━━━━━━━ 第3层: 分布式用户限流 (Redis) ━━━━━━━━━━
    // 作用: 防止单个用户疯狂点击
    if (!distributedRateLimiter.tryAcquire("limit:user:" + userId, 10, 1)) {
        throw SeckillException.rateLimitExceeded();
    }
    
    // ━━━━━━━━━━ 第4层: Lua脚本扣库存 (Redis) ━━━━━━━━━━
    // 在SeckillService中执行
    return seckillService.grabCoupon(request);
}
```

## 七、如果真的超抢了，是怎么发生的？

### 7.1 可能的原因

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  超抢可能的原因                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘

原因1: 没使用Lua脚本，用了普通的GET+DECR
┌─────────────────────────────────────────────────────────────────┐
│  // ❌ 错误代码                                                 │
│  if (redis.get("stock") > 0) {                                 │
│      redis.decr("stock");  // 这里可能超卖!!!                  │
│  }                                                              │
└─────────────────────────────────────────────────────────────────┘

原因2: Redis库存没预热，直接查数据库
┌─────────────────────────────────────────────────────────────────┐
│  // ❌ 错误代码                                                 │
│  int stock = db.query("SELECT stock FROM activity");           │
│  if (stock > 0) {                                              │
│      db.update("UPDATE activity SET stock = stock - 1");       │
│      // 数据库并发扣减，可能超卖!!!                              │
│  }                                                              │
└─────────────────────────────────────────────────────────────────┘

原因3: Lua脚本有bug
┌─────────────────────────────────────────────────────────────────┐
│  -- ❌ 错误的Lua脚本                                            │
│  local stock = redis.call('GET', key)                          │
│  -- 忘记检查stock是否为nil!!!                                   │
│  redis.call('DECR', key)                                       │
└─────────────────────────────────────────────────────────────────┘

原因4: Redis和数据库数据不一致
┌─────────────────────────────────────────────────────────────────┐
│  Redis库存=0，但数据库库存=100                                   │
│  如果有代码绕过Redis直接查数据库，就会超卖                        │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 如何验证是否会超卖？

```bash
# 压测验证
# 1. 设置库存为100
redis-cli SET seckill:stock:1 100

# 2. 用1000并发抢购
wrk -t10 -c1000 -d10s -s post.lua http://localhost:8090/api/seckill/grab

# 3. 检查结果
redis-cli GET seckill:stock:1
# 应该是: 0 (刚好抢完)
# 如果是: -50 (负数)，说明超卖了!!!
```
