# 高并发场景下库存回滚与活动结束优化方案

> **创建时间**: 2026-03-26  
> **优化目的**: 解决回滚库存、友好提示、活动真正结束的问题

---

## 📋 问题场景分析

### 用户提出的场景：

```
时间轴：
0秒：  1万人进来抢购
       ↓
       Redis扣库存成功（1万个）
       ↓
       创建1万个订单（status=0）
       ↓
       发送1万条Kafka消息

1-5秒：Kafka消费者处理
       ├─ 9900人成功（扣积分+发券）→ status=1
       └─ 100人失败（重试3次后仍失败）→ status=2
           ↓
           回滚Redis库存+100
           回滚积分100人

5秒：  Redis库存从0变成100
       ↓
       新的用户又能进来抢购！

问题：
1. Redis库存=0时，提示"已售罄"，但5秒后回滚了100，用户会疑惑
2. 活动结束应该看Redis还是DB？
3. 如果DB真没库存了，Job如何停止？
```

## 🎯 优化方案

### 优化1：Redis库存=0时的友好提示

**原来的逻辑**（不友好）：
```
Redis库存=0 → 返回"已售罄"
5秒后回滚+100 → 又能抢了
用户：？？？刚才说售罄，现在又能抢？
```

**优化后的逻辑**：
```
Redis库存=0 → 检查DB库存和待处理订单
    │
    ├─ DB库存>0 或 有待处理订单 → "活动火爆，请稍后刷新重试"
    └─ DB库存=0 且 无待处理订单 → "已售罄"
```

**实现代码**：

```java
// SeckillServiceImpl.java

/**
 * 处理Lua脚本返回结果（优化版）
 */
private SeckillGrabResult handleLuaResult(Long result, Long activityId) {
    if (result == null) {
        return SeckillGrabResult.fail("系统异常");
    }
    switch (result.intValue()) {
        case -1:
            // 2026-03-26 优化：库存不足时，智能判断提示信息
            return handleStockInsufficient(activityId);
        case -2:
            throw SeckillException.grabLimitExceeded();
        case -3:
            throw SeckillException.activityNotFound();
        default:
            return SeckillGrabResult.fail("抢购失败");
    }
}

/**
 * 处理库存不足（智能判断）
 * 
 * 场景：
 * 1. Redis库存=0，但DB还有库存 → "活动火爆，请稍后重试"（可能有回滚）
 * 2. Redis库存=0，DB也=0 → "已售罄"
 * 3. Redis库存=0，有待处理订单 → "活动火爆，请稍后重试"（等待处理结果）
 */
private SeckillGrabResult handleStockInsufficient(Long activityId) {
    // 1. 检查DB真实库存
    SeckillActivity activity = activityMapper.selectById(activityId);
    if (activity == null) {
        throw SeckillException.activityNotFound();
    }
    
    // 2. 检查是否有待处理订单
    long pendingCount = orderMapper.selectCount(
            new LambdaQueryWrapper<SeckillOrder>()
                    .eq(SeckillOrder::getActivityId, activityId)
                    .eq(SeckillOrder::getStatus, 0) // 待处理
    );
    
    // 3. 智能判断
    if (activity.getRemainStock() > 0 || pendingCount > 0) {
        // DB还有库存 或 有订单正在处理（可能会回滚库存）
        log.info("Redis库存临时为0，但DB库存={}, 待处理订单={}", 
                activity.getRemainStock(), pendingCount);
        throw new SeckillException(400, "活动火爆，请稍后刷新重试");
    } else {
        // DB库存也为0，且无待处理订单，真的售罄了
        log.info("活动真正售罄: activityId={}, DB库存={}", activityId, activity.getRemainStock());
        throw SeckillException.soldOut();
    }
}
```

### 优化2：活动真正结束的条件

**原来的逻辑**（不准确）：
```
活动结束条件：endTime已过
问题：即使还有待处理订单，也标记为结束
```

**优化后的逻辑**：
```
活动结束条件：
1. endTime已过
2. 且 DB库存=0
3. 且 无待处理订单（所有订单都处理完了）
```

**实现代码**：

```java
// CouponScheduledJob.java

/**
 * 更新秒杀活动状态 - 每分钟执行一次（优化版）
 */
@Scheduled(cron = "0 * * * * ?")
public void updateSeckillActivityStatus() {
    LocalDateTime now = LocalDateTime.now();
    
    // 1. 将已到开始时间的活动状态更新为进行中
    seckillActivityMapper.update(null, 
            new LambdaUpdateWrapper<SeckillActivity>()
                    .set(SeckillActivity::getStatus, SeckillStatus.ONGOING.getCode())
                    .eq(SeckillActivity::getStatus, SeckillStatus.PENDING.getCode())
                    .le(SeckillActivity::getStartTime, now)
                    .gt(SeckillActivity::getEndTime, now)
                    .eq(SeckillActivity::getDeleted, 0)
    );
    
    // 2026-03-26 优化：智能判断活动是否真正结束
    // 2. 检查已过结束时间的活动
    List<SeckillActivity> endingActivities = seckillActivityMapper.selectList(
            new LambdaQueryWrapper<SeckillActivity>()
                    .in(SeckillActivity::getStatus, 
                            SeckillStatus.PENDING.getCode(), 
                            SeckillStatus.ONGOING.getCode())
                    .le(SeckillActivity::getEndTime, now)
                    .eq(SeckillActivity::getDeleted, 0)
    );
    
    for (SeckillActivity activity : endingActivities) {
        checkAndFinishActivity(activity);
    }
}

/**
 * 检查并完成活动
 * 
 * 活动真正结束的条件：
 * 1. endTime已过
 * 2. DB库存=0（真没库存了）
 * 3. 无待处理订单（所有订单都处理完了）
 */
private void checkAndFinishActivity(SeckillActivity activity) {
    Long activityId = activity.getId();
    
    // 1. 检查DB真实库存
    int dbStock = activity.getRemainStock();
    
    // 2. 检查待处理订单数量
    long pendingCount = seckillOrderMapper.selectCount(
            new LambdaQueryWrapper<SeckillOrder>()
                    .eq(SeckillOrder::getActivityId, activityId)
                    .eq(SeckillOrder::getStatus, 0) // 待处理
    );
    
    // 3. 判断是否真正结束
    if (dbStock == 0 && pendingCount == 0) {
        // 真正结束：DB无库存 + 无待处理订单
        seckillActivityMapper.update(null,
                new LambdaUpdateWrapper<SeckillActivity>()
                        .set(SeckillActivity::getStatus, SeckillStatus.FINISHED.getCode())
                        .eq(SeckillActivity::getId, activityId)
        );
        
        // 清理Redis缓存
        String stockKey = RedisKeys.getSeckillStockKey(activityId);
        String activityKey = RedisKeys.getSeckillActivityKey(activityId);
        redisTemplate.delete(stockKey);
        redisTemplate.delete(activityKey);
        
        log.info("秒杀活动真正结束: activityId={}, name={}", activityId, activity.getName());
    } else {
        // 还有库存或待处理订单，暂不结束
        log.info("秒杀活动endTime已过，但暂不结束: activityId={}, DB库存={}, 待处理订单={}", 
                activityId, dbStock, pendingCount);
    }
}
```

### 优化3：库存售罄自动标记

**场景**：
- 活动还没到endTime
- 但DB库存=0 且 无待处理订单
- 应该自动标记为"已售罄"

**实现代码**：

```java
// CouponScheduledJob.java

// 2026-03-26 新增
/**
 * 检查进行中活动是否售罄 - 每30秒执行一次
 */
@Scheduled(fixedRate = 30000)
public void checkSoldOutActivities() {
    // 查询进行中的活动
    List<SeckillActivity> ongoingActivities = seckillActivityMapper.selectList(
            new LambdaQueryWrapper<SeckillActivity>()
                    .eq(SeckillActivity::getStatus, SeckillStatus.ONGOING.getCode())
                    .eq(SeckillActivity::getDeleted, 0)
    );
    
    for (SeckillActivity activity : ongoingActivities) {
        Long activityId = activity.getId();
        
        // 1. 检查DB库存
        if (activity.getRemainStock() > 0) {
            continue; // 还有库存，跳过
        }
        
        // 2. 检查待处理订单
        long pendingCount = seckillOrderMapper.selectCount(
                new LambdaQueryWrapper<SeckillOrder>()
                        .eq(SeckillOrder::getActivityId, activityId)
                        .eq(SeckillOrder::getStatus, 0)
        );
        
        if (pendingCount > 0) {
            continue; // 有待处理订单，可能会回滚库存
        }
        
        // 3. DB库存=0 且 无待处理订单 → 标记为已售罄
        seckillActivityMapper.update(null,
                new LambdaUpdateWrapper<SeckillActivity>()
                        .set(SeckillActivity::getStatus, SeckillStatus.SOLD_OUT.getCode())
                        .eq(SeckillActivity::getId, activityId)
        );
        
        log.info("秒杀活动提前售罄: activityId={}, name={}", activityId, activity.getName());
    }
}
// end 2026-03-26 新增
```

### 优化4：Job任务智能跳过

**问题**：
- 活动已售罄或已结束
- Job还在不断检查和调整
- 浪费资源

**优化**：

```java
// CouponScheduledJob.java

/**
 * 动态调整限流器 - 每5秒执行一次（优化版）
 */
@Scheduled(fixedRate = 5000)
public void adjustDynamicRateLimiters() {
    // 2026-03-26 优化：只处理进行中的活动
    List<SeckillActivity> ongoingActivities = seckillActivityMapper.selectList(
            new LambdaQueryWrapper<SeckillActivity>()
                    .eq(SeckillActivity::getStatus, SeckillStatus.ONGOING.getCode())  // ← 只查进行中
                    .eq(SeckillActivity::getDeleted, 0)
    );
    
    if (ongoingActivities.isEmpty()) {
        log.debug("当前无进行中的秒杀活动，跳过限流调整");
        return;  // ← 提前返回，节省资源
    }
    
    for (SeckillActivity activity : ongoingActivities) {
        try {
            dynamicRateLimiterManager.adjustRateIfNeeded(activity.getId());
        } catch (Exception e) {
            log.error("动态调整限流失败: activityId={}", activity.getId(), e);
        }
    }
    
    log.debug("动态限流调整完成，处理了 {} 个活动", ongoingActivities.size());
}
```
