# 分布式架构深度答疑

## 问题一：不用Lua脚本，普通GET+DECR会超卖多少件？

### 答案：不是最多超卖2件，而是可能超卖**成百上千件**！

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  错误理解：每台服务器超卖1件，3台服务器最多超卖2件？                               │
│  正确理解：超卖数量 = 并发请求数 - 实际库存                                       │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 时序分析：为什么会超卖很多

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  场景：库存=1件，1000个请求同时到达                                               │
└─────────────────────────────────────────────────────────────────────────────────┘

错误代码：
┌─────────────────────────────────────────────────────────────────┐
│  int stock = redis.get("stock");  // 第1步：读取                │
│  if (stock > 0) {                 // 第2步：判断                │
│      redis.decr("stock");         // 第3步：扣减                │
│      createOrder();               // 第4步：创建订单            │
│  }                                                              │
└─────────────────────────────────────────────────────────────────┘

时序图 - 1000个请求同时到达：

时间(微秒)  │  请求1   │  请求2   │  请求3   │ ... │  请求1000  │  Redis
────────────┼──────────┼──────────┼──────────┼─────┼────────────┼─────────
            │          │          │          │     │            │ stock=1
   0        │ GET stock│          │          │     │            │
   1        │          │ GET stock│          │     │            │
   2        │          │          │ GET stock│     │            │
   ...      │          │          │          │ ... │            │
   10       │          │          │          │     │ GET stock  │
            │          │          │          │     │            │
            │ 都读到stock=1 !!!                                 │
            │          │          │          │     │            │
   20       │ if(1>0)✓│          │          │     │            │
   21       │          │ if(1>0)✓│          │     │            │
   22       │          │          │ if(1>0)✓│     │            │
   ...      │          │          │          │ ... │            │
   30       │          │          │          │     │ if(1>0)✓  │
            │          │          │          │     │            │
            │ 1000个请求都认为库存充足!!!                        │
            │          │          │          │     │            │
   40       │ DECR     │          │          │     │            │ stock=0
   41       │          │ DECR     │          │     │            │ stock=-1
   42       │          │          │ DECR     │     │            │ stock=-2
   ...      │          │          │          │ ... │            │
   50       │          │          │          │     │ DECR       │ stock=-999
            │          │          │          │     │            │
────────────┴──────────┴──────────┴──────────┴─────┴────────────┴─────────

结果：库存=1件，但创建了1000个订单！超卖999件！！！
```

### 为什么会这样？

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  根本原因：GET 和 DECR 是两个独立的命令，中间可以被其他请求打断                    │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  时间窗口问题：                                                  │
│                                                                 │
│  GET stock    ──────────────────────────────────  DECR stock    │
│       ↑                    ↑                          ↑         │
│       │                    │                          │         │
│       │    这段时间内，其他请求也在执行GET              │         │
│       │    它们都能读到 stock=1                        │         │
│       │                                               │         │
│  读取库存              判断逻辑              扣减库存            │
│                                                                 │
│  这个"时间窗口"可能只有几微秒，但高并发下足以让                   │
│  成百上千个请求同时读到相同的库存值！                            │
└─────────────────────────────────────────────────────────────────┘
```

### 对比：Lua脚本为什么不会超卖

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Lua脚本：所有操作在Redis内部原子执行，中间不可能被打断                           │
└─────────────────────────────────────────────────────────────────────────────────┘

时序图 - Lua脚本执行：

时间(微秒)  │  请求1           │  请求2           │  请求1000        │
────────────┼──────────────────┼──────────────────┼──────────────────┤
            │                  │                  │                  │
   0        │ ┌──────────────┐│                  │                  │
            │ │ Lua脚本执行  ││  等待...          │  等待...          │
            │ │ GET stock=1  ││                  │                  │
            │ │ if(1>0)      ││                  │                  │
            │ │ DECR stock=0 ││                  │                  │
            │ │ return 成功   ││                  │                  │
            │ └──────────────┘│                  │                  │
   100      │                  │ ┌──────────────┐│                  │
            │                  │ │ Lua脚本执行  ││  等待...          │
            │                  │ │ GET stock=0  ││                  │
            │                  │ │ if(0>0) 失败 ││                  │
            │                  │ │ return 库存不足                  │
            │                  │ └──────────────┘│                  │
   200      │                  │                  │ ┌──────────────┐│
            │                  │                  │ │ Lua脚本执行  ││
            │                  │                  │ │ GET stock=0  ││
            │                  │                  │ │ return 库存不足
            │                  │                  │ └──────────────┘│

结果：只有请求1抢到，999个请求被正确拒绝！
```

## 问题二：不用Guava，只用Redis+Lua，系统会怎样？

### 答案：可以成功防超卖，但系统会非常慢，可能崩溃！

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  场景：1万库存，50万用户同时抢，不使用Guava本地限流                               │
└─────────────────────────────────────────────────────────────────────────────────┘

                    50万请求/秒
                         │
                         ▼
          ┌──────────────────────────────┐
          │         服务器集群            │
          │   没有本地限流               │
          │   50万请求全部放行!          │
          └──────────────┬───────────────┘
                         │
                         ▼
          ┌──────────────────────────────┐
          │          Redis               │
          │                              │
          │  50万请求排队执行Lua脚本      │
          │                              │
          │  每个Lua脚本执行: 0.1ms       │
          │  50万 × 0.1ms = 50秒!!!      │
          │                              │
          │  请求1: 0ms 执行              │
          │  请求2: 0.1ms 等待后执行      │
          │  请求3: 0.2ms 等待后执行      │
          │  ...                         │
          │  请求50万: 50秒后才执行!!!    │
          │                              │
          └──────────────────────────────┘

结果分析：
┌─────────────────────────────────────────────────────────────────┐
│  1. 超卖？ → 不会！Lua脚本保证原子性                            │
│                                                                 │
│  2. 响应时间？ → 大部分用户等待50秒才收到响应!!!                │
│                                                                 │
│  3. 服务器状态？                                                │
│     - 50万个HTTP连接挂起                                        │
│     - 内存耗尽 (每个连接占用内存)                               │
│     - 线程池耗尽                                                │
│     - 连接超时，大量请求失败                                    │
│     - 可能OOM崩溃!!!                                            │
│                                                                 │
│  4. 用户体验？ → 灾难级别！页面转圈50秒或直接报错               │
└─────────────────────────────────────────────────────────────────┘
```

└────────────────────────┬───────────────────────────────────────┘

│  第2层: Guava本地限流 (3台服务器，每台1万)                      │
│                                                                │
│  放行: 3万   拒绝: 2万 (返回"系统繁忙")                        │
└────────────────────────┬───────────────────────────────────────┘
                         │ 3万请求
                         ▼
┌────────────────────────────────────────────────────────────────┐
│  第3层: Redis分布式限流                                        │
│  精确控制1万/秒                                                │
│                                                                │
│  放行: 1万   拒绝: 2万 (返回"系统繁忙")                        │
└────────────────────────┬───────────────────────────────────────┘
                         │ 1万请求
                         ▼
┌────────────────────────────────────────────────────────────────┐
│  第4层: Lua脚本扣库存                                          │
│                                                                │
│  1万请求，执行时间: 1万 × 0.1ms = 1秒                          │
│  响应时间: 最慢1秒内返回                                       │
│  用户体验: 良好!!!                                             │

┌───────────────┬────────────────┬────────────────┐

## 问题三：Kafka的作用是什么？

### 理解误区澄清

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  误区：Kafka是容器，用来部署服务？                                               │
│  正确：Kafka是消息队列，用来传递消息！                                           │
└─────────────────────────────────────────────────────────────────────────────────┘

Kafka ≠ 容器 (如Docker/K8s)
Kafka = 消息队列 (如同一个"邮局")

类比:
┌─────────────────────────────────────────────────────────────────┐
│  你寄信:                                                        │
│  1. 写好信 → 交给邮局 → 邮局暂存 → 投递员送到收件人              │
│                                                                 │
│  Kafka:                                                         │
│  1. 发消息 → 发到Kafka → Kafka暂存 → 消费者取走处理             │
└─────────────────────────────────────────────────────────────────┘
```

### Kafka在架构中的位置

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  为什么Kafka在Redis之后？                                                        │
└─────────────────────────────────────────────────────────────────────────────────┘

用户抢购请求流程:

┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
│  用户   │───>│  服务器 │───>│  Redis │───>│  Kafka │───>│ 消费者  │
└────────┘    └────────┘    └────────┘    └────────┘    └────────┘
                 │              │             │             │
                 │              │             │             │
              限流+校验      扣库存        发消息       异步处理
              (同步,快)     (同步,快)     (同步,快)    (异步,慢)
                 │              │             │             │
                 │              │             │             ▼
                 │              │             │      ┌────────────┐
                 │              │             │      │ 扣积分      │
                 │              │             │      │ 发优惠券    │
                 │              │             │      │ 写数据库    │
                 │              │             │      │ 发通知      │
                 │              │             │      └────────────┘
                 │              │             │           │
                 ▼              ▼             ▼           │
              <10ms          <5ms          <5ms        可能5秒
                 │                                        │
                 └──────────── 用户立即收到响应 ────────────┘
                                "抢购成功,正在处理中"
                                (不用等5秒!)
```

### 什么是异步处理？

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  同步 vs 异步                                                                    │
└─────────────────────────────────────────────────────────────────────────────────┘

同步处理 (不用Kafka):
┌─────────────────────────────────────────────────────────────────┐
│  用户请求 → 限流 → 扣库存 → 扣积分 → 发券 → 写数据库 → 返回     │
│                                                                 │
│  总耗时: 10ms + 5ms + 100ms + 50ms + 200ms = 365ms             │
│  用户等待: 365ms                                                │
│                                                                 │
│  问题: 用户等太久！1万请求 × 365ms = 系统崩溃！                  │
└─────────────────────────────────────────────────────────────────┘

异步处理 (用Kafka):
┌─────────────────────────────────────────────────────────────────┐
│  用户请求 → 限流 → 扣库存 → 发Kafka消息 → 立即返回              │
│                     │                       │                   │
│  总耗时: 10ms + 5ms + 5ms = 20ms           │                   │
│  用户等待: 20ms  超快!!!                    │                   │
│                                            │                   │
│                                            ▼                   │
│                              Kafka消费者(后台慢慢处理):          │
│                              扣积分 → 发券 → 写数据库            │
│                              (用户不用等这些!)                   │
└─────────────────────────────────────────────────────────────────┘
```

### 什么是削峰填谷？

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│  削峰填谷 = 把瞬时高峰"削平"，变成平稳的流量                                          │
└─────────────────────────────────────────────────────────────────────────────────────┘

没有Kafka (直接处理):
                              
请求量│     ████                 
      │    ██████    ← 瞬时高峰1万请求/秒
      │   ████████               
      │  ██████████  
      │ ████████████             
      │██████████████            
      └──────────────────────────── 时间
                │
                ▼
         数据库承受1万请求/秒
         → 数据库崩溃!!!


有Kafka (异步处理):

请求量│     ████                 
      │    ██████    ← 瞬时高峰1万请求/秒
      │   ████████               
      │  ██████████  
      │ ████████████             
      │██████████████            
      └──────────┬───────────────── 时间
                 │
                 ▼
            Kafka队列暂存
                 │
                 ▼
处理量│ ─────────────────────────   
      │                          ← 消费者匀速处理1000/秒
      │                              
      └──────────────────────────── 时间
                 │
                 ▼
         数据库只承受1000请求/秒
         → 数据库健康!!!


形象比喻:
┌─────────────────────────────────────────────────────────────────┐
│  水库原理:                                                      │
│                                                                 │
│  暴雨 (高峰请求)                                                │
│      ↓                                                          │
│  ┌───────────┐                                                  │
│  │  水库      │  ← Kafka (暂存请求)                             │
│  │  暂存     │                                                  │
│  └─────┬─────┘                                                  │
│        ↓                                                        │
│    ═══════════                                                  │
│    平稳水流 (匀速处理)                                           │
│        ↓                                                        │
│    农田灌溉 (数据库)                                             │
│                                                                 │
│  没有水库: 暴雨直接冲毁农田                                      │
│  有水库: 暴雨存起来，慢慢放水                                    │
└─────────────────────────────────────────────────────────────────┘
```

### Kafka和Guava限流是两码事

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Guava限流 和 Kafka 各自的作用                                                   │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Guava限流:                                                     │
│  - 作用: 拦截请求，直接拒绝                                      │
│  - 时机: 请求进来的时候                                          │
│  - 结果: 被拒绝的请求直接返回"系统繁忙"                          │
│                                                                 │
│  Kafka:                                                         │
│  - 作用: 暂存任务，慢慢处理                                      │
│  - 时机: 请求通过限流后                                          │
│  - 结果: 请求不会被拒绝，只是排队等处理                          │
└─────────────────────────────────────────────────────────────────┘

两者配合:
┌─────────────────────────────────────────────────────────────────┐
│  100万请求                                                      │
│      │                                                          │
│      ▼                                                          │
│  Guava: 放行1万，拒绝99万 (返回"繁忙")                          │
│      │                                                          │
│      ▼                                                          │
│  Redis: 扣库存成功1万                                           │
│      │                                                          │
│      ▼                                                          │
│  Kafka: 暂存1万个"发券任务"                                     │
│      │                                                          │
│      ▼                                                          │
│  消费者: 匀速处理，每秒1000个，10秒处理完                        │
└─────────────────────────────────────────────────────────────────┘
```

### Kafka的角色总结

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Kafka在秒杀系统中的三大作用                                                     │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  作用1: 异步解耦                                                │
│  ─────────────────────────────────────────                      │
│  抢购服务 ──发消息──> Kafka ──消费──> 发券服务                  │
│                                    ──消费──> 积分服务           │
│                                    ──消费──> 通知服务           │
│                                                                 │
│  好处: 各服务独立，互不影响                                      │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  作用2: 削峰填谷                                                │
│  ─────────────────────────────────────────                      │
│  瞬时1万请求 ──> Kafka暂存 ──> 匀速1000/秒消费                  │
│                                                                 │
│  好处: 保护下游数据库                                           │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  作用3: 失败重试                                                │
│  ─────────────────────────────────────────                      │
│  发券失败 ──> 消息回到Kafka ──> 稍后重试                        │
│                                                                 │
│  好处: 保证最终成功                                             │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  四个问题的答案                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Q1: 不用Lua，GET+DECR超卖多少？                                │
│  A1: 可能超卖成百上千件！不是2件！                               │
│      因为高并发下，大量请求同时读到相同库存值                    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Q2: 不用Guava，只用Redis+Lua？                                 │
│  A2: 不会超卖，但系统会崩溃！                                    │
│      50万请求全打到Redis，响应时间50秒，服务器OOM               │
│      必须用多层限流(Nginx+Guava+Redis)层层削减                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Q3: Kafka是什么？在哪里？                                      │
│  A3: Kafka是消息队列，不是容器!                                 │
│      位置: 在Redis扣库存之后，用于异步处理后续任务               │
│      作用: 异步解耦 + 削峰填谷 + 失败重试                       │
│      与Guava无关: Guava管限流，Kafka管异步                      │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Q4: 当前方案最优吗？                                           │
│  A4: 非常好(4/5星)！行业主流方案                                │
│      Redis: 内存型，QPS可达10万+，瓶颈在网络                    │
│      Kafka: 消息队列，吞吐百万级，瓶颈在磁盘                    │
│      硬件: 中等配置即可支撑5-10万QPS                            │
└─────────────────────────────────────────────────────────────────┘
```
