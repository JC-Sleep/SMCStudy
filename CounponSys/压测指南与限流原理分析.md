# 压测指南与限流原理深度分析

## 一、QPS由什么决定？

### 1.1 影响QPS的因素

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          QPS = 并发用户数 / 平均响应时间                          │
└─────────────────────────────────────────────────────────────────────────────────┘

                    QPS受以下因素影响（木桶效应，取最短板）
                                    │
        ┌───────────────────────────┼───────────────────────────┐
        │                           │                           │
        ▼                           ▼                           ▼
┌───────────────┐           ┌───────────────┐           ┌───────────────┐
│   硬件资源     │           │   软件架构     │           │   中间件       │
├───────────────┤           ├───────────────┤           ├───────────────┤
│ • CPU核数     │           │ • 代码效率     │           │ • Redis性能   │
│ • 内存大小     │           │ • 线程池配置   │           │ • Kafka吞吐   │
│ • 网络带宽     │           │ • 连接池大小   │           │ • 数据库连接   │
│ • 磁盘IO      │           │ • 锁的粒度     │           │ • 网络延迟     │
└───────────────┘           └───────────────┘           └───────────────┘
```

### 1.2 各组件理论QPS上限

| 组件 | 单机理论QPS | 瓶颈点 | 你的场景 |
|------|------------|--------|---------|
| **Nginx** | 10-20万/秒 | CPU | 前置负载均衡 |
| **Spring Boot** | 1-3万/秒 | CPU + 线程 | 业务逻辑 |
| **Redis** | 10万+/秒 | 内存 + CPU | 库存扣减 |
| **Kafka** | 100万+/秒 | 磁盘IO | 消息队列 |
| **Oracle** | 1-2千/秒 | 磁盘IO + 锁 | 数据持久化 |

**结论**: 系统瓶颈通常在 **数据库** 或 **业务代码**！

### 4.4 代码层面的先来先得

```java
/**
 * Guava RateLimiter源码简化版
 * 展示"先来先得"的实现原理
 */
public class RateLimiter {
    
    // 令牌数量(可以是小数)
    private double storedPermits;
    
    // 上次获取令牌的时间
    private long nextFreeTicketMicros;
    
    /**
     * 尝试获取令牌
     * 关键: 这是一个同步方法(synchronized)
     * 所有请求排队等待，先到先执行
     */
    public synchronized boolean tryAcquire(long timeout, TimeUnit unit) {
        long timeoutMicros = unit.toMicros(timeout);
        long nowMicros = System.nanoTime() / 1000;
        
        // 计算需要等待的时间
        long waitMicros = reserveNextTicket(1, nowMicros);
        
        // 如果等待时间超过超时时间，返回false(拒绝)
        if (waitMicros > timeoutMicros) {
            return false;  // ← 没拿到令牌，被限流!
        }
        
        // 等待获取令牌
        sleepMicrosUninterruptibly(waitMicros);
        return true;  // ← 拿到令牌，放行!
    }
}
```

### 5.2 当前系统的方案

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  本项目采用: 先来先得 + VIP检查                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘

代码位置: SeckillController.java

@PostMapping("/grab")
public ApiResponse<SeckillGrabResult> grabCoupon(...) {
    
    // 第1层: 全局限流(先来先得)
    if (!seckillRateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
        throw SeckillException.systemBusy();
    }
    
    // 第2层: IP限流(防攻击)
    if (!distributedRateLimiter.tryAcquire("limit:ip:" + clientIp, 100, 1)) {
        throw SeckillException.rateLimitExceeded();
    }
    
    // 第3层: 用户限流(防刷)
    if (!distributedRateLimiter.tryAcquire("limit:user:" + userId, 10, 1)) {
        throw SeckillException.rateLimitExceeded();
    }
    
    // 业务处理中检查VIP
    // 如果活动设置了vipOnly=1，非VIP用户会被拒绝
    return seckillService.grabCoupon(request);
}

┌─────────────┬─────────┬─────────┬─────────┬─────────┬─────────┬─────────┐

├─────────────┼─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤

指标解读:
• Samples: 总请求数
• Average: 平均响应时间 (越低越好)
• Median: 中位数响应时间 (50%请求在此时间内完成)
• 90% Line: 90%请求在此时间内完成
• 99% Line: 99%请求在此时间内完成 (长尾)
• Error%: 错误率 (应<1%)

健康指标:
✅ 平均响应时间 < 200ms
✅ 错误率 < 1%
✅ 90%Line < 500ms
```

## 七、性能优化建议

### 7.1 根据压测结果优化

| 问题 | 表现 | 解决方案 |
|------|------|---------|
| CPU高 | CPU>80% | 增加服务器/优化代码 |
| 内存高 | 频繁GC | 增加JVM堆内存 |
| 响应慢 | Avg>500ms | 优化慢查询/加缓存 |
| 错误多 | Error>1% | 检查连接池/超时配置 |

### 7.2 JVM调优

```bash
# 启动参数优化
java -jar coupon-sys.jar \
  -Xms4g -Xmx4g \           # 堆内存4G
  -XX:+UseG1GC \            # 使用G1垃圾回收
  -XX:MaxGCPauseMillis=100  # GC暂停时间目标100ms
```

### 7.3 数据库连接池

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 50      # 最大连接数
      minimum-idle: 10           # 最小空闲连接
      connection-timeout: 3000   # 连接超时3秒
```
