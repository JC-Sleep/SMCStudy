## 一、系统现有瑕疵（诚实分析）

### 🔴 高优先级瑕疵（会被面试官直接问到）

#### 1. 兑换码发券的分布式事务问题（最大漏洞）

**问题描述**：
```
CAS更新码状态（STATUS: 0→1）  ← Oracle事务
         ↓ 成功
grantCoupon() 发放优惠券       ← 另一个DB操作

中间网络超时 or 服务崩溃 →
  STATUS=1（码已标记使用）
  但券没有发放成功
  → 用户码用掉了，什么都没拿到 ❌
```

**当前状态**：未处理，代码里直接顺序调用，没有补偿机制。

**修复方案（三选一）**：
```
方案A（推荐）：本地消息表 + 定时重试
  1. CAS成功后，在同一事务里写一条"待发券"记录到 T_GRANT_TASK 表
  2. 定时Job扫描 T_GRANT_TASK，调用 grantCoupon()
  3. 成功后标记Task=完成；失败自动重试3次；3次后人工告警

方案B：Saga补偿
  CAS成功 → 发券失败 → 回滚：把 STATUS 从1改回0（或改成特殊状态5=发券失败）
  问题：回滚窗口期内用户看到"已使用"，体验差

方案C：强一致性（性能差）
  用 Oracle 分布式事务锁住两个操作，性能下降30%+，不推荐
```

**面试必问**：CAS成功后发券失败怎么办？（这是系统最大的设计漏洞，必须补上答案）

#### 2. 无统一身份认证（API安全空白）

**问题描述**：
- 所有API（`/api/redeem-code/redeem`、`/api/seckill/grab`）没有JWT/OAuth2鉴权
- 任何人知道接口地址就可以调用
- 即使有IP限速，没有合法Token的请求也能通过前4层

**修复方案**：
```yaml
# Spring Security + JWT
dependencies:
  - spring-boot-starter-security
  - jjwt-api

# 兑换接口需要用户Token（APP登录后下发）
# 批量生成接口需要管理员Token（后台系统）
# Druid控制台必须加HTTPS + 强密码（不只是IP白名单）
```

**现状评级**：⚠️ 生产上线前必须补充，面试说"已规划，优先级P0"

#### 3. 兑换码过期状态Job缺失

**问题描述**：
- `T_COUPON_REDEEM_CODE` 表有 `EXPIRE_TIME` 字段
- 但没有定时Job批量扫描并将过期码状态更新为2（过期）
- 当前只在用户提交时懒检查，DB里大量码状态还是0（未用）但实际已过期
- 影响：统计数据不准（"激活码还有多少张"不准确）

**修复方案**：
```java
// 在 CouponScheduledJob 里新增，每小时执行一次
@Scheduled(cron = "0 0 * * * ?")
public void expireRedeemCodes() {
    // UPDATE T_COUPON_REDEEM_CODE
    // SET STATUS=2, UPDATE_TIME=SYSTIMESTAMP
    // WHERE STATUS=0 AND EXPIRE_TIME < SYSTIMESTAMP
    // AND ROWNUM <= 5000  ← 每次最多处理5000条，防止大事务
    redeemCodeMapper.batchExpire(5000);
    log.info("[兑换码过期Job] 批量标记完成");
}
```

#### 4. 同一用户对同一批次可重复兑换多张码

**问题描述**：
- 一个批次可能有100张码，A用户拿到了3张
- 当前没有限制"同一用户在同一批次里只能兑换N张"
- 羊毛党可以拿多张码全部兑换，套取多张优惠券

**修复方案**：
```sql
-- DB层：加唯一索引或限制查询
-- 方案A：一个用户只能兑换同一批次1张（严格）
CREATE UNIQUE INDEX UK_USER_BATCH ON T_COUPON_REDEEM_CODE(USER_ID, BATCH_ID)
  WHERE STATUS=1;  -- Oracle函数索引（只对已兑换生效）

-- 方案B：批次配置 MAX_PER_USER 字段
ALTER TABLE T_REDEEM_CODE_BATCH ADD MAX_PER_USER NUMBER(5) DEFAULT 1;

-- 兑换前校验
SELECT COUNT(*) FROM T_COUPON_REDEEM_CODE
WHERE USER_ID=? AND BATCH_ID=? AND STATUS=1
HAVING COUNT(*) >= (SELECT MAX_PER_USER FROM T_REDEEM_CODE_BATCH WHERE ID=?);
```

#### 7. Kafka消费者缺少死信队列处理

**问题**：秒杀订单消费失败重试3次后，消息丢进Kafka死信Topic，没有监控和补偿处理。

**现状**：`KafkaLagMonitor` 只监控积压，没有监控死信队列里有多少失败消息。

**解决**：
```java
// 监听死信Topic，记录+告警+人工补偿
@KafkaListener(topics = "seckill-order-dlt")  // dlt = Dead Letter Topic
public void handleDeadLetter(SeckillOrderMessage msg) {
    log.error("[死信消息] orderId={} 已失败3次，需要人工处理", msg.getOrderId());
    alertService.sendCriticalAlert("死信队列有消息积压，请检查！");
    deadLetterMapper.save(msg);  // 持久化，不要丢失
}
```

#### 8. requestId幂等方案只有设计，缺实现

**问题**：文档里提到"前端带requestId防止重试看到已被使用"，但没有对应代码实现。

**需要实现的**：
```java
@PostMapping("/redeem")
public ApiResponse<Long> redeem(@RequestBody RedeemRequest req) {
    // 1. 检查requestId是否已处理过
    String cacheKey = "redeem:idem:" + req.getRequestId();
    String cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return ApiResponse.success(Long.valueOf(cached));  // 幂等返回
    }
    // 2. 正常处理
    Long couponId = redeemCodeService.redeem(req.getCode(), req.getUserId(), ...);
    // 3. 缓存结果5分钟（用户重试窗口）
    redisTemplate.opsForValue().set(cacheKey, couponId.toString(), 5, TimeUnit.MINUTES);
    return ApiResponse.success(couponId);
}
```

#### 9. HMAC密钥轮换机制缺失

**问题**：如果HMAC密钥泄露（员工离职、配置文件泄露），所有历史码的签名算法对攻击者公开，可以批量伪造。

**解决**：
```
码格式扩展：CP-{version}-{data}-{sign}
例如：CP-v2-2K5XM1-A3F7
   version字段标识用哪个版本密钥验签

密钥管理：
  v1密钥（旧）→ 只用于验签旧码，不再生成新码
  v2密钥（新）→ 新码全部用v2生成
  密钥存储：HashiCorp Vault 或 K8s Secret
```

## 三、面试高频题库与答案

### 🔥 必问题（会被直接问）

---

**Q1：你的秒杀系统如何防止超卖？**

```
三层防御：

第一层 —— Redis Lua脚本原子扣减（最核心）：
  库存预热到Redis，用户抢购时执行Lua脚本：
  check库存>0 + check用户限购 + DECR库存，三步原子执行
  任何一步失败整个脚本回滚，绝对不会超卖

第二层 —— Kafka异步落库 + 状态机：
  抢购成功只是"预占库存"，生成status=0的待处理订单
  Kafka消费者真正处理：扣DB库存 + 扣积分 + 发券
  失败则回滚Redis库存，回到可抢状态

第三层 —— DB CAS：
  UPDATE ... SET STOCK=STOCK-1 WHERE STOCK>0 AND ACTIVITY_ID=?
  affected=0说明库存已被抢完，兜底防止Redis与DB不一致时的超卖
```

---

**Q2：Redis和DB库存不一致怎么处理？**

```
两种不一致场景：

场景A：Redis库存=0，但DB库存>0（有人回滚了）
→ 智能判断：读DB库存 + 待处理订单数
→ DB>0 → 返回"活动火爆，请刷新重试"（而非"已售罄"）
→ 用户刷新后Redis库存已被回滚补充，可以继续抢

场景B：Redis库存>0，但DB库存=0（Kafka消费快于Redis同步）
→ DB CAS兜底：UPDATE WHERE STOCK>0 → affected=0 → 失败
→ 回滚Redis库存+1，下次用户请求Redis也会失败

定期对账：每5分钟Job对比Redis库存和DB库存，超过阈值差异告警
```

---

**Q3：Kafka消费失败了怎么办？（重试+死信）**

```
三级重试策略：

第1次失败 → 立即重试（网络抖动）
第2次失败 → 延迟5秒重试（DB短暂不可用）
第3次失败 → 进入死信队列（Dead Letter Topic）

死信队列处理：
  1. 持久化到DB（T_DEAD_LETTER表）
  2. 发送告警（Slack/钉钉）
  3. 人工补偿或定时重试

补偿保证幂等：消费者处理前先查DB订单状态，
  status=1（已成功）→ 直接跳过，防止重复消费重复发券
```

---

**Q4：Redisson分布式锁和DB CAS都用了，是否过度设计？**

```
两层设计各有目的，不冗余：

Redisson分布式锁（第4层）：
  针对"同一个码的并发请求"
  防止：用户双击 or 网络超时重试打到不同实例
  作用：同一码同时只有1个线程进入业务逻辑
  代价：需要Redis（加一次网络往返）

DB CAS（第5层）：
  针对"分布式锁失效的极端情况"
  防止：Redis宕机 / 锁实现有BUG / 第4层被绕过
  作用：数据库原子行锁，天然防并发，不依赖外部组件
  代价：多一次DB写

去掉哪层都有风险：
  只有锁没有CAS → Redis宕机时重复兑换
  只有CAS没有锁 → 高并发时所有请求打到DB，CAS冲突率高性能差
  两层一起：先用锁减少DB并发压力，再用CAS保证最终正确性
```

---

**Q5：你的HMAC签名能防止所有攻击吗？**

```
诚实回答：只防格式层攻击，不是万能的。

HMAC能防：
  ✅ 随机枚举/暴力猜码（签名4位=36⁴≈167万种，配合限速，破解需2799小时）
  ✅ 伪造新码（无私钥无法生成合法签名）

HMAC不能防：
  ❌ 重放真实码（这是DB CAS的职责）
  ❌ 内部人员泄露码（属于管理问题）
  ❌ 密钥泄露后伪造（需要密钥版本轮换机制）

所以系统是分层设计：
  格式合法性 → HMAC（第1层）
  重复使用 → DB CAS（第5层）
  两层职责不同，缺一不可
```

**Q6：3实例部署，兑换码防刷体系还有效吗？**

```
完全有效，因为所有状态存在共享存储：

IP/用户限速 → Redis共享，三实例原子INCR同一个Key

唯一需要注意的：

  用K8s Secret统一注入：

  禁止在各实例application.yml里各自配置
```

**Q7：什么是CAS？为什么不用SELECT FOR UPDATE（加锁）？**

```
CAS = Compare And Swap（比较并交换）
一条UPDATE带WHERE条件实现原子状态变更：
  UPDATE T_COUPON_REDEEM_CODE
  SET STATUS=1, USER_ID=?
  WHERE CODE=? AND STATUS=0   ← 只在未使用时才更新

affected=1 → 成功（当前请求赢得了兑换权）
affected=0 → 失败（已被其他请求改变STATUS，CAS检测到冲突）

为什么不用 SELECT FOR UPDATE（悲观锁）：
  SELECT FOR UPDATE需要：先SELECT加锁 → 业务判断 → UPDATE → COMMIT解锁
  高并发下后续请求全部阻塞等待，串行化，性能差
  还有死锁风险（异常没有释放锁）

CAS优势：
  无等待（失败直接返回，不阻塞其他请求）
  无死锁（不持有锁）
  性能高（兑换码场景STATUS只有0→1一次跳转，CAS冲突率极低）
```

**Q8：Kafka为什么比ActiveMQ更适合这个系统？**

```
对比三个关键维度：

吞吐量：
  ActiveMQ: 1-2万条/秒（基于JMS，内存模式）
  Kafka: 10-100万条/秒（顺序写磁盘，零拷贝）
  → 5-10W QPS的秒杀场景，ActiveMQ直接成为瓶颈

消费模式：
  Kafka: 消费者组，多分区并行消费，concurrency=20
  ActiveMQ: 点对点，并行度有限
  → Kafka消费速度是ActiveMQ的10倍+

回溯能力：
  Kafka: 消息保留7天，消费失败可以reset offset重消费
  ActiveMQ: 消费后即删除，不可回溯
  → 生产故障后，Kafka可以从故障点重新消费，ActiveMQ数据丢失

代价：
  Kafka不支持原生延迟消息（优惠券过期延迟处理用定时Job替代）
  运维复杂度略高（需要ZooKeeper / KRaft）
```

**Q9：动态限流怎么实现的？为什么不用固定限流？**

```
固定限流的问题：
  活动开始：库存10000，限流10000 QPS → 合理
  活动后期：库存只剩100，还是允许10000 QPS打进来
  → 9900个请求在Redis做无效Lua执行（扣减失败）
  → Redis白白消耗了90%的计算资源

动态限流实现（DynamicRateLimiterManager）：
  1. 每5秒读取Redis库存
  2. 新限流速率 = 剩余库存 × 1.5倍（留20%缓冲防止回滚）
  3. 调用 rateLimiter.setRate(newRate) 动态调整

效果：
  库存10000 → 限流15000 QPS
  库存1000  → 限流1500 QPS
  库存0     → 限流100 QPS（只允许少量请求到智能判断逻辑）

  Redis无效Lua执行↓87.5%，Redis压力↓70%
```

**Q10：项目中遇到的最难的技术问题是什么？怎么解决的？**

```
推荐答案：Kafka回滚库存后的"幽灵售罄"问题

现象：
  1万人抢购，Redis库存被扣至0，返回"已售罄"
  5秒后Kafka消费，100人失败 → 回滚Redis库存+100
  但前5秒已经告诉用户"已售罄"，用户不再重试 → 100张券浪费

分析：
  根本原因：Redis库存=0不等于"真正售罄"
  可能情况：① 临时回滚 ② DB真正为0

解决：智能库存不足判断（handleStockInsufficient）
  读取三个信号：Redis库存、DB库存、待处理订单数
  Redis=0 + DB>0 → "活动火爆，请稍后刷新重试"（提示可重试）
  Redis=0 + DB=0 + 待处理=0 → "已售罄"（真正售罄）

效果：
  库存利用率从99%提升到100%
  用户转化率提升（友好提示引导重试）
```

**Q11：手写Redis分布式锁（不用Redisson）**

```java
// 加锁：SET key value NX EX（原子操作）
String lockKey = "redeem:lock:" + code;
String lockValue = UUID.randomUUID().toString();
Boolean locked = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, lockValue, 5, TimeUnit.SECONDS);

if (!Boolean.TRUE.equals(locked)) {
    throw new CouponException(429, "请求频繁，请稍后重试");
}

try {
    // 业务逻辑
    doRedeem(code, userId);
} finally {
    // 解锁：Lua脚本保证原子性（只删自己的锁）
    String luaScript =
        "if redis.call('get',KEYS[1])==ARGV[1] then " +
        "return redis.call('del',KEYS[1]) else return 0 end";
    redisTemplate.execute(new DefaultRedisScript<>(luaScript, Long.class),
        Collections.singletonList(lockKey), lockValue);
}

// 为什么不直接 DEL key：
// 场景：锁超时自动释放，此时其他请求拿到锁，
// 当前请求执行完再DEL，会把别人的锁删掉 → 必须比对value
```

---

**Q12：Lua脚本为什么能保证原子性？**

```
Redis是单线程执行模型：
  客户端发送Lua脚本 → Redis用单线程执行整个脚本 → 执行期间不处理其他请求
  相当于把"检查+修改"合并成一个不可分割的操作

与事务对比：
  Redis MULTI/EXEC 是乐观锁，执行期间不阻塞，EXEC时可能因WATCH失败回滚
  Lua脚本是真正原子：执行中间不可能插入其他命令

秒杀Lua脚本原子保证：
  check库存 + check限购 + DECR库存 三步合一
  任何一步check失败 → return错误码（不执行DECR）
  绝对不会出现：库存check通过了，但DECR前被其他请求抢先DECR到0
```

**Q13：如何设计一个高可用的兑换码生成方案（100万张）？**

```
问题：100万张码，单线程HMAC计算 + DB插入，约需10分钟

优化方案：
  1. 异步批量生成（后台Job）：
     管理员点击"生成"→ 立即返回taskId → 后台Job处理
     Job分片：10个线程，每线程生成10万张
     进度查询：GET /api/redeem-code/batch/{taskId}/progress

  2. 批量插入优化：
     每1000张码，MyBatis-Plus batchInsert一次（减少DB往返）
     约需：100万/1000 = 1000次batch → 每次约100ms → 约100秒

  3. 生成速度估算：
     HMAC计算：每张约0.1ms → 100万张 = 100秒
     DB插入：每批1000条约50ms → 1000批 = 50秒
     10线程并行：总计约15秒完成100万张码

  4. 码的唯一性保证：
     序列号用Redis INCRBY(1000)批量预取（减少Redis访问）
     CODE列有UNIQUE约束（DB层兜底）
```

```
这个系统我认为有三个核心亮点：

第一，秒杀防超卖的分层设计：

  比如Kafka消费失败回滚库存后的"幽灵售罄"问题，
  分布式锁和CAS之间微妙的职责分工，
  FAIL_COUNT只应该在CAS并发失败时累加的精细化设计——
```
