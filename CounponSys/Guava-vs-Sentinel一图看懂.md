# Guava vs Sentinel 一图看懂

> **创建时间**: 2026-03-26  
> **目的**: 快速决策使用Guava还是Sentinel

---

## 🎯 一句话总结

**Guava**：轻量高性能，适合5-10万QPS的单体应用  
**Sentinel**：功能强大，适合10万+QPS的微服务体系

## 🔍 深度对比

### 1. 架构对比

#### Guava架构（单机）
```
┌────────────────────────┐
│  应用服务器             │
│  ┌──────────────────┐  │
│  │  业务代码        │  │
│  │      ↓           │  │
│  │  Guava限流器     │  │
│  │  (内存操作)      │  │
│  └──────────────────┘  │
└────────────────────────┘

优点：
✅ 零依赖
✅ 性能极高
✅ 部署简单

缺点：
❌ 单机限流（多实例无法协调）
❌ 无可视化
```

#### Sentinel架构（嵌入模式）
```
┌────────────────────────┐
│  应用服务器             │
│  ┌──────────────────┐  │
│  │  业务代码        │  │
│  │      ↓           │  │
│  │  Sentinel Client │  │
│  │  (内存+网络)     │  │
│  └─────────┬────────┘  │
└────────────┼───────────┘
             │
             ↓
┌────────────────────────┐
│  Sentinel Dashboard    │
│  (可选，监控+规则管理)  │
└────────────────────────┘

优点：
✅ 可视化监控
✅ 动态规则推送
✅ 功能丰富

缺点：
❌ 需要Dashboard
❌ 学习成本高
❌ 仍是单机限流
```

#### Sentinel架构（Token Server模式）
```
┌────────────┐  ┌────────────┐
│  应用实例1  │  │  应用实例2  │
└─────┬──────┘  └──────┬─────┘
      │                │
      └────────┬───────┘
               ↓
┌────────────────────────┐
│  Token Server          │
│  (全局令牌桶)          │
└───────────┬────────────┘
            │
            ↓
┌────────────────────────┐
│  Sentinel Dashboard    │
└────────────────────────┘

优点：
✅ 全局精确限流
✅ 多实例协调

缺点：
❌❌ 需要独立Token Server
❌❌ 高可用成本高
❌❌ 网络开销大
```

### 2. 代码复杂度对比

#### Guava代码（简单）
```java
// 配置（1行）
@Bean
public RateLimiter rateLimiter() {
    return RateLimiter.create(10000);
}

// 使用（3行）
if (!rateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
    throw SeckillException.systemBusy();
}
// 业务逻辑...
```

#### Sentinel代码（复杂）
```java
// 配置（10+行）
@PostConstruct
public void initRules() {
    List<FlowRule> rules = new ArrayList<>();
    FlowRule rule = new FlowRule();
    rule.setResource("seckill_grab");
    rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
    rule.setCount(10000);
    rules.add(rule);
    FlowRuleManager.loadRules(rules);
}

// 使用方式1：编程式（7行）
Entry entry = null;
try {
    entry = SphU.entry("seckill_grab");
    // 业务逻辑...
} catch (BlockException e) {
    throw SeckillException.systemBusy();
} finally {
    if (entry != null) entry.exit();
}

// 使用方式2：注解式（3行+额外方法）
@SentinelResource(value = "seckill_grab", blockHandler = "handleBlock")
public Result grab() {
    // 业务逻辑...
}
// 还需要定义handleBlock方法...
```

**复杂度**：Guava < Sentinel ✅

## 📋 场景推荐

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **单体应用，QPS 5-10万** | Guava ⭐⭐⭐⭐⭐ | 轻量、高性能、够用 |
| **微服务，需要统一管理** | Sentinel | 功能强大、可视化 |
| **需要熔断降级** | Sentinel | 内置熔断功能 |
| **追求极致性能** | Guava | 性能最高 |
| **需要可视化，但不想用Sentinel** | Guava + Prometheus | 通用方案 |
| **QPS 10万+，需要全局限流** | Sentinel + Token Server | 集群协调 |
| **网关层限流** | Nginx / Gateway | 统一入口 |
