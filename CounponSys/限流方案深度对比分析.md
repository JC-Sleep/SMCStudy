## 1. 当前方案：Guava RateLimiter

### 1.1 基本原理

**算法**：令牌桶（Token Bucket）

```
┌─────────────────────────────────────────────────────────────────┐
│  令牌桶原理                                                      │
└─────────────────────────────────────────────────────────────────┘

                    ┌───────────────┐
                    │  令牌生成器    │
                    │ (每秒放10000个)│
                    └───────┬───────┘
                            │
                   令牌  ○ ○ ○ ○ ○
                            │
                            ▼
            ┌───────────────────────────────┐
            │         令牌桶               │
            │   容量: 10000个令牌          │
            │   ┌───────────────────┐      │
            │   │ ○ ○ ○ ○ ○ ○ ○ ○ │      │
            │   └───────────────────┘      │
            └───────────────┬───────────────┘
                            │
              ┌─────────────┼─────────────┐
              │             │             │
              ▼             ▼             ▼
         ┌────────┐    ┌────────┐    ┌────────┐
         │ 请求1  │    │ 请求2  │    │ 请求3  │
         │ 获取令牌│    │ 获取令牌│    │ 无令牌 │
         │   ✓    │    │   ✓    │    │   ✗    │
         │ 放行   │    │ 放行   │    │ 拒绝   │
         └────────┘    └────────┘    └────────┘
```

### 1.2 当前实现

```java
// 配置
@Bean
public RateLimiter seckillRateLimiter() {
    return RateLimiter.create(10000);  // 每秒10000个令牌
}

// 使用
if (!seckillRateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
    throw SeckillException.systemBusy();
}
```

### 1.3 优点 ✅

| 优点 | 说明 | 重要性 |
|------|------|-------|
| **1. 轻量级** | 单个jar包，无需额外服务 | ⭐⭐⭐⭐⭐ |
| **2. 零依赖** | 不依赖Redis、ZooKeeper等中间件 | ⭐⭐⭐⭐⭐ |
| **3. 性能高** | 纯内存操作，无网络开销 | ⭐⭐⭐⭐⭐ |
| **4. 使用简单** | API简洁，学习成本低 | ⭐⭐⭐⭐⭐ |
| **5. 稳定可靠** | Google开源，经过大规模验证 | ⭐⭐⭐⭐⭐ |
| **6. 允许突发** | 令牌桶算法天然支持突发流量 | ⭐⭐⭐⭐ |
| **7. 平滑限流** | 不会出现"锯齿"现象 | ⭐⭐⭐⭐ |

### 1.4 缺点 ❌

| 缺点 | 说明 | 影响程度 |
|------|------|---------|
| **1. 单机限流** | 每个实例独立限流，无法全局控制 | ⚠️ 中等 |
| **2. 无法动态调整** | 需要重启才能修改速率（我们已优化） | ⚠️ 已解决 |
| **3. 无可视化** | 没有监控面板，无法查看实时限流情况 | ⚠️ 轻微 |
| **4. 功能单一** | 只有限流，没有熔断、降级等功能 | ⚠️ 中等 |
| **5. 无规则管理** | 无法动态配置限流规则 | ⚠️ 轻微 |

### 1.5 本项目的优化

我们已经实现了**动态限流管理器**，解决了部分缺点：

```java
// 优化：根据库存动态调整限流速率
public class DynamicRateLimiterManager {
    public void adjustRateIfNeeded(Long activityId) {
        int remainStock = getRemainStock(activityId);
        double newRate = remainStock * 1.5;  // 库存的1.5倍
        limiter.setRate(newRate);  // 动态调整 ✅
    }
}
```

**效果**：
- ✅ 支持动态调整（不需要重启）
- ✅ 根据业务自动调整（库存驱动）
- ✅ Redis压力减少70%

## 2. 方案对比：Sentinel

### 2.1 什么是Sentinel？

**Sentinel** = 阿里巴巴开源的流量控制组件

**核心功能**：
1. 流量控制（限流）
2. 熔断降级
3. 系统负载保护
4. 热点参数限流
5. 实时监控面板

### 2.5 Sentinel代码示例

#### 基础限流
```java
// 1. 添加依赖
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-core</artifactId>
    <version>1.8.6</version>
</dependency>

// 2. 配置限流规则
List<FlowRule> rules = new ArrayList<>();
FlowRule rule = new FlowRule();
rule.setResource("seckill_grab");  // 资源名
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);  // 限流类型：QPS
rule.setCount(10000);  // 阈值：10000
rules.add(rule);
FlowRuleManager.loadRules(rules);

// 3. 使用
Entry entry = null;
try {
    entry = SphU.entry("seckill_grab");
    // 业务逻辑
} catch (BlockException e) {
    // 被限流，拒绝请求
    throw SeckillException.systemBusy();
} finally {
    if (entry != null) {
        entry.exit();
    }
}
```

#### 注解方式（更简洁）
```java
@SentinelResource(
    value = "seckill_grab",
    blockHandler = "handleBlock",
    fallback = "handleFallback"
)
public SeckillGrabResult grabCoupon(SeckillGrabRequest request) {
    // 业务逻辑
}

// 限流处理
public SeckillGrabResult handleBlock(SeckillGrabRequest request, BlockException e) {
    return SeckillGrabResult.fail("系统繁忙，请稍后重试");
}

// 异常降级
public SeckillGrabResult handleFallback(SeckillGrabRequest request, Throwable e) {
    return SeckillGrabResult.fail("服务异常");
}
```

#### 热点参数限流（强大功能）
```java
// 针对不同activityId限流
ParamFlowRule rule = new ParamFlowRule("seckill_grab")
    .setParamIdx(0)  // 第0个参数（activityId）
    .setCount(1000);  // 每个activityId限流1000

// 特殊活动单独限流
rule.setParamFlowItemList(Arrays.asList(
    new ParamFlowItem().setObject("1001").setCount(5000),  // 活动1001限流5000
    new ParamFlowItem().setObject("1002").setCount(3000)   // 活动1002限流3000
));
```

## 3. 其他方案

### 3.1 Nginx限流（推荐⭐⭐⭐⭐⭐）

**优点**：
- 在最前端拦截，保护后端服务
- 性能极高（C语言）
- 配置简单

**示例**：
```nginx
# nginx.conf
http {
    # 定义限流zone
    limit_req_zone $binary_remote_addr zone=seckill:10m rate=10000r/s;
    
    server {
        location /api/seckill/grab {
            # 应用限流
            limit_req zone=seckill burst=20000 nodelay;
            
            proxy_pass http://backend;
        }
    }
}
```

**适用场景**：
- ✅ 防止恶意刷接口
- ✅ 整体流量控制
- ✅ 配合后端限流使用

---

### 3.2 Spring Cloud Gateway限流

**优点**：
- 集成在网关层
- 支持Redis分布式限流
- 配置简单

**示例**：
```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: seckill
          uri: lb://coupon-system
          predicates:
            - Path=/api/seckill/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10000  # 每秒补充10000个令牌
                redis-rate-limiter.burstCapacity: 20000  # 桶容量20000
                key-resolver: "#{@userKeyResolver}"      # 按用户限流
```

---

### 3.3 Redis + Lua限流（当前已使用）

**优点**：
- 分布式限流
- 原子操作
- 性能高

**当前实现**：
```java
public class DistributedRateLimiter {
    private static final String LUA_SCRIPT = 
        "local key = KEYS[1] " +
        "local limit = tonumber(ARGV[1]) " +
        "local current = tonumber(redis.call('INCR', key)) " +
        "if current == 1 then " +
        "    redis.call('EXPIRE', key, ARGV[2]) " +
        "end " +
        "if current > limit then " +
        "    return 0 " +
        "else " +
        "    return 1 " +
        "end";
}
```

### 3.4 Resilience4j

**特点**：
- 轻量级，Spring Boot友好
- 功能类似Hystrix（已停更）
- 支持限流、熔断、重试

**示例**：
```java
// 配置
RateLimiterConfig config = RateLimiterConfig.custom()
    .limitForPeriod(10000)       // 每个周期10000个请求
    .limitRefreshPeriod(Duration.ofSeconds(1))  // 周期1秒
    .timeoutDuration(Duration.ofMillis(100))    // 超时100ms
    .build();

RateLimiter rateLimiter = RateLimiter.of("seckill", config);

// 使用
Supplier<SeckillGrabResult> supplier = RateLimiter
    .decorateSupplier(rateLimiter, () -> grabCoupon(request));

return supplier.get();
```

## 4. 方案选择建议

### 4.1 场景对比

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **简单限流** | Guava | 轻量、高性能、零依赖 |

| **需要熔断降级** | Sentinel / Resilience4j | 功能完善 |
| **全局精确限流** | Sentinel（Token Server） | 集群协调 |
| **网关层限流** | Spring Cloud Gateway | 统一入口 |
| **Nginx层限流** | Nginx | 最前端保护 |
| **分布式限流** | Redis + Lua | 简单高效 |

### 4.3 是否需要迁移到Sentinel？

#### 不需要迁移的理由 ❌

1. **当前方案已经很好**
   - ✅ 动态限流已实现
   - ✅ 多层防护完善
   - ✅ Redis压力已降低70%
   - ✅ 性能已满足5-10万QPS

2. **迁移成本高**
   - ❌ 需要部署Dashboard
   - ❌ 需要改造大量代码
   - ❌ 团队学习成本
   - ❌ 运维复杂度增加

3. **收益不明显**
   - Sentinel的核心优势（可视化、集群限流）对当前场景不是刚需
   - 我们已经通过动态限流解决了主要问题

#### 可以考虑迁移的场景 ✅

1. **需要可视化监控**
   - 运营需要实时看限流数据
   - 需要快速调整限流规则

2. **需要熔断降级**
   - 依赖的服务（如积分服务）不稳定
   - 需要自动熔断保护

3. **多服务协调**
   - 有多个微服务需要统一限流管理
   - 需要全局限流

### 6.2 可选的增强（无需迁移Sentinel）

#### 增强1：添加Prometheus监控

```java
// 添加依赖
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>

// 暴露限流指标
@Component
public class RateLimiterMetrics {
    
    @Autowired
    private MeterRegistry meterRegistry;
    
    @Scheduled(fixedRate = 1000)
    public void recordMetrics() {
```

```java
@Component
public class RateLimiterAspect {
    
    @Around("@annotation(RateLimit)")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        boolean acquired = rateLimiter.tryAcquire();
        
        if (!acquired) {
            log.warn("限流拒绝: method={}, time={}", 
                pjp.getSignature().getName(), 
                System.currentTimeMillis() - start);
            throw SeckillException.systemBusy();
        }
        
        return pjp.proceed();
```
