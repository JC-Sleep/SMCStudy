## 项目概述

设计并实现一个 **SpringBoot 企业级支付网关系统**，对接渣打银行（Standard Chartered Bank）及多个支付渠道（支付宝、建设银行等），重点解决：

### 核心问题
1. **回调处理与幂等性** - 防止重复扣款
2. **交易对账机制** - 确保账目一致
3. **TIMEOUT问题** - "客户已付款但订单失败"的死锁场景
4. **分布式幂等性** - 多实例部署的并发安全
5. **高并发性能** - TPS>1000的系统稳定性

## 问题分析

### 当前问题

根据案例日志分析：
- **日期**: 2026-02-03 20:01:50
- **订单编号**: CSP001xxxxx
- **用户ID**: 33xxxx
- **金额**: 176
- **支付状态**: TIMEOUT
- **会话ID**: cs_acq

**死锁场景描述：**
1. 客户在自己端看到"支付成功"
2. 系统显示"支付失败"（因为TIMEOUT）
3. 银行确认钱已经转给公司
4. 财务部门说没有收到款项
5. 客户无法从银行获得退款，公司也没有成功支付记录

### 根本原因

1. **回调未到达**: 渣打银行和支付系统之间的网络问题导致回调通知未送达
2. **回调处理失败**: 回调到达但系统处理失败（异常、数据库锁等）
3. **超时阈值太短**: 系统在银行发送确认之前就标记交易为TIMEOUT
4. **缺少对账机制**: 系统缺乏自动验证银行支付状态的方法
5. **缺少幂等性**: 重复回调处理可能导致状态不一致
6. **缺乏交易审计追踪**: 日志不足，无法追溯发生了什么

```

客户端

  ↓

           │

✅ 数据库唯一索引
✅ 乐观锁（VERSION）
✅ 终态检查
```

#### 增强版架构（分布式/高并发）

```
┌──────────────────────────────────────────────────────────┐
│                      增强版架构                           │
└──────────────────────────────────────────────────────────┐

客户端
  │
  ↓
负载均衡器（Nginx）
  │
  ├──→ 实例A ─┐
  ├──→ 实例B ─┼──→ PaymentServiceEnhanced
  └──→ 实例C ─┘
         │
         ├──→ Redis集群 ← ⭐ 分布式锁 + 回调去重

         ├──→ PaymentGatewayRouter

幂等性保障（三层防护）：
✅ Redis分布式锁（第一层）
✅ 数据库唯一索引（第二层）
✅ 乐观锁+终态检查（第三层）
```

### 支付流程

#### 标准版流程

```
客户 → 发起支付请求 → 网关 → 渣打银行
                ↓                    ↓
          存储交易记录            处理支付
                ↓                    ↓
          返回待处理状态          发送回调
                ↓                    ↓
    轮询状态（如需要） ← 回调处理器
                ↓
          更新交易状态
                ↓
    通知客户/订单系统
```

```
客户（前端生成UUID）
  │
  ↓

  │
  ↓
负载均衡器 → 实例A/B/C
  │
  ↓
⭐ Redis分布式锁检查
  │
  ├─ 已锁定 → 返回已存在交易 ✅
  │
  └─ 未锁定 → 获取锁
       │
       ↓
  存储交易记录（数据库）
       │
       ↓
  PaymentGatewayRouter（选择渠道）
       │
       ├─ 渣打银行
       ├─ 支付宝
       └─ 建设银行
       │
       ↓
  返回支付URL
       │
       ↓
  释放Redis锁
       
       
银行回调
  │
  ↓
⭐ Redis去重检查
  │
  ├─ 已处理 → 直接返回 ✅
  │
  └─ 首次 → 标记已处理
       │
       ↓
  验证签名
       │
       ↓

       │

### 步骤4：实现支付服务层

**目录**: `PaymentSys/src/main/java/sys/smc/payment/service`

#### 4.1 支付发起服务
```java
@Service
@Slf4j
public class PaymentService {
    
    @Autowired
    private PaymentTransactionMapper transactionMapper;
    
    @Autowired
    private StandardCharteredGatewayClient gatewayClient;
    
    @Transactional(rollbackFor = Exception.class)
    public PaymentInitResponse initiatePayment(PaymentInitRequest request) {
        // 1. 生成幂等键
        String idempotencyKey = generateIdempotencyKey(request);
        
        // 2. 检查交易是否已存在（幂等性）
        PaymentTransaction existing = transactionMapper.selectByIdempotencyKey(idempotencyKey);
        if (existing != null) {
            log.warn("检测到重复支付请求: {}", idempotencyKey);
            return buildResponseFromTransaction(existing);
        }
        
        // 3. 创建INIT状态的交易记录
        PaymentTransaction transaction = buildTransaction(request, idempotencyKey);
        transaction.setPaymentStatus(PaymentStatus.INIT.name());
        transactionMapper.insert(transaction);
        
        try {
            // 4. 调用渣打银行API
            GatewayPaymentResponse gatewayResponse = gatewayClient.createPayment(request);
            
            // 5. 更新交易信息
            transaction.setGatewayTransactionId(gatewayResponse.getTransactionId());
            transaction.setPaymentStatus(PaymentStatus.PENDING.name());
            transaction.setStatusUpdateTime(new Date());
            transactionMapper.updateById(transaction);
            
            // 6. 返回响应给客户端
            return PaymentInitResponse.builder()
                .transactionId(transaction.getTransactionId())
                .paymentUrl(gatewayResponse.getPaymentUrl())
                .status(PaymentStatus.PENDING.name())
                .build();
                
        } catch (Exception e) {
            log.error("支付发起失败: {}", request, e);
            transaction.setPaymentStatus(PaymentStatus.FAILED.name());
            transaction.setErrorMessage(e.getMessage());
            transactionMapper.updateById(transaction);
            throw new PaymentException("支付发起失败", e);
        }
    }
    
    private String generateIdempotencyKey(PaymentInitRequest request) {
        return request.getOrderReference() + "_" + 
               request.getAmount() + "_" + 
               System.currentTimeMillis();
    }
}

#### 5.2 回调服务
```java
@Service
@Slf4j
public class PaymentCallbackService {
    
    @Autowired
    private PaymentTransactionMapper transactionMapper;
    
    @Autowired
    private PaymentCallbackLogMapper callbackLogMapper;
    
    @Autowired
    private SignatureVerifier signatureVerifier;
    
    @Async("callbackExecutor")
    @Transactional(rollbackFor = Exception.class, 
                   isolation = Isolation.READ_COMMITTED,
                   propagation = Propagation.REQUIRES_NEW)
    public void processCallback(String rawBody, Map<String, String> headers, String clientIp) {
        long startTime = System.currentTimeMillis();
        PaymentCallbackLog callbackLog = new PaymentCallbackLog();
        
        try {
            // 1. 解析回调数据
            PaymentCallbackData callbackData = parseCallbackData(rawBody);
            
            // 2. 验证签名
            boolean signatureValid = signatureVerifier.verify(rawBody, headers.get("X-Signature"));
            callbackLog.setSignatureValid(signatureValid ? 1 : 0);
            
            if (!signatureValid) {
                log.error("回调签名无效: {}", rawBody);
                callbackLog.setProcessingStatus("FAILED");
                callbackLog.setErrorMessage("签名无效");
                return;
            }
            
            // 3. 查找交易
            PaymentTransaction transaction = transactionMapper.selectByGatewayTransactionId(
                callbackData.getGatewayTransactionId()
            );
            
            if (transaction == null) {
                log.error("交易未找到: {}", callbackData.getGatewayTransactionId());
                callbackLog.setProcessingStatus("FAILED");
                callbackLog.setErrorMessage("交易未找到");
                return;
            }
            
            callbackLog.setTransactionId(transaction.getTransactionId());
            
            // 4. 检查是否已处理（幂等性）
            if (isTerminalStatus(transaction.getPaymentStatus())) {
                log.warn("交易已处于终态: {}", transaction.getPaymentStatus());
                callbackLog.setProcessingStatus("DUPLICATE");
                return;
            }
            
            // 5. 使用乐观锁更新交易状态
            boolean updated = updateTransactionStatus(transaction, callbackData);
            
            if (!updated) {
                log.warn("由于版本冲突更新交易失败，将重试");
                callbackLog.setProcessingStatus("RETRY");
                throw new OptimisticLockException("版本冲突");
            }
            
            // 6. 触发下游动作（订单确认、通知等）
            if (PaymentStatus.SUCCESS.name().equals(callbackData.getPaymentStatus())) {
                notifyOrderSystem(transaction);
            }
            
            callbackLog.setProcessingStatus("SUCCESS");
            
        } catch (Exception e) {
            log.error("处理回调时出错", e);
            callbackLog.setProcessingStatus("FAILED");
            callbackLog.setErrorMessage(e.getMessage());
            throw e;
            
        } finally {
            // 总是记录回调尝试
            long processingTime = System.currentTimeMillis() - startTime;
            callbackLog.setProcessingTimeMs((int) processingTime);
            callbackLog.setRequestBody(rawBody);
            callbackLog.setRequestHeaders(JSON.toJSONString(headers));
            callbackLog.setRequestIp(clientIp);
            callbackLog.setCallbackTime(new Date());
            
            callbackLogMapper.insert(callbackLog);
        }
    }
    
    private boolean updateTransactionStatus(PaymentTransaction transaction, PaymentCallbackData callbackData) {
        // 使用乐观锁
        PaymentTransaction update = new PaymentTransaction();
        update.setId(transaction.getId());
        update.setVersion(transaction.getVersion()); // 乐观锁的关键
        update.setPreviousStatus(transaction.getPaymentStatus());
        update.setPaymentStatus(callbackData.getPaymentStatus());
        update.setStatusUpdateTime(new Date());
        update.setCallbackReceived(1);
        update.setCallbackCount(transaction.getCallbackCount() + 1);
        update.setLastCallbackTime(new Date());
        update.setGatewayTransactionId(callbackData.getGatewayTransactionId());
        
        int rows = transactionMapper.updateById(update);
        return rows > 0;
    }
    
    private boolean isTerminalStatus(String status) {
        return PaymentStatus.SUCCESS.name().equals(status) ||
               PaymentStatus.FAILED.name().equals(status) ||
               PaymentStatus.REFUNDED.name().equals(status);
    }
}

@Component
@Slf4j
public class PaymentReconciliationJob {
    
    @Autowired
    private PaymentTransactionMapper transactionMapper;
    
    @Autowired
    private StandardCharteredGatewayClient gatewayClient;
    
    @Autowired
    private PaymentReconciliationMapper reconciliationMapper;
    
    /**
     * 每30分钟自动对账TIMEOUT和PENDING交易
     * 这可以防止"客户已付款但系统显示超时"的问题
     */
    @Scheduled(cron = "${payment.reconciliation.cron:0 */30 * * * ?}")
    @Transactional(rollbackFor = Exception.class)
    public void reconcileTimeoutTransactions() {
        log.info("开始自动对账TIMEOUT/PENDING交易");
        
        // 查找超过30分钟的TIMEOUT或PENDING交易
        Date cutoffTime = DateUtils.addMinutes(new Date(), -30);
        
        List<PaymentTransaction> timeoutTransactions = transactionMapper.selectList(
            new LambdaQueryWrapper<PaymentTransaction>()
                .in(PaymentTransaction::getPaymentStatus, 
                    PaymentStatus.TIMEOUT.name(), 
                    PaymentStatus.PENDING.name())
                .lt(PaymentTransaction::getStatusUpdateTime, cutoffTime)
                .orderByAsc(PaymentTransaction::getCreateTime)
        );
        
        log.info("找到 {} 笔需要对账的交易", timeoutTransactions.size());
        
        int matched = 0;
        int mismatch = 0;
        List<String> mismatchIds = new ArrayList<>();
        
        for (PaymentTransaction transaction : timeoutTransactions) {
            try {
                // 查询渣打银行的实际状态
                GatewayTransactionStatus gatewayStatus = gatewayClient.queryTransactionStatus(
                    transaction.getGatewayTransactionId()
                );
                
                transaction.setLastQueryTime(new Date());
                
                // 比较状态
                if (!transaction.getPaymentStatus().equals(gatewayStatus.getStatus())) {
                    log.warn("发现状态不匹配 - 交易: {}, 本地: {}, 银行: {}", 
                        transaction.getTransactionId(),
                        transaction.getPaymentStatus(),
                        gatewayStatus.getStatus());
                    
                    // 更新为银行状态（银行是真实来源）
                    transaction.setPreviousStatus(transaction.getPaymentStatus());
                    transaction.setPaymentStatus(gatewayStatus.getStatus());
                    transaction.setReconciliationStatus("MISMATCH");
                    transaction.setReconciliationTime(new Date());
                    
                    // 如果银行说成功但我们标记为TIMEOUT，立即通知
                    if (PaymentStatus.SUCCESS.name().equals(gatewayStatus.getStatus())) {
                        sendAlertEmail(transaction, "严重: 支付成功但标记为TIMEOUT");
                        notifyOrderSystem(transaction);
                    }
                    
                    mismatch++;
                    mismatchIds.add(transaction.getTransactionId());
                } else {
                    transaction.setReconciliationStatus("MATCHED");
                    matched++;
                }
                
                transactionMapper.updateById(transaction);
                
            } catch (Exception e) {
                log.error("对账交易失败: {}", transaction.getTransactionId(), e);
            }
        }
        
        // 保存对账记录
        PaymentReconciliation reconciliation = new PaymentReconciliation();
        reconciliation.setReconciliationDate(new Date());
        reconciliation.setReconciliationType("AUTO_TIMEOUT");
        reconciliation.setTotalTransactions(timeoutTransactions.size());
        reconciliation.setMatchedCount(matched);
        reconciliation.setMismatchCount(mismatch);
        reconciliation.setMismatchTransactionIds(JSON.toJSONString(mismatchIds));
        reconciliation.setReconciliationStatus("COMPLETED");
        reconciliation.setStartTime(new Date());
        reconciliation.setEndTime(new Date());
        
        reconciliationMapper.insert(reconciliation);
        
        log.info("对账完成 - 匹配: {}, 不匹配: {}", matched, mismatch);
    }

@Component

public class StandardCharteredGatewayClient {
    
    @Value("${scb.api.endpoint}")
    private String apiEndpoint;
    
    @Value("${scb.api.key}")
    private String apiKey;
    
    @Value("${scb.api.secret}")
    private String apiSecret;
    
    @Autowired
    private RestTemplate restTemplate;
    
    public GatewayPaymentResponse createPayment(PaymentInitRequest request) {
        String url = apiEndpoint + "/create";
        
        // 构建请求
        Map<String, Object> gatewayRequest = new HashMap<>();
        gatewayRequest.put("merchantId", apiKey);
        gatewayRequest.put("orderReference", request.getOrderReference());
        gatewayRequest.put("amount", request.getAmount());
        gatewayRequest.put("currency", "HKD");
        gatewayRequest.put("callbackUrl", getCallbackUrl());
        gatewayRequest.put("returnUrl", request.getReturnUrl());
        
        // 签名请求
        String signature = generateSignature(gatewayRequest);
        
        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_JSON);
        headers.set("X-API-Key", apiKey);
        headers.set("X-Signature", signature);
        
        HttpEntity<Map<String, Object>> entity = new HttpEntity<>(gatewayRequest, headers);
        
        try {
            ResponseEntity<GatewayPaymentResponse> response = restTemplate.exchange(
                url,
                HttpMethod.POST,
                entity,
                GatewayPaymentResponse.class
            );
            
            return response.getBody();
            
        } catch (Exception e) {
            log.error("网关创建支付失败", e);
            throw new GatewayException("创建支付失败", e);
        }
    }
    
    public GatewayTransactionStatus queryTransactionStatus(String gatewayTransactionId) {
        String url = apiEndpoint + "/query/" + gatewayTransactionId;
        
        HttpHeaders headers = new HttpHeaders();
        headers.set("X-API-Key", apiKey);
        
        HttpEntity<Void> entity = new HttpEntity<>(headers);
        
        try {
            ResponseEntity<GatewayTransactionStatus> response = restTemplate.exchange(
                url,
                HttpMethod.GET,
                entity,
                GatewayTransactionStatus.class
            );
            
            return response.getBody();
            
        } catch (Exception e) {
            log.error("查询交易状态失败", e);
            throw new GatewayException("查询状态失败", e);
        }
    }
    
    private String generateSignature(Map<String, Object> data) {
        // 实现HMAC-SHA256签名
        // 按字母顺序排序键，连接值，用密钥签名

## 深入考量

### 1. 为什么会出现TIMEOUT问题

死锁场景发生的原因：

1. **网络延迟**: 渣打银行的回调被延迟或丢失
2. **系统超时太短**: 系统在银行处理完成前就标记为TIMEOUT
3. **回调处理失败**: 回调到达但更新数据库失败（异常、锁等）
4. **缺少对账**: 系统没有机制检查银行的实际状态

**已实施的解决方案**:
- 将超时阈值增加到5分钟（可配置）
- 异步回调处理防止阻塞
- 对账任务每30分钟运行一次
- 所有回调都记录日志，无论处理结果
- 乐观锁防止竞态条件

### 2. 幂等性策略

**实现方式**:
- `idempotencyKey` = `orderReference` + `amount` + `timestamp`
- 数据库中`IDEMPOTENCY_KEY`有唯一约束
- 创建新交易前检查是否已存在
- 通过检查交易状态检测重复回调

**为什么有效**:
- 防止用户刷新支付页面时重复扣款
- 处理网络重试场景
- 对并发请求安全

### 3. 交易状态机

**状态**:
```
INIT → PENDING → SUCCESS
                → FAILED
                → TIMEOUT → RECONCILING → SUCCESS/FAILED
                                        → REFUNDED
```

**终态**: SUCCESS, FAILED, REFUNDED（不再转换）
**可对账状态**: TIMEOUT, PENDING（可被对账更新）

### 4. 乐观锁 vs 悲观锁

**选择**: **乐观锁**配合`@Version`

**原因**:
- 回调很少（每笔交易通常1-2次）
- 低竞争场景
- 性能优于悲观锁
- MyBatis-Plus原生支持

**实现**:
```java
// 交易实体有版本字段
@Version
private Integer version;

// 如果版本变化，更新将失败
int rows = transactionMapper.updateById(transaction);
if (rows == 0) {
    throw new OptimisticLockException("检测到并发更新");
}
```

### 5. 回调重试机制

**来自银行侧**:
- 渣打银行通常会重试失败的回调
- 重试间隔：1分钟、5分钟、15分钟、30分钟、1小时

**来自我们侧**:
- 立即返回200 OK防止重试风暴
- 异步处理回调
- 记录所有尝试用于调试
- 如果处理失败，对账任务会修复它

### 6. 安全考虑

1. **签名验证**: 验证所有回调的HMAC-SHA256签名
2. **IP白名单**: 只接受来自渣打银行IP的回调
3. **仅HTTPS**: 所有通信通过TLS
4. **API密钥加密**: 以加密格式存储API密钥
5. **限流**: 防止回调洪水攻击

### 7. 监控和告警

**需跟踪的指标**:
- 支付成功率
- 平均支付处理时间
- 回调处理成功率
- TIMEOUT交易数量
- 对账不匹配数量

**告警**:
- TIMEOUT → SUCCESS转换（严重问题）
- 回调处理失败率 > 5%
- 对账不匹配 > 10笔/天
- 支付网关API响应时间 > 10秒

### 8. 性能优化

1. **数据库索引**: 查询列的适当索引
2. **连接池**: HikariCP优化配置
3. **异步处理**: 回调异步处理
4. **批量对账**: 单次查询处理多笔交易
5. **缓存**: 缓存网关配置

### 9. 灾难恢复

**备份策略**:

- 交易日志归档1年
- 回调日志归档6个月

## 总结

此支付系统设计通过以下方式解决了关键的"客户已付款但系统显示超时"问题：

6. **手动干预工具**: 处理边缘情况的手册

**关键洞察**是：**网关是真实来源**。当有疑问时，查询网关并相应更新本地状态。对账任务作为安全网，捕获任何遗漏的回调或处理失败。

## 分布式与高并发场景升级方案

> 详细技术分析请参阅：
> - `标准版vs增强版技术深度分析.md`
> - `支付系统幂等性设计与实现.md`
> - `幂等性增强功能使用指南.md`

### 为什么需要升级

#### 标准版存在的问题

**单机环境（TPS<500）**：✅ **当前标准版足够**
- 并发窗口短（25ms）
- 数据库唯一索引有效
- 冲突概率极低（0.1%）

**分布式环境（多实例部署）**：⚠️ **建议升级**
- 并发窗口放大5倍（125ms）
- 幂等键时间戳问题（每个实例不同）
- 冲突概率上升到5%
- 可能重复扣款

**高并发场景（TPS>1000）**：🔴 **必须升级**
- 数据库锁竞争严重
- CPU飙升到95%以上
- 响应时间从150ms → 5000ms
- 系统可能雪崩

### 核心问题分析

#### 问题1：幂等键生成缺陷

**当前实现**：
```java
// 标准版 - 使用时间戳
private String generateIdempotencyKey(PaymentInitRequest request) {
    return request.getOrderReference() + "_" +
           request.getAmount().toString() + "_" +
           System.currentTimeMillis();  // ⚠️ 每个实例时间戳不同
}
```

**问题场景**：
```
用户点击"支付" → 实例A：CSP001_176_1712000001
用户再点击     → 实例B：CSP001_176_1712000002  ← 不同的幂等键！
                ↓
数据库认为是两个不同的支付 ❌
                ↓
创建2笔交易，重复扣款
```

#### 问题2：并发窗口放大

```
单机环境：
    查询数据库(10ms) → 插入数据库(10ms)
    并发窗口 = 20ms

分布式环境：
    网络延迟(50ms) → 查询(10ms) → 网络延迟(50ms) → 插入(10ms)
    并发窗口 = 120ms （6倍）
    
    ↓
    
冲突概率大幅提升
```

#### 问题3：数据库锁竞争

```
高并发场景（TPS>1000）：

100个并发请求 → 数据库唯一索引
    ↓
请求1：获得锁，插入成功 ✅
请求2-100：串行等待锁，逐个检测冲突 ⏳
    ↓
99个请求 × 20ms = 1980ms延迟
    ↓
数据库CPU 100%，系统雪崩 ❌
```

### 升级方案：增强版架构

#### 核心改进

```
┌─────────────────────────────────────────────────────────┐
│           增强版三层防护体系                             │
└─────────────────────────────────────────────────────────┘

第一层：Redis分布式锁
    ↓ 拦截重复请求，并发窗口=0ms
    
第二层：前端幂等Token
    ↓ 幂等键固定，不依赖时间戳
    
第三层：数据库唯一索引
    ↓ 最后防线
```

#### 新增组件

@Service
public class PaymentServiceEnhanced {
    
    @Autowired
    private RedissonClient redissonClient;
    
    @Transactional
    public PaymentInitResponse initiatePayment(PaymentInitRequest request) {
        String idempotencyKey = generateIdempotencyKey(request);
        
        // ⭐ 使用Redis分布式锁
        RLock lock = redissonClient.getLock("payment:lock:" + idempotencyKey);
        
        try {
            // 最多等待5秒，锁定30秒
            boolean locked = lock.tryLock(5, 30, TimeUnit.SECONDS);
            if (!locked) {
                throw new PaymentException("系统繁忙，请稍后重试");
            }
            
            // 双重检查
            PaymentTransaction existing = transactionMapper.selectByIdempotencyKey(idempotencyKey);
            if (existing != null) {
                return buildResponseFromTransaction(existing);
            }
            
            // 执行支付创建...
            
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
    
    // ⭐ 改进的幂等键生成
    private String generateIdempotencyKey(PaymentInitRequest request) {
        // 优先使用前端传递的Token
        if (request.getIdempotencyToken() != null) {
            return request.getIdempotencyToken();
        }
        // 兜底：订单号+支付方式
        return request.getOrderReference() + "_" + request.getPaymentMethod();
    }
}

**4. 前端幂等Token（⭐ 重要：由前端生成）**

> **关键说明**：幂等Token **必须由前端生成**，不能由后端生成！
> 详细解释请参阅：`幂等Token生成策略-前端vs后端.md`

**为什么前端生成**：
```
前端生成：
  页面打开 → 生成Token-A → 存储
  用户点击 → 发送请求（Token-A）
  网络超时 → 用户再点击 → 发送请求（还是Token-A）✅
  → 后端识别重复 ✅

后端生成：
  用户点击 → 发送请求 → 后端生成Token-A → 创建交易
  响应丢失 → 用户再点击 → 后端生成Token-B ❌
  → 后端无法识别重复 ❌
```

**前端实现示例**：

```javascript
// 方式1：使用UUID库（推荐）
import { v4 as uuidv4 } from 'uuid';

class PaymentService {
    constructor() {
        this.tokenCache = new Map();
    }
    
    // 获取或创建Token（订单级别）
    getIdempotencyToken(orderId) {
        // 先查缓存
        if (this.tokenCache.has(orderId)) {
            return this.tokenCache.get(orderId);
        }
        
        // 再查sessionStorage（防刷新）
        const storageKey = `payment_token_${orderId}`;
        let token = sessionStorage.getItem(storageKey);
        
        if (!token) {
            token = uuidv4();  // ⭐ 生成新Token
            sessionStorage.setItem(storageKey, token);
            console.log(`订单${orderId}生成Token:`, token);
        }
        
        this.tokenCache.set(orderId, token);
        return token;
    }
    
    async initiatePayment(orderId, amount) {
        const token = this.getIdempotencyToken(orderId);
        
        const response = await fetch('/api/payment/initiate', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({
                orderReference: orderId,
                amount: amount,
                paymentMethod: 'ALIPAY',
                idempotencyToken: token,  // ⭐ 前端生成的Token
                returnUrl: window.location.origin + '/payment/return'
            })
        });
        
        return await response.json();
    }
}

// 使用
const paymentService = new PaymentService();
paymentService.initiatePayment('CSP001234567', 176.00);
```

### Bug-2：`@Transactional` 范围包住了 Redis 锁释放 → 并发窗口重复创建

**代码位置**：`PaymentServiceEnhanced.java` → `initiatePayment()` 方法的 `@Transactional` 注解  
**危害等级**：🟡 中危 — 极端并发下重复创建交易（概率低但真实存在）  
**触发条件**：分布式部署（≥2实例）+ 同一订单毫秒级并发重试 + Redis可用  

#### 旧代码（已修复）

```java
// 文件：PaymentServiceEnhanced.java
// 问题：@Transactional 注解加在了 initiatePayment() 上

// ❌ 旧代码：
@Transactional(rollbackFor = Exception.class)  // ← 事务在这里开始
public PaymentInitResponse initiatePayment(PaymentInitRequest request) {
    PaymentGateway gateway = gatewayRouter.selectGateway(request.getPaymentMethod());
    return doInitiatePaymentWithDistributedLock(request, gateway);
}   // ← 事务在这里提交（方法返回时）

// 而锁释放在内部的 finally 块：
private PaymentInitResponse doPaymentWithLock(...) {
    RLock lock = ...;
    try {
        lock.tryLock(...);
        // ... INSERT 交易（事务未提交）...

    } finally {
        lock.unlock();  // ← 锁在这里释放！
    }
    // 此时外层 initiatePayment() 的 @Transactional 还未 commit！
    // 从 finally 结束到 initiatePayment() return，存在一个危险窗口
}

#### 修复方案（已落地）

```java
// ✅ 新代码：
// 1. 去掉 initiatePayment() 上的 @Transactional（外层不开事务）
public PaymentInitResponse initiatePayment(PaymentInitRequest request) {
    PaymentGateway gateway = gatewayRouter.selectGateway(request.getPaymentMethod());
    return doInitiatePaymentWithDistributedLock(request, gateway);
}

// 2. 在锁内用 TransactionTemplate 控制事务，确保事务在锁释放前提交
private PaymentInitResponse doPaymentWithLock(...) {
    RLock lock = redissonClient.getLock(lockKey);
    lock.tryLock(5, 30, SECONDS);
    try {
        return transactionTemplate.execute(txStatus -> {
            // 双重检查 + INSERT 在同一事务内
            PaymentTransaction existing = transactionMapper.selectByIdempotencyKey(key);
            if (existing != null) return buildResponseFromTransaction(existing);
            return doInitiatePayment(request, gateway, key);
        }); // ← execute() 结束，事务此刻提交，锁还持有！
    } finally {
        lock.unlock(); // ← 事务已提交后才释放锁，窗口彻底消除
    }
}
```

#### ⚠️ 修复后仍存在的问题

| 残留问题 | 说明 | 进一步解决方案 |
|---------|------|--------------|
| **网关调用在事务内** | `doInitiatePayment()` 在 `transactionTemplate.execute()` 内，包含了 `gateway.createPayment()` 调用。如果网关成功但之后 `updateById()` 抛异常，事务整体回滚 → 银行已有待支付订单但 DB 里没有记录 | 把 gateway 调用移到 TransactionTemplate 外，只在事务内做 DB 操作；网关失败才回滚 |
| **TransactionTemplate 异常语义** | `execute()` 内抛 `RuntimeException` 会触发回滚，但 `CheckedException` 不会，需要显式 `txStatus.setRollbackOnly()` | 所有业务异常统一用 `RuntimeException`（已是当前做法） |
| **Redis 宕机时无降级事务** | Redis 不可用时走 `doPaymentWithoutLock()`，此路径没有任何事务包裹 | 给 `doPaymentWithoutLock()` 也加 `@Transactional`，并加 DB 唯一索引兜底 |
| **DB 唯一约束缺失** | 目前 `IDEMPOTENCY_KEY` 没有 `UNIQUE INDEX`，Bug-2 修复后如果出现极端情况仍可写入重复数据 | **立刻执行**：`CREATE UNIQUE INDEX UNQ_PT_IDEMPOTENCY_KEY ON PAYMENT_TRANSACTION(IDEMPOTENCY_KEY)` |

// 支付回调更新状态时，在同一事务内写消息（原子）
@Transactional
private boolean updateTransactionStatus(PaymentTransaction tx, PaymentCallbackData data) {
    // 1. 乐观锁更新支付状态（已有）
    int rows = transactionMapper.updateById(buildUpdate(tx, data));
    if (rows == 0) return false;

    // 2. 同一事务写本地消息（新增）
    if ("SUCCESS".equals(data.getPaymentStatus())) {
        messageOutboxMapper.insert(MessageOutbox.builder()
            .msgType("PAYMENT_SUCCESS")
            .payload(JSON.toJSONString(Map.of(
                "orderId",  tx.getOrderReference(),
                "txnId",    tx.getTransactionId(),
                "amount",   tx.getAmount(),
                "currency", tx.getCurrency()
            )))
            .status("PENDING")
            .createTime(new Date())
            .build());
    }
    return true;
}
// 定时任务扫描 PENDING 消息 → 投递MQ → 订单服务幂等消费
```

| **同步HTTP超时阻塞** | 方案A | 订单系统响应慢时，回调线程被阻塞，大量请求堆积，线程池耗尽 | 设置短超时（如500ms）+ 异步化；或改用方案B |
| **同步HTTP失败无重试** | 方案A | 订单服务宕机时，HTTP失败，支付SUCCESS但订单未更新，问题依旧 | 加补偿表 + 定时重试；或改用方案B |
| **消息重复投递** | 方案B | 定时任务多实例同时扫描PENDING消息，同一条消息被发送两次 | 定时任务加分布式锁；或状态机：PENDING→SENDING→SENT（原子CAS） |
| **消息消费失败** | 方案B | 订单服务消费失败，消息进入死信队列（DLQ），需人工处理 | 配置 DLQ 告警 + 运营后台支持人工重发 |

## 总结与最佳实践

### 核心设计原则

1. **幂等性第一**：防止重复扣款是支付系统的生命线
2. **网关是真相**：有疑问时查询银行状态
3. **异步处理**：回调立即返回，避免重试风暴
4. **乐观锁+终态保护**：防止并发修改
5. **完整审计**：所有状态变更必须记录
6. **自动对账**：定期验证与银行状态一致
7. **可扩展架构**：支持多渠道支付

### 系统成熟度评估

| 级别 | 特征 | 适用场景 |
|-----|------|---------|
| **Level 1** | 基本幂等性（数据库） | 单机、小型应用 |
| **Level 2** | + 对账机制 | 中小型应用 |
| **Level 3** | + Redis分布式锁 | 分布式应用 |
| **Level 4** | + 多渠道支付 | 大型电商 |
| **Level 5** | + 全链路监控 | 金融级应用 |

**本系统当前级别**：
- 标准版：Level 2
- 增强版：Level 4（v2.1修复Bug-1/2后；Bug-3 notifyOrderSystem 实现后可达 Level 5）
