## 幂等性是什么？用大白话讲清楚

### 一句话定义

> **同一个操作，不管执行多少次，最终结果都和执行一次一样。**

### 生活中的例子

| 操作 | 幂等？ | 原因 |
|------|--------|------|
| 按电梯"开门"按钮 | ✅ | 门开着，再按还是开着，没变化 |
| 银行转账 100 元 | ❌ 天然不幂等 | 按两次就扣 200 元了！ |
| 设置灯亮度为 50% | ✅ | 不管设多少次，结果都是 50% |
| "给我下一杯咖啡" | ❌ | 说两次就拿两杯 |

---

### 支付系统里的幂等性场景

```
用户点击「支付」按钮
     ↓
前端发送请求 → 网络超时 → 前端看不到响应
     ↓
用户再次点击「支付」（或自动重试）
     ↓
服务器收到两次请求
```

**没有幂等性的结果：**
```
请求1 → 扣款 100 元 ✅
请求2 → 又扣款 100 元 ❌  ← 用户被扣了 200 元！！
```

**有幂等性的结果：**
```
请求1 → 扣款 100 元 ✅  → 返回 txnId=TXN001
请求2 → 检测到重复   ✅  → 直接返回 TXN001（不再扣款）
```

---

### 本项目怎么实现幂等性的？

**核心思路：给每个请求贴一个"唯一标签"（幂等键），处理前先查标签是否存在**

```
前端生成 UUID（幂等Token）：abc-123-xyz
           ↓
第一次请求携带这个 UUID
           ↓
服务端：有没有处理过 abc-123-xyz？
  → 没有 → 正常处理 → 记录"已处理abc-123-xyz"
  → 有了 → 直接返回上次结果，不再处理
```

**本项目三层实现：**

| 层 | 技术手段 | 作用 |
|----|---------|------|
| ① Redis 分布式锁 | `Redisson.tryLock(key)` | 支付发起时，同一幂等键只有一个线程能进 |
| ② DB 幂等键 | `IDEMPOTENCY_KEY` 字段查重 | 双重检查，锁释放后再查一次DB |
| ③ 回调 Redis 去重 | `setIfAbsent(callbackKey)` | 同一回调消息只处理一次 |

---

### 幂等性 vs 去重 vs 乐观锁 的关系（很多人搞混）

```
幂等性（目标）
    ├── 去重（快速判断：这个请求来过吗？）
    │       → Redis setIfAbsent / DB 查询
    └── 乐观锁（并发安全：两个线程同时通过去重怎么办？）
                → @Version + WHERE version=? 只有一个能写成功
```

> 去重是"门卫"，乐观锁是"最后一道闸"，两者配合才能真正做到幂等。

## 乐观锁详解

### 代码位置

**第一步：实体类声明 `@Version`（BaseEntity.java）**

```java
@Version
@TableField(value = "VERSION")
private Integer version;
// → 对应数据库 VERSION NUMBER(10) DEFAULT 0 字段
```

**第二步：注册乐观锁拦截器（MyBatisPlusConfig.java）**

```java
interceptor.addInnerInterceptor(new OptimisticLockerInnerInterceptor());
// → 拦截所有 updateById()，自动追加 AND VERSION = ? 条件
```

**第三步：回调处理时使用乐观锁（PaymentCallbackServiceEnhanced.java）**

```java
private boolean updateTransactionStatus(PaymentTransaction transaction,
                                         PaymentCallbackData callbackData) {
    PaymentTransaction update = new PaymentTransaction();
    update.setId(transaction.getId());
    update.setVersion(transaction.getVersion()); // ← 关键：携带当前版本号
    update.setPaymentStatus(callbackData.getPaymentStatus());
    // ...

    int rows = transactionMapper.updateById(update);
    // MyBatis-Plus自动翻译为：
    // UPDATE PAYMENT_TRANSACTION
    //   SET PAYMENT_STATUS=?, VERSION=VERSION+1, UPDATE_TIME=?
    //   WHERE ID=? AND VERSION=?  ← 版本号不匹配则更新0行
    return rows > 0; // 0行=版本冲突，>0行=成功
}
```

### 乐观锁生命周期

```
初始状态：VERSION = 0

交易记录入库 → VERSION = 0
第一次回调更新 → WHERE VERSION=0 → 成功 → VERSION = 1
第二次重复回调 → WHERE VERSION=0 → 失败（0行）→ 抛出OptimisticLockException
```

## Redis分布式锁详解

### 代码位置（PaymentServiceEnhanced.java）

```java
private PaymentInitResponse doPaymentWithLock(
        PaymentInitRequest request, PaymentGateway gateway, String idempotencyKey) {

    String lockKey = "payment:lock:" + idempotencyKey;
    RLock lock = redissonClient.getLock(lockKey);  // ① 获取锁对象（还没加锁）

    try {
        // ② 尝试加锁：等待最多5秒，加锁成功后持有30秒
        boolean locked = lock.tryLock(lockWaitTime, lockLeaseTime, TimeUnit.SECONDS);

        if (!locked) {
            throw new PaymentException("系统繁忙，请稍后重试"); // 超时未拿到锁
        }

        // ③ 双重检查（拿到锁后再次查DB，防止前一个请求刚刚创建）
        PaymentTransaction existing = transactionMapper.selectByIdempotencyKey(idempotencyKey);
        if (existing != null) {
            return buildResponseFromTransaction(existing); // 已存在，直接返回
        }

        // ④ 真正创建支付
        return doInitiatePayment(request, gateway, idempotencyKey);

    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        throw new PaymentException("系统繁忙，请稍后重试", e);
    } finally {
        // ⑤ 只有当前线程持有时才释放（防止误删他人的锁）
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

### Redis去重（回调去重）

```java
private boolean isDuplicateCallbackByRedis(PaymentCallbackData callbackData) {
    String callbackKey = "payment:callback:processed:" +
                         callbackData.getGatewayTransactionId() + ":" +
                         callbackData.getPaymentStatus();

    // 原子操作：SET key value EX 86400 NX
    // 只有key不存在时才设置，成功=首次，失败=重复
    Boolean result = redisTemplate.opsForValue().setIfAbsent(
        callbackKey,
        System.currentTimeMillis(),
        callbackDedupExpire,   // 86400秒 = 24小时
        TimeUnit.SECONDS
    );

    return Boolean.FALSE.equals(result); // false=key已存在=重复回调
}
```

## 为什么不用悲观锁？

### 悲观锁是什么（这里指的是）

```sql
-- 悲观锁写法（本项目没用）：
SELECT * FROM PAYMENT_TRANSACTION 
WHERE ID = ? 
FOR UPDATE;  -- ← 行级排它锁，其他线程阻塞
```

### 对比分析

| 维度           | 乐观锁（本项目选用）               | 悲观锁（未选用）                     |
|--------------|-------------------------------|----------------------------------|
| **实现方式**   | `@Version` + WHERE version=?  | `SELECT FOR UPDATE`              |
| **加锁时机**   | 更新时才检查冲突                   | 查询时立即加锁                        |
| **线程阻塞**   | ❌ 不阻塞，直接失败                 | ✅ 阻塞等待锁释放                      |
| **并发性能**   | ✅ 高（无等待）                    | ❌ 低（串行化）                        |
| **适用场景**   | 冲突概率低（一笔订单一般只有1次回调）  | 冲突概率高（账户余额、库存扣减）           |
| **死锁风险**   | ✅ 无死锁                         | ❌ 有死锁风险（多表锁顺序问题）             |
| **DB连接占用** | ✅ 低（不持有锁等待）               | ❌ 高（持锁期间连接不释放）               |
| **Oracle兼容**| ✅ 完全兼容                        | ✅ 兼容，但需要显式 FOR UPDATE          |
| **失败处理**   | 返回冲突异常，上层重试或报错           | 等待超时或死锁回滚                      |

### 支付回调场景为什么适合乐观锁

**核心判断依据：冲突概率低**

```
正常情况：
  一笔交易 → 银行只发一次回调 → 冲突概率 ≈ 0%
  
异常情况：
  网络超时，银行重发 → 极低概率 → 第一次成功后版本已变
  → 第二次触发乐观锁冲突 → 直接拦截，没有问题

使用悲观锁的代价：
  每次处理回调都要 SELECT FOR UPDATE
  → 锁定期间（网关验签 + 业务处理 ≈ 100ms~500ms）
  → 所有其他交易的回调都被阻塞
  → 高并发时吞吐量断崖式下跌
```

## 乐观锁冲突场景图解

### 什么是版本号机制（数据库视角）

```
初始状态：
┌────────┬───────────────┬─────────┐
│   ID   │ PAYMENT_STATUS│ VERSION │
├────────┼───────────────┼─────────┤
│  1001  │    PENDING    │    0    │
└────────┴───────────────┴─────────┘

回调A读取：version=0, status=PENDING
回调B读取：version=0, status=PENDING

回调A执行UPDATE：
UPDATE ... SET STATUS='SUCCESS', VERSION=1 WHERE ID=1001 AND VERSION=0
→ 影响行数：1  ✅ 成功

┌────────┬───────────────┬─────────┐
│   ID   │ PAYMENT_STATUS│ VERSION │
├────────┼───────────────┼─────────┤
│  1001  │    SUCCESS    │    1    │  ← VERSION已更新
└────────┴───────────────┴─────────┘

回调B执行UPDATE（但它拿的还是version=0）：
UPDATE ... SET STATUS='SUCCESS', VERSION=1 WHERE ID=1001 AND VERSION=0
→ 影响行数：0  ❌ 乐观锁冲突！

→ 抛出 OptimisticLockException("版本冲突")
→ 上层捕获，记录 processStatus = RETRY，返回200
→ 支付状态没有被破坏 ✅
```

## 有哪些坑？会发生什么问题？

### 🔴 坑1：幂等键策略3（时间戳）导致幂等失效

**代码（PaymentServiceEnhanced.java）：**
```java
// 策略3：订单号 + 金额 + 时间戳（兜底，但不推荐）
String key = request.getOrderReference() + "_" +
             request.getAmount().toString() + "_" +
             System.currentTimeMillis(); // ← 每次调用都不同！
log.warn("使用时间戳幂等键（不推荐），订单号：{}", request.getOrderReference());
```

**问题场景：**
```
用户点击支付 → 请求1（key: ORDER001_100.00_1710000001000）→ 超时
用户重试   → 请求2（key: ORDER001_100.00_1710000001500）→ 成功

结果：两条记录都被插入，同一订单被扣款两次！！
```

**解决方案：**
- 强制要求前端传递 `idempotencyToken`（UUID）
- 或策略2：订单号+支付方式（一般够用）

---

### 🔴 坑2：Redis分布式锁与@Transactional的边界问题

**代码逻辑：**
```java
@Transactional(rollbackFor = Exception.class)  // ← 事务开始
public PaymentInitResponse initiatePayment(...) {
    return doInitiatePaymentWithDistributedLock(...);
}

private PaymentInitResponse doPaymentWithLock(...) {
    RLock lock = redissonClient.getLock(lockKey);
    lock.tryLock(5, 30, SECONDS);

    // ... 业务逻辑 ...

    lock.unlock(); // ← 锁释放了
    // 但事务还没提交！
    // 如果另一个线程在锁释放后、事务提交前进来，
    // 双重检查会查到旧数据（事务隔离）
}
// ← 事务在这里才提交
```

**问题场景（极端情况）：**
```
线程A：获取锁 → 插入记录（未提交）→ 释放锁
线程B：获取锁 → 双重检查（READ COMMITTED，看不到A未提交的数据）→ 再次插入！
              → 触发数据库唯一约束异常（如果有的话，可以作为最终兜底）
```

**解决方案：**
- 在锁内完成整个事务（把`@Transactional`移到`doPaymentWithLock`内部）
- 或数据库加 `UNIQUE INDEX(IDEMPOTENCY_KEY)` 作为终极兜底

### 🟡 坑3：乐观锁冲突后没有自动重试

**当前代码：**
```java
if (!updated) {
    log.warn("由于版本冲突更新交易失败");
    callbackLog.setProcessingStatus("RETRY");
    throw new OptimisticLockException("版本冲突"); // ← 抛异常，没有重试！
}
```

**问题：**
- 银行回调收到非200响应 → 银行认为回调失败 → 继续重发
- 重发时Redis去重会拦截（如果用增强版）
- 如果用基础版`PaymentCallbackService`，Redis没有去重，会无限抛异常

**解决方案：**
```java
// 加入重试逻辑（最多3次）
for (int retry = 0; retry < 3; retry++) {
    PaymentTransaction fresh = transactionMapper.selectById(transaction.getId());
    if (isTerminalStatus(fresh.getPaymentStatus())) return; // 已被其他线程处理
    boolean updated = updateTransactionStatus(fresh, callbackData);
    if (updated) return;
    Thread.sleep(50 * (retry + 1)); // 指数退避
}
throw new OptimisticLockException("多次重试后仍版本冲突");
```

### 🟡 坑4：Redisson未配置时的降级方案不完善

**代码：**
```java
if (redissonClient != null) {
    return doPaymentWithLock(...); // 分布式锁
```

## 面试必问30题

### 🔵 基础锁概念（5题）

**Q1：乐观锁和悲观锁的区别是什么？**

> **乐观锁**：假设不会冲突，提交时验证（通过version/CAS）。读不加锁，写时检查，冲突则失败重试。
> **悲观锁**：假设一定会冲突，操作前先加锁（`SELECT FOR UPDATE`），别人等待。
> 适用场景：乐观锁适合读多写少/冲突低；悲观锁适合写多/冲突高/强一致。

---

**Q2：乐观锁的CAS是什么原理？**

> Compare And Swap（比较并交换）。包含三个值：内存地址V、期望值A、新值B。
> 当且仅当V的值等于A时，才将V更新为B，否则不修改并返回当前值。
> 本项目实现：`UPDATE ... WHERE id=? AND version=?`（SQL层CAS）

---

**Q3：版本号方案和时间戳方案哪个好？**

> 版本号更好：
> - 时间戳在分布式环境中有时钟漂移问题（NTP同步误差）
> - 时间戳精度不够时可能相同（同一毫秒内两次更新）
> - 版本号单调递增，绝对无冲突歧义

---

**Q4：MySQL的MVCC和乐观锁有什么关系？**

> MVCC是MySQL内部实现的多版本并发控制，对应用层透明，用于支持`READ COMMITTED`和`REPEATABLE READ`隔离级别。
> 乐观锁是应用层手动实现的并发控制（version字段）。
> 两者都是"乐观"思想，但层次不同。MVCC解决读写冲突，应用层乐观锁解决写写冲突。

---

**Q5：什么是ABA问题？乐观锁有没有ABA问题？**

> ABA问题：值从A变为B再变回A，CAS检查时看到A误以为没变化。
> 乐观锁（版本号）没有ABA问题：每次更新版本号递增（0→1→2），即使值回滚，版本号也不会回滚。

---

### 🔵 Redis分布式锁（8题）

**Q6：为什么用Redis做分布式锁而不是Zookeeper？**

> | 维度      | Redis（Redisson）     | Zookeeper              |
> |---------|----------------------|----------------------|
> | 性能      | ✅ 极高（内存操作）    | ❌ 较低（磁盘+网络往返） |
> | 可靠性    | 主从切换可能丢锁       | ✅ CP强一致（Paxos）   |
> | 实现复杂度 | ✅ 简单（SET NX EX） | ❌ 复杂（临时节点+Watch）|
> 支付场景：对性能要求高，短暂重复支付通过幂等键可兜底，选Redis合理。

---

**Q7：Redisson的`tryLock`和普通`lock`有什么区别？**

> - `lock()`：一直阻塞到获取锁（危险！可能永久等待）
> - `tryLock(waitTime, leaseTime, unit)`：最多等待`waitTime`，获得后持有`leaseTime`后自动释放
> 本项目：`tryLock(5, 30, SECONDS)` → 等5秒、持有30秒

---

**Q8：Redisson为什么能防止锁过期后被误删？**

> 普通`SET NX EX`问题：A持锁，业务超时，锁自动过期，B获得锁，A执行完后删除了B的锁。
> Redisson解决：
> 1. 看门狗（Watchdog）：每10秒自动续期，避免业务未完成时锁过期
> 2. 释放前校验：`lock.isHeldByCurrentThread()` → 只有当前线程才能释放
> 本项目代码：`if (lock.isHeldByCurrentThread()) { lock.unlock(); }`

---

**Q9：什么是Redlock（Redis红锁）？本项目有用到吗？**

> Redlock是Redis官方提出的多节点分布式锁方案：向N个（通常5个）独立Redis节点申请锁，超过半数成功才算获锁，防止单点失败。
> 本项目没用Redlock，只用了单节点Redisson锁。
> 对于支付场景，建议使用Redlock，因为Redis主从切换可能导致锁丢失。

---

**Q10：如果Redis宕机了，分布式锁失效，会发生什么？**

> 本项目有降级方案：
> ```java
> if (redissonClient != null) {
>     return doPaymentWithLock(...);
> } else {
>     log.warn("未配置Redis，仅适用于单机");
>     return doPaymentWithoutLock(...); // 仅DB查重
> }
> ```
> 但降级后在多实例部署时，两个实例可能同时通过查重检查，需要DB唯一约束兜底。

---

**Q11：Redis的`setIfAbsent`为什么能保证原子性？**

> `setIfAbsent`对应Redis的`SET key value NX EX seconds`命令。
> Redis是单线程执行命令，所以这是一个原子操作——判断key不存在和设置key是同一个命令，不会有并发问题。
> 注意：不能用先`exists`再`set`的两步操作（非原子，有竞态）。

---

**Q12：为什么回调去重用Redis而支付发起用Redisson？**

> 回调去重：用`setIfAbsent`，只需要"标记已处理"，不需要"排队等待"。一次成功，后续拦截。
> 支付发起：用Redisson`tryLock`，需要"互斥"，同一时间只有一个请求进行支付创建流程，避免并发创建重复交易。
> 本质区别：setIfAbsent是"幂等标记"，Lock是"临界区保护"。

---

**Q13：如果两个回调同时通过了Redis去重（极端情况），如何保证安全？**

> 本项目有三层防护：
> 1. Redis setIfAbsent（大多数情况拦截）
> 2. 终态检查（一个更新成功后状态变为SUCCESS，另一个判断终态直接返回）
> 3. 乐观锁version（即使两个都过了前两关，UPDATE WHERE version=? 只有一个能成功）
> 三层机制确保最终一致性。

---

### 🔵 事务与并发（8题）

**Q14：本项目的事务隔离级别是什么？为什么这样设置？**

> 回调处理设置了`isolation = Isolation.READ_COMMITTED`（提交读）。
> 原因：
> - 避免脏读（读到其他事务未提交的修改）
> - 不用可重复读（REPEATABLE READ）是因为：每次处理回调都是独立的新数据，不需要多次读取一致性
> - Oracle默认就是READ COMMITTED，显式声明是为了代码可读性

---

**Q15：`@Transactional(propagation = Propagation.REQUIRES_NEW)`是什么意思？**

> REQUIRES_NEW：无论外层是否有事务，都新建一个独立事务。外层事务挂起，内层独立提交/回滚。
> 回调处理用这个：
> - 回调是异步的（@Async），本来就没有外层事务
> - 即使有外层，也要独立提交回调日志（即使业务失败，日志也要留下）

---

**Q16：@Async + @Transactional一起用有什么注意事项？**

> 两者配合正常工作，但要注意：
> 1. @Async会让方法在新线程执行，@Transactional绑定的是线程本地（ThreadLocal），所以各自独立事务，没有冲突。
> 2. 主线程的事务不会传播到异步线程（异步线程有自己的事务上下文）。
> 3. @Async方法必须在不同类中调用（同类调用绕过AOP代理，@Async不生效）。

---

**Q17：为什么回调处理不用`synchronized`而用乐观锁？**

> `synchronized`只在单JVM有效，多实例部署时失效。
> 乐观锁在数据库层面生效，多实例都遵守同一个version约束。
> 本项目是分布式系统（多实例部署），`synchronized`完全不够用。

---

**Q18：幂等性、去重、乐观锁三者的关系是什么？**

> - **幂等性**：同一操作执行多次结果相同（业务目标）
> - **去重**：防止同一请求被处理多次（技术手段，快速判断）
> - **乐观锁**：防止并发写入冲突（技术手段，数据一致性）
> 
> 关系：去重和乐观锁都是实现幂等性的技术手段，去重在前（快速）、乐观锁在后（兜底）。

---

**Q19：MyBatis-Plus乐观锁在底层翻译成什么SQL？**

> 原代码：`transactionMapper.updateById(update)`（entity里有@Version字段）
> 
> 翻译后的SQL：
> ```sql
> UPDATE PAYMENT_TRANSACTION
>   SET PAYMENT_STATUS = ?,
>       VERSION = VERSION + 1,   -- 自动加1
>       UPDATE_TIME = ?
>   WHERE ID = ?
>     AND VERSION = ?             -- 带入原版本号
> ```
> 如果版本号不匹配，影响行数为0，MyBatis-Plus返回0，业务判断失败。

---

**Q20：乐观锁失败后，异常会触发事务回滚吗？**

> 会！`@Transactional(rollbackFor = Exception.class)`遇到任何异常都回滚。
> 但乐观锁失败时，通常只是`updateById`返回0，本项目通过返回值判断，不抛异常。
> 如果手动`throw new OptimisticLockException()`，则触发回滚（连已记录的callback_log也回滚）。
> 这是一个需要注意的副作用：finally里的callbackLog插入也会被回滚！

---

**Q21：为什么回调服务要用`REQUIRES_NEW`事务而不是默认的`REQUIRED`？**

> 回调是异步处理，默认没有外层事务，REQUIRED效果等同于REQUIRES_NEW。
> 但显式声明REQUIRES_NEW有明确语义：确保每次回调处理都是独立事务，不受任何外部事务影响。
> 另外，如果未来改成同步处理且在某个大事务内调用，REQUIRES_NEW确保回调日志一定被提交（即使外层事务回滚）。

### 🔵 系统设计（9题）

**Q22：支付系统为什么要有幂等性？**

> 支付系统常见场景：
> 1. 网络超时 → 用户重试 → 两次请求到达服务器
> 2. 银行回调 → 服务器处理中断 → 银行再次回调
> 3. MQ消息重发 → 消费者重复消费
> 
> 没有幂等性 → 用户被多次扣款 → 资金损失 → 法律风险
> 幂等性 = 支付系统的基本生命线

---

**Q23：支付状态机设计是什么？**

> 本项目状态流转（单向，不可逆）：
> ```
> INIT → PENDING → SUCCESS → REFUNDED
>      ↘          ↘
>       FAILED    FAILED
>      ↘
>       TIMEOUT
> ```
> 终态：SUCCESS / FAILED / REFUNDED（一旦到达，不可修改）
> 状态机保证：不能从SUCCESS变回PENDING，不能重复扣款

---

**Q24：支付系统的对账是什么？**

> 对账 = 核对本地交易记录与银行记录是否一致
> 本项目`PaymentReconciliationJob`（定时任务）：
> 1. 查询本地PENDING超过N分钟的交易
> 2. 主动调用银行API查询最新状态
> 3. 比较结果：MATCHED（一致）/ MISMATCH（不一致，告警）/ PENDING（银行也不确定）
> 对账是支付系统的最后一道安全网，弥补异步回调的漏洞。

---

**Q25：如果网关回调超时了怎么办？本项目是怎么处理的？**

> 本项目的处理链：
> 1. 银行发回调 → 我们异步处理 → 处理完返回200
> 2. 如果我们宕机，银行一般会重试N次（通常1分钟、5分钟、15分钟...）
> 3. 重试时Redis去重拦截或乐观锁保护
> 4. 如果银行重试都失败：定时对账Job主动查询银行状态补偿

---

**Q26：本项目的异步处理有什么风险？**

> `@Async`使用自定义线程池`callbackExecutor`，风险：
> 1. **队列满**：大量回调时线程池队列满，新回调被拒绝（银行认为失败，重试）
> 2. **线程异常**：异步线程异常不会传播到调用方（回调controller已返回200）
> 3. **事务失效**：需要确保异步线程内有独立事务（本项目已处理）
> 4. **应用重启**：线程池队列中未处理的消息丢失（建议用MQ持久化）

---

**Q27：如何防止退款被重复执行？**

> 当前代码（`PaymentService.refund()`）只检查状态是否为SUCCESS，没有防重复退款！
> 问题：连续调用两次退款API，都可能通过状态检查（两次都是SUCCESS）
> 
> 改进方案：
> ```java
> // 1. 退款也需要幂等键（refundNo唯一约束）
> // 2. 退款时用乐观锁：UPDATE WHERE STATUS='SUCCESS' AND VERSION=?
> // 3. 退款成功后状态改为REFUNDED（终态），下次检查直接拒绝
> ```

---

**Q28：支付系统如何保证资金安全？**

> 多层保障：
> 1. **幂等性**（Redis锁 + DB约束）：防止重复扣款
> 2. **签名验证**：防止伪造回调（`verifyCallback`）
> 3. **状态机**：防止状态非法转换（终态不可逆）
> 4. **乐观锁**：防止并发写入错误
> 5. **对账**：发现并修复异常交易
> 6. **事务**：原子性保证操作完整
> 7. **日志**：`CALLBACK_LOG`全量记录，可审计

---

**Q29：如果让你设计一个支付系统的分布式事务方案，你怎么做？**

> 支付场景适合**最终一致性**方案：
> 
> **Saga模式（推荐）：**
> ```
> 支付成功 → [发MQ消息] → 订单系统消费 → 更新订单状态
>          ↓ 失败            ↓ 消费失败
>       补偿事务           重试 + 死信队列
> ```
> 
> **本地消息表方案：**
> - 支付事务成功时，同库写入一条消息记录（原子）
> - 定时扫描未发送消息，投递到MQ
> - 消费端幂等消费
> 
> **不推荐**：2PC/XA（性能差，DB长期持锁）、Seata AT（侵入性强）

---

**Q30：面试官问："这个系统在高并发下会有什么问题？"怎么回答？**

> **已解决的问题：**
> - 重复支付：Redis分布式锁 + 幂等键
> - 并发回调：三层防护（Redis去重 + 终态检查 + 乐观锁）
> 
> **需要进一步优化的问题：**
> 1. **热点订单**：同一订单被高频查询，加Redis缓存
> 2. **回调洪峰**：银行大量回调时，需要限流（Sentinel）+ 削峰（MQ缓冲）
> 3. **DB连接数**：高并发时Oracle连接池可能耗尽，需要连接池调优（HikariCP）
> 4. **网关超时**：调用渣打API超时，需要熔断降级（Resilience4j）
> 5. **单点Redis**：Redis主节点挂了，分布式锁失效，需要Redis Cluster或Redlock
> 6. **对账效率**：百万级交易对账时，需要分片 + 并行处理

### 修复1：退款重复执行（🔴 高危，已修复）

**修复方案：引入 `REFUNDING` 中间状态 + 乐观锁原子抢占**

```java
// Step1：原子性地把状态 SUCCESS → REFUNDING（乐观锁保护）
PaymentTransaction claimUpdate = new PaymentTransaction();
claimUpdate.setId(transaction.getId());
claimUpdate.setVersion(transaction.getVersion());  // ← 携带当前版本
claimUpdate.setPaymentStatus(PaymentStatus.REFUNDING.name());

int claimed = transactionMapper.updateById(claimUpdate);
// MyBatis-Plus SQL：UPDATE ... SET STATUS='REFUNDING', VERSION=VERSION+1
//                  WHERE ID=? AND VERSION=?
//                  只有第一个线程：影响行数=1 ✅
//                  第二个线程：VERSION已经变了，影响行数=0 → 直接报错

if (claimed == 0) {
    throw new PaymentException("退款正在处理中，请勿重复提交");
}

// Step2：凭独占权调用银行网关（此刻状态=REFUNDING，其他线程已被拦截）
GatewayRefundResponse response = gateway.refund(request);

// Step3：根据结果更新最终状态
if (response.isSuccess()) {
    → REFUNDED（终态）
} else {
    → REFUND_FAILED（需人工干预）
}
```

**同时新增状态枚举（`PaymentStatus.java`）：**
```java
REFUNDING("退款处理中"),   // 中间状态，防并发
REFUND_FAILED("退款失败"), // 网关异常，需人工核查
```

**修复后的状态机：**
```
INIT → PENDING → SUCCESS → REFUNDING → REFUNDED（终态）
                         ↘ REFUND_FAILED（需人工干预）
```

## 分布式事务深度对比

### 为什么支付系统需要分布式事务？

```
支付成功后，需要同时做三件事：
  ① 更新 PAYMENT_TRANSACTION 状态 = SUCCESS   （支付DB）
  ② 通知订单系统：ORDER_001 已付款           （订单DB）
  ③ 给用户发短信通知                          （短信服务）

问题：这三件事在不同服务/数据库里，不是一个本地事务！
      ② 或 ③ 失败了，① 要不要回滚？
```

---

### 方案一：2PC / XA（两阶段提交）

#### 是什么

> 由一个**协调者**分两个阶段统一协调所有**参与者**（数据库）的提交/回滚。

```
阶段1 - Prepare（预提交）：
  协调者 → 参与者A：你能提交吗？→ A：能（锁住资源，写redo log，回复Yes）
  协调者 → 参与者B：你能提交吗？→ B：能（锁住资源，写redo log，回复Yes）

阶段2 - Commit（真正提交）：
  协调者 → 参与者A：提交！→ A：提交完成，释放锁
  协调者 → 参与者B：提交！→ B：提交完成，释放锁
```

#### 最坏的结果时序图

```
时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

T1  协调者   → 支付DB(A)：Prepare   支付DB回复：Yes，行锁已持有
T2  协调者   → 订单DB(B)：Prepare   订单DB回复：Yes，行锁已持有
T3  协调者   → 支付DB(A)：Commit    
T4  ⚡ 协调者宕机！！！（Commit消息发出去但没收到ACK）
                                     
    支付DB(A)：已提交 ✅  状态=SUCCESS （支付钱已扣）
    订单DB(B)：一直等待 Commit 指令... ← 行锁没有释放！

T5  订单DB(B) 持有行锁等待... 5秒
T6  订单DB(B) 持有行锁等待... 30秒
T7  订单DB(B) 持有行锁等待... 5分钟
    └─ 所有读这个订单的查询全部阻塞！
    └─ 用户刷订单列表：转圈圈...

T8  DBA介入，手动处理，或等协调者恢复
    └─ 协调者恢复后重新发 Commit → 订单DB(B) 提交
    └─ 总计阻塞时间：N 分钟

最坏结果：
  - 支付DB已扣款，但订单状态长时间不更新（用户：我的订单呢？）
  - 订单行锁长期持有，其他用户查询该订单超时
  - 协调者是单点，宕机期间所有分布式事务全部卡死
  - DB连接耗尽：等待锁的连接堆积，引发雪崩
```

**缺点总结：**
- 协调者单点故障 → 所有参与者阻塞
- 参与者持锁等待（阶段1到阶段2之间全程持锁）
- 不适合高并发：100 TPS * 200ms = 20 个并发锁
- 跨数据库（Oracle + MySQL）实现复杂

---

### 方案二：Seata AT 模式

#### 是什么

> Seata 框架的 AT（Auto Transaction）模式：**自动生成回滚SQL**，对业务代码侵入极小（加一个注解）。

```
工作原理：
  Seata 代理数据源，在每次 SQL 执行时：
    - 执行前：保存 before_image（旧数据快照）
    - 执行后：保存 after_image（新数据快照）
    - 生成 undo_log 记录

  需要回滚时：用 undo_log 的 before_image 生成反向 SQL：
    UPDATE ... SET STATUS='PENDING' WHERE ID=? AND STATUS='SUCCESS'
```

#### 最坏的结果时序图

```
时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

T1  全局事务开始，Seata TC（事务协调器）分配 XID=12345

T2  支付服务：UPDATE STATUS='SUCCESS'
    → Seata 代理保存 undo_log：
      before: {STATUS:'PENDING', VERSION:1}
      after:  {STATUS:'SUCCESS', VERSION:2}

T3  订单服务：UPDATE ORDER_STATUS='PAID'
    → 保存 undo_log

T4  ⚡ 脏读问题出现！！
    ─────────────────────────────────────────
    此时全局事务尚未提交（T2、T3 都只是分支提交）
    另一个查询线程：SELECT * FROM PAYMENT_TRANSACTION WHERE ID=?
    → 读到了 STATUS='SUCCESS'（AT模式默认是本地提交，其他线程可读）
    → 但全局事务还没确认！
    → 如果接下来全局回滚... 这条"已成功"的数据其实是"脏数据"！
    ─────────────────────────────────────────

T5  订单服务：UPDATE ORDER_STATUS 失败（库存不足）

T6  Seata TC 发出全局回滚指令

T7  支付服务收到回滚：
    执行反向SQL：UPDATE STATUS='PENDING' WHERE ID=? AND STATUS='SUCCESS'
    ← 但如果 T4 那个线程已经基于"SUCCESS"做了后续业务呢？
       例如：已经发了"支付成功"短信 ← 短信无法撤回！

T8  ⚡ 另一个更严重的问题：undo_log 被篡改
    ─────────────────────────────────────────
    在 T4 的脏读期间，另一个事务把数据改成了 STATUS='REFUNDED'
    Seata 回滚时：before_image 是 PENDING
    当前值是 REFUNDED（已被改）
    → undo_log 校验失败！→ 抛出异常！
    → 数据不一致！需要人工干预！
    ─────────────────────────────────────────

最坏结果：
  - 脏读：全局事务未提交时，本地事务已可见（READ COMMITTED 隔离级别）
  - 短信/通知已发，但事务回滚，业务语义矛盾
  - undo_log 被并发事务破坏，回滚失败，数据永久不一致
  - 每张业务表都需要一张对应的 UNDO_LOG 表（侵入性强）
  - Seata TC 宕机 → 全局事务阻塞
```

**缺点总结：**
- 脏读问题（AT 模式是 READ COMMITTED，不是全局隔离）
- 需要额外的 UNDO_LOG 表，改造成本高
- 需要部署 Seata Server（TC），多一个基础设施
- 不适合支付核心链路（脏读风险无法接受）

---

### 方案三：Saga 模式（本项目推荐）

#### 是什么

> 把一个长事务拆成多个本地事务，每个本地事务成功后发消息触发下一个；**失败时执行补偿操作（逆向操作）**。没有全局锁，没有协调者单点。

```
正向流程：
  支付成功 → [发MQ消息] → 订单服务消费 → [发MQ消息] → 短信服务消费
  
补偿流程（某步失败）：
  短信发送失败 → 无需补偿（短信不影响资金）
  订单更新失败 → 补偿：把订单状态改回去 → 通知支付服务退款
```

#### 最坏的结果时序图（对比前两个，Saga 的"最坏"轻得多）

```
时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

T1  支付服务：UPDATE STATUS='SUCCESS' ← 本地事务立即提交 ✅
    发送 MQ 消息：{type: PAYMENT_SUCCESS, orderId: ORDER_001}

T2  MQ 消息持久化到 Broker ✅

T3  订单服务消费消息：UPDATE ORDER_STATUS='PAID'
    ⚡ 订单服务宕机！！消费失败！

T4  MQ Broker 检测到消费失败 → 自动重试（1s, 5s, 30s, ...）

T5  订单服务重启 → 重新消费消息 → 幂等消费（UPDATE WHERE STATUS!='PAID'）
    → 最终消费成功 ✅

最坏情况1（MQ 消息丢失）：
  T1  支付成功，消息发送失败（MQ Broker 宕机）
  T2  本地没有保存"待发消息"记录
  →   结果：支付成功，但订单永远不知道
  →   解决：本地消息表（把消息写在同一个DB事务里）

本地消息表方案：
  ┌─────────────────────────────────────────────────┐
  │  BEGIN TRANSACTION (同一个本地事务)               │
  │    UPDATE PAYMENT_TRANSACTION SET STATUS='SUCCESS'│
  │    INSERT INTO MESSAGE_OUTBOX                    │
  │      (type, payload, status='PENDING')           │
  │  COMMIT  ← 原子提交，要么都成，要么都不成         │
  └─────────────────────────────────────────────────┘
  定时任务扫描 PENDING 消息 → 投递 MQ → 更新为 SENT

最坏情况2（补偿失败）：
  订单服务补偿重试了 5 次，仍然失败（订单DB宕机超过重试间隔）
  → 进入死信队列（DLQ）→ 人工处理
  → 支付已成功但订单未更新，需要人工运营介入
  → 最终结果：数据最终一致，延迟可能几分钟到几小时
  → 不会出现资金问题，只是订单状态延迟
```

**Saga 的代价（比前两个轻得多）：**
- 最终一致性，不是强一致性（有时间窗口）
- 需要为每个操作写补偿逻辑
- 业务逻辑复杂度增加

```
对比维度          2PC/XA              Seata AT            Saga（推荐）

一致性类型         强一致              准强一致             最终一致
```

## 异步（@Async）是做什么用的？

### 一句话：银行回调不能让你慢慢处理

支付系统收到银行回调时，需要做很多事：验签名、查数据库、更新状态、通知订单系统……  
这些加起来可能要 **200ms～500ms**。

问题来了：

```
银行服务器发送回调 POST /callback
       ↓
等待你的服务器响应（HTTP 连接挂着）
       ↓
如果你超过 5~30 秒没有回复
       ↓
银行认为回调失败 → 再重发 → 再重发 → 再重发...
（渣打银行标准：重试 5次，间隔1分钟/5分钟/15分钟/30分钟/60分钟）
```

### @Async 的作用：立刻回复，后台慢慢处理

```
银行 → POST /callback
            ↓
       Controller 收到请求
            ↓
       callbackService.processCallback(...)  ← @Async：扔进线程池，立刻返回
            ↓
       立刻 return "SUCCESS"  ← HTTP 连接关闭
            ↓
       银行收到 200，认为成功，不会重发
            ↓ （后台 callback- 线程池继续处理）
       验签名 → 查DB → 乐观锁更新 → 通知订单系统
```

### 代码对照（AsyncConfig.java）

```java
// 这就是那个"后台线程池"
@Bean("callbackExecutor")
public Executor callbackExecutor() {
    executor.setCorePoolSize(10);   // 常驻10个线程，随时待命
    executor.setMaxPoolSize(50);    // 最多扩到50个（应对回调洪峰）
    executor.setQueueCapacity(100); // 最多排队100个（超过则由HTTP线程直接处理）
    executor.setRejectedExecutionHandler(
        new ThreadPoolExecutor.CallerRunsPolicy() // 队列满→HTTP线程兜底，背压保护
    );
    executor.setWaitForTasksToCompleteOnShutdown(true); // 优雅关闭：先把队列处理完
}

// 回调方法用 @Async 标注
@Async("callbackExecutor")
@Transactional(propagation = REQUIRES_NEW)
public void processCallback(String rawBody, ...) {
    // 这段代码在 callback-1, callback-2... 线程上运行，不占用 HTTP 线程
}
```

### 不用 @Async 会发生什么？（对比时序）

```
不用 @Async（同步处理）：
银行         Tomcat HTTP线程(共200个)     DB
  │                    │                  │
  │── 回调1 ───────────→│                  │
  │                    │── 验签(50ms) ─────│
  │                    │── 查DB(100ms) ────│
  │                    │── 更新(50ms) ─────│
  │                    │── 通知订单(200ms) ─│  ← 整个过程 HTTP线程被占用！
  │← 200 OK（400ms后）─│                  │
  │                    │                  │
  银行在等待期间连接挂着，消耗1个HTTP线程

高峰期同时来100个回调：
  → 100个 HTTP 线程被占用（Tomcat默认200个线程）
  → 剩余100个线程处理普通用户请求
  → 如果回调处理慢（DB慢），HTTP 线程耗尽
  → 用户发起支付请求：连接超时！（明明是回调搞的鬼）

用 @Async（异步处理）：
银行         Tomcat HTTP线程     callback-线程池(50个)     DB
  │                 │                    │                  │
  │── 回调1 ────────→│                   │                  │
  │                 │── 扔进线程池 ───────→│                  │
  │← 200 OK(1ms) ───│                   │── 验签/查DB/更新 ─→│
  │  （HTTP线程立刻释放）                 │   （后台慢慢来）  │
  │── 回调2 ────────→│                   │                  │
  │← 200 OK(1ms) ───│  HTTP线程从不阻塞  │                  │
```

### AsyncConfig 当前参数够不够用？

```
当前配置：core=10, max=50, queue=100

支撑能力估算：
  每个回调处理时间 ≈ 200ms
  50个线程 × (1000ms / 200ms) = 250 TPS 回调处理能力
  
  正常情况：每秒支付笔数 ≤ 250 → 足够
  峰值情况：回调积压进入队列（100个缓冲）
  极端情况：CallerRunsPolicy 让HTTP线程兜底，自动降速

如果要支撑 1000 TPS 回调：
  → 把 maxPoolSize 调到 200
  → queueCapacity 调到 1000
  → 或者把回调处理改成 MQ 消费（无上限）
```

## 性能极限分析：1000万并发会怎样？

### 先搞清楚"1000万同时支付"的真实含义

```
"1000万同时支付" 实际上是：
  1000万笔请求在 1秒内 同时到达 → QPS = 10,000,000

对比现实：
  双十一高峰：支付宝 峰值 54万笔/秒（54万 QPS）
  微信支付：  高峰   约 100万 QPS
  本系统（单机 Spring Boot）：预估 500~2000 QPS（取决于DB性能）
  
  所以 1000万 QPS = 约是支付宝高峰的 18倍，绝对宕机
```

### 1000万并发打进来会发生什么（时序）

```
T+0s    1000万请求同时到达 Nginx/负载均衡
T+0.1s  Tomcat 线程池（200线程）瞬间全满
         → 请求队列（默认100）瞬间全满
         → 新请求被拒绝（Connection Refused 或 503）
         → 前端：支付失败，请重试
T+0.2s  Redis 连接池（默认50）全满
         → "Cannot get a connection, pool is exhausted"
         → 获取分布式锁失败，抛异常
T+0.3s  Oracle 连接池（HikariCP，默认10）全满
         → "Connection is not available, timeout after 30000ms"
         → 所有DB操作超时
T+0.5s  JVM GC 压力：堆内大量 Request 对象 → Full GC
         → STW（Stop The World）几百ms
         → 所有线程暂停 → 请求全部超时
T+1s    OOM（Out Of Memory）可能触发
         → 应用进程 Kill
         → 服务器宕机

总结：不是"可能宕机"，是"1秒内必定宕机"
```

### 本系统各层的实际瓶颈

```
层次              当前配置           实际瓶颈 QPS     备注
──────────────────────────────────────────────────────────
Tomcat HTTP       maxThreads=200     ~500 QPS         每请求处理约400ms
Redis 分布式锁    连接池=50          ~2000 QPS        内存操作快
Oracle DB         HikariCP=10        ~100~300 QPS     INSERT是瓶颈
callback线程池    max=50             ~250 QPS         异步不占HTTP线程
网关调用(渣打)    HTTP超时5s         ~40 QPS          外部依赖，瓶颈所在

整个系统的最终瓶颈 = min(所有层) ≈ 100~300 QPS（DB限制）
```

## 限流方案：Guava vs Sentinel（削峰填谷）

### 为什么要限流？"越多人付钱不是越好吗"

> 这是面试官最喜欢问的问题，答案分三层：

**第一层（直觉层）**：钱越多越好，不应该拒绝用户。  
**第二层（技术层）**：系统有物理上限，超过就全部崩溃，所有人都付不了。  
**第三层（金融安全层）**：支付系统异常洪峰 ≠ 正常流量，可能是 DDoS 攻击、账号盗刷、程序 bug 导致的死循环请求，需要保护。

```
没有限流的结果（1000万并发）：
  1000万人 → 全部失败 → 0笔成功 → 收入 = 0

有限流的结果（系统容量300 QPS）：
  300人/秒 → 全部成功 → 收入 = 300笔/秒
  剩余 999.97万人 → 排队或友好提示"稍后重试"
  
  随着时间推移（30秒内）：
  300 × 30 = 9000笔成功 → 大部分人最终完成支付

限流 = 保护系统稳定 = 保证大多数人能付款成功 = 收入最大化
```

### 方案一：Guava RateLimiter（单机，已在项目结构中）

```
适用场景：单机部署 / 简单限流 / 快速接入
限流维度：单个 JVM 进程内的全局限速
```

#### 核心原理（令牌桶算法）

```
令牌桶示意：
  系统以固定速率往桶里放令牌（如 300个/秒）
  每个请求来了先拿令牌，拿到才能处理，没拿到等待或拒绝

  时间轴：
  0s   桶里有 300 令牌
  0s   来了 1000 请求 → 300 个拿到令牌处理 → 700 个等待/拒绝
  1s   桶里又有 300 令牌 → 700 等待中的 300 个继续处理
  2s   桶里再有 300 令牌 → 最后 400 个处理
  
  效果：流量被"削平"到 300 QPS，系统稳定
```

#### 本项目接入代码（可直接用）

```java
// 新建：PaymentRateLimiterConfig.java
@Configuration
public class PaymentRateLimiterConfig {
    
    // 支付发起限流：300笔/秒（根据服务器实际QPS调整）
    @Bean("paymentRateLimiter")
    public RateLimiter paymentRateLimiter() {
        return RateLimiter.create(300.0); // permits per second
    }
    
    // 回调处理限流：500次/秒（回调比支付多，因为有重发）
    @Bean("callbackRateLimiter")
    public RateLimiter callbackRateLimiter() {
        return RateLimiter.create(500.0);
    }
}

// 修改：PaymentController.java
@Autowired
@Qualifier("paymentRateLimiter")
private RateLimiter paymentRateLimiter;

@PostMapping("/initiate")
public ApiResponse<PaymentInitResponse> initiatePayment(@RequestBody PaymentInitRequest request) {
    // tryAcquire(0)：不等待，立刻返回是否拿到令牌
    if (!paymentRateLimiter.tryAcquire()) {
        return ApiResponse.error("系统繁忙，请稍后重试");  // 友好提示
    }
    // 拿到令牌，正常处理
    return ApiResponse.success(paymentService.initiatePayment(request));
}
```

#### Guava 的问题（单机方案天花板）

```
问题1：单机限流，多实例不协调
  实例A 限流 300/s
  实例B 限流 300/s
  → 实际流量 600/s 打到DB，DB还是撑不住

问题2：重启后令牌桶重置
  → 重启瞬间没有历史状态，可能短暂放开大量请求

问题3：无法动态调整
  → 修改限流阈值需要重新部署
  
适合场景：单机部署、TPS < 500、快速上线
```

### 方案二：Sentinel（微服务，推荐）

```
适用场景：微服务/分布式部署 / 需要动态调整 / 需要多维度保护
限流维度：集群级别（所有实例合计）、接口级别、用户级别
```

#### 和 Guava 的核心区别

```
Guava（单机）：               Sentinel（集群）：
  实例A: 300/s 令牌桶           所有实例共享 300/s 配额
  实例B: 300/s 令牌桶           实例A 用了 200/s
  实例C: 300/s 令牌桶           实例B 用了 100/s
  ────────────────             → 总量不超过 300/s ✅
  总放行 900/s（超标！）          DB 安全
```

#### 本项目接入代码

```java
// pom.xml 新增依赖
// <dependency>
//   <groupId>com.alibaba.cloud</groupId>
//   <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
// </dependency>

// application.yml
// spring.cloud.sentinel.transport.dashboard: localhost:8080

// PaymentController.java
@PostMapping("/initiate")
@SentinelResource(
    value = "payment:initiate",        // 资源名称
    blockHandler = "handleBlock",      // 被限流时调用
    fallback = "handleFallback"        // 业务异常时调用
)
public ApiResponse<PaymentInitResponse> initiatePayment(@RequestBody PaymentInitRequest request) {
    return ApiResponse.success(paymentService.initiatePayment(request));
}

// 限流降级处理
public ApiResponse<PaymentInitResponse> handleBlock(
        PaymentInitRequest request, BlockException ex) {
    log.warn("支付限流触发，订单号：{}", request.getOrderReference());
    return ApiResponse.error("系统繁忙，请稍后重试（限流）");
}

public ApiResponse<PaymentInitResponse> handleFallback(
        PaymentInitRequest request, Throwable ex) {
    log.error("支付降级触发", ex);
    return ApiResponse.error("支付服务暂时不可用，请稍后重试");
}
```

#### Sentinel 控制台配置（无需改代码）

```
Sentinel Dashboard 界面动态配置：

资源名：payment:initiate
规则类型：流量控制
阈值类型：QPS
单机阈值：100（每个实例100，3个实例总计300）

熔断降级（额外保护）：
  资源名：payment:initiate
  熔断策略：异常比例
  阈值：50%（超过50%请求失败，自动熔断10秒）
  ← 防止DB故障时雪崩
```

### 削峰填谷是什么意思？

```
没有削峰（裸奔）：
请求量  ████████████████████  ← 瞬间洪峰 1000 QPS
服务器  ██████    宕机 !!!

有削峰（令牌桶/MQ缓冲）：
请求量  ████████████████████  ← 洪峰 1000 QPS
限流层  ████                  ← 只放 300 QPS，其余排队/拒绝
服务器  ████ ████ ████ ████   ← 平稳处理，不宕机

填谷 = 低峰期提前处理排队请求（MQ积压慢慢消化）：
凌晨3点  → 正常 10 QPS
服务器   → 100 QPS 处理能力闲置
MQ里     → 还有积压的 10000 笔请求
           → 利用空闲时间全部处理完

最终：每一笔支付请求都会被处理，只是有些延迟
      不会因为瞬间洪峰而丢失请求（除非明确配置超时丢弃）
```

### 除了物理扩容，还有哪些其他办法？

> 面试官最喜欢问：**"不加机器，怎么提升系统处理能力？"**

```
纯软件优化层面（不买服务器）：
────────────────────────────────────────────────────────
① 异步化（已实现）
   同步 → 异步 @Async
   效果：回调处理不占 HTTP 线程，吞吐量提升 3~5x

② 数据库连接池调优
   HikariCP: maxPoolSize 从 10 → 50
   效果：并发 DB 操作从 10 → 50，提升 5x
   注意：Oracle 需要对应的 session 授权

③ 批量写入（Batch Insert）
   回调日志：从逐条 INSERT → 批量 INSERT（每 100 条提交一次）
   效果：DB 写 QPS 提升 10~20x

④ Redis 缓存热点数据
   支付状态查询：SELECT → Redis GET（1ms vs 10ms）
   效果：读 QPS 从 500 → 50000

⑤ 读写分离（Oracle Data Guard）
   写操作 → 主库
   查询操作 → 只读副本
   效果：主库压力减半，查询 QPS 翻倍

⑥ 限流 + 削峰（上面讲的）
   效果：保护系统稳定，不是提升吞吐，是保证不宕机

⑦ MQ 解耦（终极方案）
   支付请求 → 写入 MQ（微秒级）→ 消费者慢慢处理
   接入层 QPS 从 300 → 100000（MQ 是内存操作）
   处理层 QPS 还是 300，但不会阻塞前端

物理扩容层面（加机器）：
────────────────────────────────────────────────────────
⑧ 水平扩展（加实例）
   当前：1台 300 QPS
   扩到：3台 900 QPS（需要 Nginx 负载均衡）
   注意：DB 是共享的，不能无限扩，最终瓶颈还是 DB

⑨ 数据库分片（Sharding）
   按订单号哈希，分到 3 个 Oracle 实例
   效果：DB QPS 从 300 → 900
   代价：跨分片查询复杂，事务需要分布式事务

⑩ CDN + 静态资源分离
   支付页面静态资源走 CDN
   只有支付 API 打到服务器
   效果：服务器压力降低 60%~80%
```

#### 架构演进路线（面试标准答案）

```
阶段1（当前）：单机 + Oracle
  瓶颈：300 QPS，靠限流保护

阶段2（短期）：多实例 + Redis Sentinel + 连接池调优
  可支撑：1000 QPS，Sentinel 集群限流

阶段3（中期）：MQ 削峰 + Redis 缓存 + 读写分离
  可支撑：10000 QPS，支付请求异步化

阶段4（长期）：数据库分片 + 多地多活
  可支撑：100000+ QPS，金融级高可用

每个阶段的核心原则：
  先用限流保稳定 → 再做异步提吞吐 → 再做缓存降延迟 → 最后做分片扩容
  不要第一步就想着分片，成本太高，收益不确定
```

### 面试连环问答（关于限流和性能）

**Q：为什么要限流？不是越多人付钱越好吗？**
> 没有限流时，1000人同时付钱 → 系统崩溃 → 0人成功 → 收入为0。有限流时，系统稳定在300 QPS，其余人排队，最终大部分人都能付款成功。**限流是为了让更多人能付款成功，不是为了拒绝用户。**

**Q：Guava RateLimiter 和 Sentinel 怎么选？**
> 单机用 Guava（一行代码，零运维），分布式必须用 Sentinel（多实例需要协调总限流额度，Guava 在每个实例上独立限流会导致超标）。

**Q：令牌桶和漏桶的区别？**
> 漏桶：固定速率流出，瞬间洪峰全部排队（对下游绝对平滑）。令牌桶：允许短时间突发（桶里积累的令牌可以瞬间消费），更符合真实业务。支付系统用令牌桶，因为早上开门瞬间的购买洪峰是真实的正常业务。

**Q：限流了用户等着，用户体验很差怎么办？**
> 友好降级：不要返回 503，返回"系统繁忙，预计等待30秒，是否继续？"前端展示排队进度条。配合 MQ 可以做真正的排队（用户拿到排队号，系统异步处理完再通知）。

**Q：物理扩容（加机器）是不是最简单的解决方案？**
> 短期是的，但有边际效应：加 10 台机器，支付服务能力 10x，但 Oracle DB 还是同一个，DB 成了新瓶颈。纯加机器最终会卡在 DB，必须配合读写分离、分片、缓存才能真正线性扩展。

### 精确 QPS 计算（3实例 × 上述配置）

#### 第1层：支付发起 QPS

```
单次支付请求耗时分析（每步串行）：
  ┌───────────────────────────────────────────────────┐
  │ Redis 分布式锁 tryLock()              ≈   5 ms    │
  │ DB SELECT（幂等键查重）               ≈  10 ms    │
  │ DB INSERT（创建交易记录）             ≈  20 ms    │
  │ 渣打银行 API createPayment()          ≈ 2000 ms   │ ← 最大瓶颈！
  │ DB UPDATE（更新网关返回信息）         ≈  20 ms    │
  │ Redis lock.unlock()                   ≈   2 ms    │
  └───────────────────────────────────────────────────┘
  总耗时 ≈ 2057 ms/请求

单实例 Tomcat（200线程）：
  200 线程 ÷ 2.057 s = 97 QPS/实例

3 实例合计：
  97 × 3 = 291 QPS 支付发起

DB 验证（会不会 DB 先撑不住）：
  每笔支付 = 3 次 DB 操作（SELECT + INSERT + UPDATE）
  291 QPS × 3 = 873 次 DB 操作/秒
  60 个 Oracle 连接（3实例×20） × (1000ms/30ms) = 2000 次 DB 操作/秒
  873 < 2000 → DB 不是瓶颈 ✅

结论：支付发起瓶颈 = 渣打银行 API 响应速度（2s/次）
      3实例最多：~ 300 QPS 支付发起
```

#### 第2层：回调处理 QPS（@Async 异步）

```
单次回调处理耗时（callback-线程池执行）：
  ┌───────────────────────────────────────────────────┐
  │ Redis setIfAbsent（去重）             ≈   2 ms    │
  │ DB SELECT（查交易记录）               ≈  10 ms    │
  │ 终态检查（内存操作）                  ≈   0 ms    │
  │ DB UPDATE（乐观锁更新状态）           ≈  20 ms    │
  │ notifyOrderSystem()（目前只打log）    ≈   1 ms    │
  └───────────────────────────────────────────────────┘
  总耗时 ≈ 33 ms/回调

单实例 callbackExecutor（50线程）：
  50 线程 ÷ 0.033 s = 1515 回调/s/实例

3 实例合计（理论值）：
  1515 × 3 = 4545 回调/s

DB 验证（会不会 DB 先撑不住）：
  每次回调 = 2 次 DB 操作（SELECT + UPDATE）
  4545 回调/s × 2 = 9090 次 DB 操作/秒
  60 个 Oracle 连接 × (1000ms/30ms) ≈ 2000 次 DB 操作/秒 容量
  9090 >> 2000 → DB 是瓶颈！

实际回调 QPS（受 DB 限制）：
  2000 DB ops/s ÷ 2 ops/回调 = 1000 回调/s

结论：回调处理 DB 是瓶颈，实际上限 ≈ 1000 回调/s（3实例共享）
```

#### 第3层：查询状态 QPS

```
单次查询耗时：
  DB SELECT（按transactionId查）       ≈  10 ms

DB 可支持：60 连接 × (1000ms/10ms) = 6000 次/s
3实例 Tomcat 限制：600线程 × 100 QPS/线程 = 60000（Tomcat 不是瓶颈）

结论：状态查询上限 ≈ 6000 QPS（Oracle 决定）
      加 Redis 缓存可提升到 50000+ QPS
```

#### 最终 QPS 汇总（3实例 Nginx 部署）

```
┌──────────────────────────────────────────────────────────────────┐
│  操作类型         │ 单实例 QPS │ 3实例合计 │  真正瓶颈              │
├──────────────────────────────────────────────────────────────────┤
│  支付发起         │  ~100 QPS  │  ~300 QPS │  渣打银行 API（2s/次） │
│  回调处理（异步） │  ~330 QPS  │  ~1000 QPS│  Oracle DB（60连接）   │
│  查询状态         │  ~2000 QPS │  ~6000 QPS│  Oracle DB（60连接）   │
│  退款             │  ~50 QPS   │  ~150 QPS │  渣打银行 API + DB     │
├──────────────────────────────────────────────────────────────────┤
│  混合流量综合上限 │            │  ~300 QPS │  银行API（支付是主力） │
└──────────────────────────────────────────────────────────────────┘

没有限流的话：
  正常业务 ≤ 300 QPS → 系统稳定
  峰值突破 300 QPS  → HikariCP 连接池耗尽（30s timeout）→ 用户看到报错
  峰值突破 1000 QPS → Tomcat 队列满（accept-count=100）→ Connection Refused
```

### 分布式部署存在的问题（11个，含严重程度）

---

#### 🔴 问题1：对账 Job 在 3 个实例上同时跑（最严重）

**代码位置**：`PaymentReconciliationJob.java` - `@Scheduled` 两个方法  
**现象**：`@Scheduled` 注解在每个 Spring Boot 实例上都会执行，没有分布式协调

```
凌晨 3:00:00  实例A 启动 dailyFullReconciliation()
凌晨 3:00:00  实例B 启动 dailyFullReconciliation()  ← 同时！
凌晨 3:00:00  实例C 启动 dailyFullReconciliation()  ← 同时！

三个实例同时：
  ① 查询昨日所有交易（3次相同的全表扫描，DB负载3×）
  ② 对每笔 PENDING 交易调用渣打银行 API（3倍银行API请求，可能被限流）
  ③ INSERT INTO PAYMENT_RECONCILIATION → 插入3条重复对账记录！
  ④ 对同一笔交易 updateById() → 三个版本冲突，只有一个成功，其他两个静默失败

后果：
  - 对账报告数据×3，财务对账结果错误
  - 银行API调用量3倍，可能触发渣打限流
  - 凌晨3点DB负载3倍，影响业务
```

**修复方案**：
```java
// 方案A：ShedLock（推荐，最简单）
// pom.xml 加：net.javacrumbs.shedlock:shedlock-spring + shedlock-provider-redis-spring
@Scheduled(cron = "0 0 3 * * ?")
@SchedulerLock(name = "dailyFullReconciliation", 
               lockAtMostFor = "2h",   // 最多持锁2小时（防止宕机后永久锁住）
               lockAtLeastFor = "10m") // 至少持锁10分钟（防止抢锁重跑）
public void dailyFullReconciliation() { ... }

// 方案B：Redisson 分布式锁手动控制
@Scheduled(cron = "0 0 3 * * ?")
public void dailyFullReconciliation() {
    RLock lock = redissonClient.getLock("lock:reconciliation:daily");
    if (!lock.tryLock()) return; // 拿不到锁就退出，其他实例在跑
    try { /* 对账逻辑 */ } finally { lock.unlock(); }
}
```

#### 🔴 问题2：Snowflake ID 三个实例 workerId 相同 → ID 冲突

**代码位置**：全局使用 `IdUtil.getSnowflakeNextIdStr()`（Hutool 工具类）  
**现象**：Hutool Snowflake 默认 `workerId=0, datacenterId=0`，3个实例相同

```
Snowflake 64位结构：
  [41位时间戳][5位datacenterId][5位workerId][12位序列号]

3个实例都用 workerId=0, datacenterId=0：
  同一毫秒内：
    实例A 生成：time=1000, dc=0, worker=0, seq=1 → ID=XXX001
    实例B 生成：time=1000, dc=0, worker=0, seq=1 → ID=XXX001 ← 相同！
    实例C 生成：time=1000, dc=0, worker=0, seq=1 → ID=XXX001 ← 相同！

后果：
  同一毫秒高并发时，三个实例可能生成相同 ID
  → INSERT 失败（主键冲突）
  → 支付交易无法创建
  → 用户支付失败
```

**修复方案**：
```yaml
# application.yml 三个实例分别配置
# 实例A：
hutool.snowflake.worker-id: 1
hutool.snowflake.datacenter-id: 1
# 实例B：worker-id=2, 实例C：worker-id=3
```
```java
// 启动时从配置读取 workerId，初始化全局 Snowflake
@Configuration
public class SnowflakeConfig {
    @Value("${hutool.snowflake.worker-id:0}")
    private long workerId;
    @Value("${hutool.snowflake.datacenter-id:0}")
    private long datacenterId;

    @Bean
    public Snowflake snowflake() {
        return IdUtil.getSnowflake(workerId, datacenterId);
    }
}
// 所有地方改用注入的 snowflake.nextIdStr() 而不是 IdUtil.getSnowflakeNextIdStr()
```

#### 🟡 问题5：银行 API 无熔断器 → 银行慢时整个系统瘫痪

**代码位置**：所有 `gateway.createPayment()` 调用  
**配置**：`scb.timeout.seconds: 120`（2分钟！）

```
渣打银行 API 突然变慢（响应 60 秒）：

T+0s    用户A 发起支付 → 一个 Tomcat 线程占用 60 秒
T+1s    用户B 发起支付 → 又一个 Tomcat 线程占用 60 秒
...
T+120s  200 个 Tomcat 线程全部卡在等待银行 API
T+121s  新请求进来 → Tomcat 队列（100个）满了 → Connection Refused
T+122s  所有用户支付失败，网站无响应

不只影响支付！
  查询状态、退款的请求也进不来（同一个 Tomcat）
  整个服务完全不可用
```

**修复**（接入 Resilience4j 熔断器）：
```java
@CircuitBreaker(name = "scb-gateway", fallbackMethod = "createPaymentFallback")
@TimeLimiter(name = "scb-gateway")  // 超时控制
public GatewayPaymentResponse createPayment(PaymentInitRequest request, String txnId) {
    // 调用渣打银行 API
}

// 熔断时的降级响应
public GatewayPaymentResponse createPaymentFallback(
        PaymentInitRequest request, String txnId, Throwable ex) {
    log.error("渣打银行 API 不可用，熔断触发", ex);
    throw new GatewayException("支付网关暂时不可用，请稍后重试");
}
```

#### 🟡 问题6：Redis 单点配置 → Redis 宕机后分布式锁全失效

**代码位置**：`application.yml`

```yaml
redis:
  host: ${REDIS_HOST:localhost}
  port: ${REDIS_PORT:6379}
  # 没有 Sentinel / Cluster 配置
```

```
Redis 宕机后：
  redissonClient 连接失败 → redissonClient = null（或抛异常）
  
  PaymentServiceEnhanced.doInitiatePaymentWithDistributedLock()：
    if (redissonClient != null) → false
    → 走 doPaymentWithoutLock()（无锁模式）
    → Bug-2 问题重现（事务边界问题）
    → 高并发下可能重复创建交易

  PaymentCallbackServiceEnhanced：
    redisTemplate == null → 跳过 Redis 去重
    → 第一层防护失效，直接靠乐观锁兜底
```

**修复**：
```yaml
# Redis Sentinel 模式（高可用）
spring:
  redis:
    sentinel:
      master: mymaster
      nodes:
        - 192.168.100.21:26379
        - 192.168.100.22:26379
        - 192.168.100.23:26379
```

#### 🟡 问题7：对账时乐观锁 version 未设置 → 3实例并发对账冲突静默失败

**代码位置**：`PaymentReconciliationJob.reconcileTimeoutTransactions()`

```java
// 对账更新（问题代码）：
transaction.setPreviousStatus(transaction.getPaymentStatus());
transaction.setPaymentStatus(gatewayStatus.getStatus());
transactionMapper.updateById(transaction);
// ↑ transaction 对象里 version 是从 DB 读来的值
// 三个实例同时更新同一笔交易：
//   实例A：UPDATE ... WHERE id=? AND version=0 → 成功，version变1
//   实例B：UPDATE ... WHERE id=? AND version=0 → 0行（version已经是1了）
//   实例C：UPDATE ... WHERE id=? AND version=0 → 0行
// 实例B/C 的更新静默失败，没有任何日志！但对账报告里 matched+1 计数了
// → 对账报告数据不可信
```

**修复**：配合问题1（分布式锁）一起解决，Job 只跑一个实例就不存在并发问题。

#### 🟡 问题8：Nginx 没有健康检查 → 实例宕机后仍转发请求

```
假设实例B 宕机：
  Nginx upstream（没有 health_check）：
    → 继续轮询 A、B、C
    → 1/3 的请求打到宕机的 B → Connection Refused
    → 用户看到：约 33% 的支付请求直接失败

Nginx 默认行为：
  等到请求失败后才标记该实例为 down
  这中间的失败请求已经损失了
```

**修复**：
```nginx
upstream payment_backend {
    server 192.168.1.1:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.2:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.3:8080 max_fails=3 fail_timeout=30s;
    
    # 或者使用 nginx-plus 的主动健康检查
    # health_check interval=5s fails=3 passes=2;
}
```
或者换成 Spring Cloud + Nacos，服务下线自动摘除。

#### 🟡 问题9：日志分散 3 台机器 → 排查问题要登录 3 台服务器

```
用户投诉：支付失败，交易ID = TXN12345

运维流程（没有集中日志）：
  1. 登录实例A → grep TXN12345 logs/payment-system.log → 没找到
  2. 登录实例B → grep TXN12345 logs/payment-system.log → 找到了！
  3. 结合实例A的日志（Redis在A上）→ 再登录A
  总耗时：30分钟 ~ 2小时

有集中日志（ELK/Loki）：
  Kibana 搜索 TXN12345 → 所有实例的日志都在一起
  总耗时：30秒
```

**修复**：接入 ELK（Elasticsearch + Logstash + Kibana）或 Grafana Loki。

#### 🟢 问题11：日志级别生产环境太详细 → I/O压力 + 安全风险

```yaml
logging:
  level:
    sys.smc.payment: DEBUG  # ← 生产应该是 INFO
```

```
DEBUG 级别会打印：
  - 所有 SQL 语句（含用户ID、订单号、金额）
  - Redis 锁的获取/释放细节
  - 每个 HTTP 请求参数（可能含支付卡信息）

问题：
  1. I/O压力：日志量是 INFO 的 5~10 倍，高QPS时磁盘成为瓶颈
  2. 安全风险：日志文件包含敏感信息，需要严格访问控制
  3. 合规风险：PCI-DSS 要求卡号等数据不能明文出现在日志中
```

### 问题严重程度总览

```
🔴 上线即出问题（必须在上线前修复）：
  ① 对账Job 3实例并发执行 → 数据重复，财务报告错误
  ② Snowflake ID 冲突 → INSERT 主键冲突，支付失败
  ③ DB/Redis localhost → 实例各连自己的DB，数据孤岛
  ④ 多渠道对账缺失 → 支付宝/建行 PENDING 永不结束

🟡 上线后遇到压力/故障时暴露（尽快修复）：
  ⑤ 银行API无熔断 → 银行慢时整个服务瘫痪
  ⑥ Redis单点 → Redis宕机后幂等防护下降
  ⑦ 对账乐观锁 → 对账报告数据不可信（配合①修复）
  ⑧ Nginx无健康检查 → 实例宕机后33%请求失败
  ⑨ 日志分散 → 排查问题效率极低
  ⑩ 无限流 → 突发流量打穿连接池

🟢 优化项（有时间就做）：
  ⑪ 日志级别 DEBUG → 改为 INFO，减少I/O，提升安全
```
