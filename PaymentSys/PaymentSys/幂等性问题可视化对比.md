# 幂等性问题可视化对比

## 📖 技术术语通俗解释（必读）

### 1. 实例（Instance）
**是什么**：一台运行着支付系统的服务器程序。

**生活例子**：
```
想象一家银行有3个营业窗口：
- 窗口A（实例A）- 一台服务器
- 窗口B（实例B）- 另一台服务器  
- 窗口C（实例C）- 第三台服务器

客户来办业务时，会被分配到其中一个窗口。

单机 = 只有1个窗口（1台服务器）
分布式 = 有多个窗口（多台服务器）
```

**为什么要多个实例**：
- 1个窗口处理不了太多客户（单机性能有限）
- 1个窗口坏了，整个银行就关门了（单点故障）
- 多个窗口可以分担压力（负载均衡）

---

### 2. 并发窗口（Concurrency Window）
**是什么**：从"检查是否重复"到"创建交易"之间的时间间隔。

**生活例子**：
```
你去电影院买票：

步骤1: 你问："A1座位有人吗？"
步骤2: 售票员查电脑："没人"
步骤3: 你说："好，我买A1"
步骤4: 售票员登记："A1卖给你了"

并发窗口 = 步骤2到步骤4之间的时间

如果在这个时间内，另一个人也问A1并且也想买：
- 窗口短（0.1秒）→ 几乎不可能同时
- 窗口长（5秒）→ 很容易同时买到同一个座位 ❌
```

**支付系统的并发窗口**：
```
单机系统：
步骤1: 查询订单是否已支付 (10ms)
步骤2: 没有，创建支付交易 (10ms)
并发窗口 = 20ms

分布式系统：
步骤1: 通过网络查询数据库 (50ms)
步骤2: 应用处理 (10ms)
步骤3: 通过网络创建交易 (50ms)
并发窗口 = 110ms ← 容易冲突！
```

### 3. 冲突概率（Collision Probability）
**是什么**：两个人同时做同一件事导致出问题的可能性。

**生活例子**：
```
抢购演唱会门票：

场景A（冲突概率低）：
- 只有10个人抢
- 有100张票
- 系统响应快（0.1秒）
→ 几乎不会有人抢到同一张票

场景B（冲突概率高）：
- 有1000人抢
- 只有10张票
- 系统响应慢（5秒）
→ 很多人会同时点击同一张票 ❌
```

**支付系统的冲突**：
```
冲突 = 同一个订单被创建多次

低冲突场景（0.1%）：
- 并发窗口短（25ms）
- 请求量少（每秒100个）
- 用户不会疯狂点击

高冲突场景（15%）：
- 并发窗口长（125ms）
- 请求量大（每秒1000个）
- 网络卡顿，用户重复点击
→ 同一订单可能创建多次 ❌
```

### 4. TPS（Transactions Per Second）
**是什么**：每秒处理的交易数量。

**生活例子**：
```
餐厅服务员：

TPS = 100 
= 每秒服务100个客人
= 每分钟6000个客人
→ 需要很多服务员（多个实例）

TPS = 10
= 每秒服务10个客人
= 每分钟600个客人
→ 1-2个服务员就够了（单机）
```

**常见场景**：
| TPS | 规模 | 场景 |
|-----|------|------|
| < 100 | 小型 | 企业内部系统 |
| 100-500 | 中小型 | 小型电商 |
| 500-1000 | 中型 | 中型电商 |
| > 1000 | 大型 | 淘宝、京东 |
| > 10000 | 超大型 | 双十一抢购 |

### 5. 数据库锁（Database Lock）
**是什么**：防止两个人同时修改同一条数据。

**生活例子**：
```
银行柜台办业务：

你要取钱：
1. 柜员打开你的账户（获取锁）
2. 查看余额：1000元
3. 扣除500元
4. 余额变成500元
5. 关闭账户（释放锁）

如果没有锁：
你在柜台A取500元（同时进行）
你在柜台B取500元（同时进行）
→ 两个柜台都看到余额1000元
→ 都扣除500元
→ 你取出了1000元，但余额只减了500元 ❌

有锁的情况：
柜台A：锁定账户 → 取500元 → 释放锁
柜台B：等待锁 → 取500元 → 释放锁
→ 正确
```

**支付系统的锁问题**：
```
100个请求同时支付同一订单：

没有锁：
→ 创建100笔交易 ❌

有唯一索引（数据库锁）：
→ 第1个：成功创建 ✅
→ 第2-100个：排队等待，检测重复，回滚 ⏳
→ 数据库压力大 ⚠️

有Redis锁：
→ 第1个：获得锁，创建交易 ✅
→ 第2-100个：Redis直接拦截，不走数据库 ⚡
→ 数据库压力小 ✅
```

### 6. 乐观锁 vs 悲观锁

#### 悲观锁（Pessimistic Lock）
**态度**：我先锁住，你们都别动！

**生活例子**：
```
去餐厅吃饭：
1. 你坐下，服务员立刻锁定这张桌子
2. 其他客人不能靠近这张桌子
3. 你吃完，服务员解锁桌子
→ 安全，但效率低
```

#### 乐观锁（Optimistic Lock）
**态度**：先不锁，如果冲突了再说。

**生活例子**：
```
去餐厅吃饭（自助式）：
1. 你看到桌子是空的（版本=1）
2. 你去拿食物
3. 回来准备坐下
4. 检查：桌子还是版本1吗？
   - 是 → 坐下 ✅
   - 不是（有人占了，版本=2）→ 换桌子 ❌
→ 效率高，但可能冲突
```

**支付系统使用乐观锁**：
```sql
-- 交易记录有个VERSION字段

第一次更新：
UPDATE PAYMENT_TRANSACTION 
SET STATUS = 'SUCCESS',
    VERSION = 2        -- 版本+1
WHERE ID = 123 
  AND VERSION = 1;     -- 必须是版本1才能更新

第二次更新（冲突）：
UPDATE PAYMENT_TRANSACTION 
SET STATUS = 'SUCCESS',
    VERSION = 2
WHERE ID = 123 
  AND VERSION = 1;     -- 版本已经是2了，不匹配
→ 更新失败，返回0行 ❌
```

### 7. 终态（Terminal State）
**是什么**：最终状态，一旦到达就不能再改变。

**生活例子**：
```
快递状态：
派送中 → 派送中 → 派送中  ← 可以反复
派送中 → 已签收             ← 终态
已签收 → 派送中？           ❌ 不能回退

支付状态：
待支付 → 支付中 → 支付中   ← 可以反复
支付中 → 支付成功           ← 终态
支付成功 → 支付中？         ❌ 不能回退
```

**为什么需要终态保护**：
```
没有终态保护：
订单状态：支付成功
回调再次到达 → 可能改成"支付中"？ ❌
→ 订单状态混乱

有终态保护：
订单状态：支付成功 ← 终态
回调再次到达 → 检查到是终态 → 直接返回 ✅
→ 状态稳定
```

### 8. 幂等性（Idempotency）
**是什么**：重复执行多次，结果都一样。

**生活例子**：
```
开电灯：

没有幂等性：
按1次开关 → 灯亮
按2次开关 → 灯灭
按3次开关 → 灯亮
→ 结果不同

有幂等性：
按1次"开灯"按钮 → 灯亮
按2次"开灯"按钮 → 灯还是亮
按100次"开灯"按钮 → 灯还是亮
→ 结果相同 ✅
```

**支付系统的幂等性**：
```
没有幂等性：
点1次"支付" → 扣款100元
点2次"支付" → 又扣款100元
→ 共扣200元 ❌

有幂等性：
点1次"支付"（订单CSP001）→ 扣款100元
点2次"支付"（订单CSP001）→ 检测到已支付，不扣款
→ 共扣100元 ✅
```

---

### 9. 幂等键（Idempotency Key）
**是什么**：用来识别"是不是同一件事"的唯一标识。

**生活例子**：
```
身份证号 = 人的幂等键
- 同一个人，身份证号永远不变
- 去任何地方办事，都用这个号
- 系统一看身份证号，就知道是你

订单号 = 支付的幂等键
- 同一个订单，订单号永远不变
- 无论点多少次支付，都是同一个订单号
- 系统一看订单号，就知道是重复的
```

**问题：时间戳作为幂等键**：
```
❌ 错误方式：
幂等键 = 订单号 + 时间戳
第1次：CSP001_2026-04-02_10:00:00
第2次：CSP001_2026-04-02_10:00:05  ← 不同！
→ 系统认为是2个不同的支付 ❌

✅ 正确方式：
幂等键 = 订单号
第1次：CSP001
第2次：CSP001  ← 相同！
→ 系统认为是同一个支付 ✅
```

### 10. 回调（Callback）
**是什么**：银行处理完支付后，主动通知你结果。

**生活例子**：
```
你在淘宝下单：

方式A（轮询）：
你：付款成功了吗？（每5秒问一次）
银行：还在处理...
你：好了吗？
银行：还在处理...
你：好了吗？
银行：成功了！
→ 你需要不停地问 ❌

方式B（回调）：
你：我付款了，处理完告诉我一声
银行：好的，去处理...
（你可以干别的事）
银行：处理完了，主动通知你！
→ 银行主动告诉你 ✅
```

**支付系统的回调**：
```
1. 用户点击"支付" → 跳转到银行页面
2. 用户在银行输入密码，支付成功
3. 银行记录：这笔钱到账了
4. 银行通过HTTP POST发送回调：
   POST http://你的系统/api/callback
   {
     "orderId": "CSP001",
     "status": "SUCCESS"
   }
5. 你的系统收到回调 → 更新订单状态
```

### 11. 唯一索引（Unique Index）
**是什么**：数据库保证某个字段的值不能重复。

**生活例子**：
```
学校学号：
- 每个学生有唯一的学号
- 不能有两个学生学号相同
- 新生入学，系统自动检查学号是否重复

唯一索引 = 数据库的"学号检查器"
```

**支付系统的唯一索引**：
```sql
CREATE UNIQUE INDEX UK_IDEMPOTENCY 
ON PAYMENT_TRANSACTION(IDEMPOTENCY_KEY);

作用：
INSERT INTO PAYMENT_TRANSACTION 
VALUES ('CSP001_176_T1', ...);  -- ✅ 成功

INSERT INTO PAYMENT_TRANSACTION 
VALUES ('CSP001_176_T1', ...);  -- ❌ 错误：已存在
```

### 12. 雪崩效应（Cascade Failure）
**是什么**：一个小问题引发连锁反应，导致整个系统崩溃。

**生活例子**：
```
高速公路堵车：

1. 一辆车急刹车（小问题）
      ↓
2. 后面的车也急刹车
      ↓
3. 更多车急刹车
      ↓
4. 整条高速堵死（系统崩溃）
```

**支付系统的雪崩**：
```
1. 数据库有点慢（响应200ms）
      ↓
2. 请求开始堆积
      ↓
3. 连接池占满
      ↓
4. 新请求无法获取连接
      ↓
5. 请求超时，客户端重试
      ↓
6. 更多请求涌入
      ↓
7. 数据库完全卡死
      ↓
8. 整个系统崩溃 ❌
```

|-----|---------|------|
| **实例** | 一台服务器 | 银行的一个窗口 |

| **悲观锁** | 先锁住再操作 | 包厢需要预定 |
| **终态** | 最终状态，不可改变 | 快递签收后不能退回 |

## 🎨 图解核心概念

### 什么是"并发窗口"

**图1：单机场景（窗口短=安全）**
```
时间线 (每格=10ms)
0ms   10ms  20ms  30ms  40ms  50ms
│─────┼─────┼─────┼─────┼─────┼─────│

请求1: 
      查询 ──→ 创建
      └─┬─┘
        └─ 并发窗口(20ms)

请求2:
                    查询 ──→ 创建
                    
👉 请求2到达时，请求1已经创建完成
    → 请求2能查到已存在的交易 ✅
```

**图2：分布式场景（窗口长=危险）**
```
时间线 (每格=25ms)
0ms   25ms  50ms  75ms  100ms 125ms 150ms
│─────┼─────┼─────┼─────┼─────┼─────┼─────│

实例A请求: 
      查询 ────────────────────────────→ 创建
      └──────────┬─────────────────────┘
                 └─ 并发窗口(125ms)

实例B请求:
            查询 ────────────────────────────→ 创建
            
👉 两个请求都在对方创建之前查询
    → 两个都认为"不存在" ❌
    → 两个都创建成功 ❌
```

### 什么是"冲突"

**生活场景：电影院抢座位**

```
窗口短（0.1秒）：
════════════════════════════════════════
你：A1有人吗？
售票员：查...没人
你：买！
售票员：卖给你了 ✅

（0.1秒后）
朋友：A1有人吗？
售票员：查...有了，刚卖出去
朋友：那算了 ✅

冲突概率：0% ← 时间太短，不会同时
════════════════════════════════════════

窗口长（10秒）：
════════════════════════════════════════
你：A1有人吗？
售票员：查中...（系统慢）
（5秒后）
朋友：A1有人吗？
售票员：查中...（还在查）
（7秒后）
售票员对你说：没人
售票员对朋友说：没人
你：买！
朋友：买！
结果：❌ 两个人都买了A1座位！

冲突概率：50% ← 时间太长，很容易同时
════════════════════════════════════════
```

**支付系统的冲突**：
```
用户点击"支付"按钮

窗口短（单机25ms）：
点击1 → [查询-创建] → 完成  (25ms)
点击2 →                    [查询-创建] → 发现已存在 ✅

窗口长（分布式125ms）：
点击1 → [查询---网络---创建] → 完成  (125ms)
点击2 →   [查询---网络---创建] → 也成功 ❌
        ↑ 两个都在对方完成前查询，都认为"不存在"
```

### 什么是"实例"和"分布式"

**图解：单机 vs 分布式**

```
单机部署（1个窗口）：
════════════════════════════════════════════════
                    用户浏览器
                         │
                         ↓
              ┌──────────────────┐
              │   应用服务器      │ ← 只有1台
              │   (实例A)        │
              └──────────────────┘
                         │
                         ↓
              ┌──────────────────┐
              │   数据库          │
              └──────────────────┘

所有请求都到同一台服务器
时间戳都在同一台机器生成 → 相对准确
════════════════════════════════════════════════


分布式部署（3个窗口）：
════════════════════════════════════════════════
                    用户浏览器
                         │
                         ↓
              ┌──────────────────┐
              │   负载均衡器      │ ← 自动分配请求
              └──────────────────┘
                         │
        ┌────────────────┼────────────────┐
        ↓                ↓                ↓
   ┌────────┐      ┌────────┐      ┌────────┐
   │ 实例A   │      │ 实例B   │      │ 实例C   │ ← 3台服务器
   │ 上海    │      │ 北京    │      │ 深圳    │
   └────────┘      └────────┘      └────────┘
        │                │                │
        └────────────────┼────────────────┘
                         ↓
              ┌──────────────────┐
              │   数据库          │
              └──────────────────┘

请求可能到不同服务器
时间戳在不同机器生成 → 可能不一样 ⚠️
════════════════════════════════════════════════
```

### 为什么分布式会有"时间戳不同"问题

**图解：时钟同步问题**

```
3台服务器，虽然都显示10:00:00，但实际有偏差：

服务器A（上海）：  10:00:00.000
服务器B（北京）：  10:00:00.335  ← 偏差335毫秒
服务器C（深圳）：  10:00:00.678  ← 偏差678毫秒

用户第1次点击 → 实例A → 幂等键：CSP001_176_1712000000000
用户第2次点击 → 实例B → 幂等键：CSP001_176_1712000000335
                                            ↑ 不同！

结果：系统认为是2个不同的支付请求 ❌
```

### 什么是"连接池"

**生活例子：餐厅的桌子**

```
餐厅有10张桌子（连接池大小=10）

正常情况（5个客人）：
┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│ 客 │ │ 客 │ │ 客 │ │ 客 │ │ 客 │  5张桌子被占用
└───┘ └───┘ └───┘ └───┘ └───┘
┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│ 空 │ │ 空 │ │ 空 │ │ 空 │ │ 空 │  5张桌子空闲 ✅
└───┘ └───┘ └───┘ └───┘ └───┘

高峰期（15个客人想进来）：
┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│ 客 │ │ 客 │ │ 客 │ │ 客 │ │ 客 │  10张桌子全满
└───┘ └───┘ └───┘ └───┘ └───┘
┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│ 客 │ │ 客 │ │ 客 │ │ 客 │ │ 客 │
└───┘ └───┘ └───┘ └───┘ └───┘

还有5个客人在门口等待 ⏳
```

### 数据指标含义

#### P99响应时间
**是什么**：99%的请求响应时间。

**通俗理解**：
```
100个客人来餐厅吃饭：
- 99个人等了5分钟就上菜了
- 1个人等了20分钟（厨房出问题）

P99响应时间 = 5分钟
→ 99%的客人体验好 ✅
```

**支付系统**：
```
P99 = 150ms → 99%的支付在150ms内完成 ✅
P99 = 5000ms → 99%的支付要等5秒 ❌ 用户会觉得卡
```

#### 错误率
**是什么**：失败请求占总请求的比例。

```
餐厅例子：
100个客人点餐
- 95个成功
- 5个失败（食材没了）

错误率 = 5/100 = 5%

支付系统：
错误率 0.1% → 1000笔交易，1笔失败 ✅ 可接受
错误率 15% → 1000笔交易，150笔失败 ❌ 灾难
```

#### 数据库CPU
**是什么**：数据库服务器的处理器使用率。

```
CPU = 30% → 轻松，处理速度快 ✅
CPU = 70% → 有点忙，开始变慢 🟡
CPU = 95% → 非常忙，快撑不住 ⚠️
CPU = 100% → 崩溃边缘 ❌
```

## 🧮 冲突概率计算（简化）

### 公式
```
冲突概率 ≈ 并发窗口 × 每秒请求数 × 重复提交率

单机：
= 25ms × 100请求/秒 × 10%重复率
= 0.025 × 100 × 0.1
= 0.25%  ✅ 很低

分布式：
= 125ms × 1000请求/秒 × 30%重复率
= 0.125 × 1000 × 0.3
= 37.5%  ❌ 非常高！
```

### 实际含义

```
0.1% 冲突率 = 1000笔交易，1笔可能重复 ✅
5% 冲突率 = 1000笔交易，50笔重复 ⚠️
15% 冲突率 = 1000笔交易，150笔重复 ❌
```

**时间线**：
```
10:00:00.000  小明点击"支付"
                 ↓
              负载均衡器分配 → 实例A（北京）
                 ↓
              实例A的时钟：10:00:00.000
              生成幂等键：CSP001_176_1712000000000
                 ↓
              通过网络查询数据库...（50ms网络延迟）
                 ↓

10:00:00.050  数据库响应：没有找到
              实例A准备创建交易...
              通过网络发送INSERT命令...（50ms）
                 ↓

10:00:00.070  小明等了70ms，页面没反应
              网络卡顿，他以为没成功
              又点了一次 🖱️🖱️
                 ↓
              负载均衡器分配 → 实例B（上海）⚠️
                 ↓
              实例B的时钟：10:00:00.335（偏差335ms）
              生成幂等键：CSP001_176_1712000000335 ⚠️
                                            ↑ 不同！
                 ↓
              通过网络查询数据库...（50ms）
                 ↓

10:00:00.100  实例A的INSERT到达数据库
              创建交易TXN001 ✅
                 ↓

10:00:00.120  数据库响应实例B：没有找到
              （因为查询发生在100ms之前）
              实例B准备创建交易...
                 ↓

10:00:00.170  实例B的INSERT到达数据库
              创建交易TXN002 ❌
              （幂等键不同，数据库认为是不同的交易）
                 ↓

结果：❌ 创建了2笔交易
     TXN001：订单CSP001，金额176元
     TXN002：订单CSP001，金额176元
     小明被扣款352元！
```

**问题分析**：
1. **时间戳不同** → 幂等键不同 → 数据库认为是不同的支付
2. **并发窗口长** → 两个请求都在对方完成前查询 → 都认为"不存在"
3. **负载均衡** → 请求分配到不同实例 → 时钟不同步

### 第三幕：增强版拯救世界

```
同样的场景，使用增强版：

10:00:00.000  前端生成幂等Token（UUID）：
              abc-123-def-456
                 ↓
              小明点击"支付"
              请求带着Token发送
                 ↓
              负载均衡器 → 实例A
                 ↓
              实例A：
              1. 尝试获取Redis锁：
                 LOCK "payment:lock:abc-123-def-456"
              2. 成功获得锁 🔒
              3. 查询数据库：没有
              4. 创建交易TXN001
              5. 释放锁 🔓
                 ↓

10:00:00.070  小明又点一次
              同样的Token：abc-123-def-456
                 ↓
              负载均衡器 → 实例B
                 ↓
              实例B：
              1. 尝试获取Redis锁：
                 LOCK "payment:lock:abc-123-def-456"
              2. ❌ 锁已被占用（实例A持有）
              3. 等待锁释放...（等了0.001秒）
              4. 锁释放了，但发现交易已创建
              5. 查询到TXN001
              6. 直接返回TXN001 ✅
                 ↓

结果：✅ 只有1笔交易TXN001
     ✅ 小明只被扣176元
     ✅ Redis在毫秒级就拦截了重复
     ✅ 数据库压力小
```

## 💡 一分钟速记卡

### 核心概念

| 概念 | 一句话理解 |
|-----|-----------|
| **实例** | 一台服务器，就像银行的一个窗口 |
| **分布式** | 多台服务器协作，就像银行有多个窗口 |
| **并发窗口** | 检查到创建之间的时间，越短越安全 |
| **冲突概率** | 重复操作的可能性，受窗口大小影响 |
| **TPS** | 每秒处理多少笔，餐厅每秒服务多少客人 |
| **幂等性** | 重复做结果一样，开灯按钮按多次灯还是亮 |
| **幂等键** | 识别是否重复的身份证号 |
| **终态** | 不能改变的最终状态，快递签收了就不能撤销 |
| **回调** | 银行主动通知结果，不用你一直问 |
| **乐观锁** | 先不锁，冲突了再说，自助餐选座位 |
| **悲观锁** | 先锁住再操作，包厢要预定 |
| **雪崩** | 小问题引发崩溃，高速公路堵车 |

### 核心关系

```
单机 → 并发窗口短 → 冲突概率低 → 标准版够用
  ↓
分布式 → 并发窗口长 → 冲突概率高 → 需要增强版
  ↓
高并发 → 数据库压力大 → 可能雪崩 → 必须增强版
```

### 记住这一张图

```
                    你的支付系统

单机部署          分布式部署        高并发场景
(1台服务器)      (3台服务器)       (TPS>1000)
     ↓               ↓                ↓
  窗口短           窗口长            锁竞争
     ↓               ↓                ↓
  冲突少           冲突多            数据库爆
     ↓               ↓                ↓
  标准版✅         增强版🟡          增强版🔴
```
