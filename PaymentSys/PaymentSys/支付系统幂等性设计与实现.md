## 什么是幂等性

### 定义
**幂等性（Idempotency）** 是指：同一个操作执行一次和执行多次的结果完全相同。

### 数学角度
```
f(f(x)) = f(x)
```
无论调用多少次，结果都一样。

### 支付场景举例

| 操作 | 是否幂等 | 说明 |
|-----|---------|------|
| 查询订单状态 | ✅ 是 | 查询100次结果都一样 |
| 扣款100元 | ❌ 否 | 扣两次就是200元 |
| **幂等扣款**：订单号ABC扣款100元 | ✅ 是 | 同一订单号只扣一次 |

## 为什么幂等性至关重要

### 🔴 真实灾难场景

#### 案例1：重复扣款灾难（某电商平台2019年）
```
客户点击"支付"按钮
    → 网络卡顿，客户以为没反应
    → 客户再点一次
    → 系统扣款两次
    → 客户投诉 + 退款成本 + 品牌损失
```
**损失**：单日退款10万笔，赔付500万元

#### 案例2：回调重复处理（本系统TIMEOUT场景）
```
银行发送回调：支付成功
    → 系统处理中，数据库锁超时
    → 银行没收到响应，5分钟后重试
    → 系统又处理一次
    → 订单被重复确认，库存重复扣减
```

### 🎯 支付系统的核心地位

幂等性在支付系统中的重要性：

```
安全性 > 幂等性 ≈ 一致性 > 性能
         ↑
    防止重复扣款的最后防线
```

### 需要幂等性的场景

| 场景 | 风险 | 幂等性保护 |
|-----|------|-----------|
| **1. 用户重复提交** | 用户点击多次"支付"按钮 | 前端防抖 + 后端幂等键 |
| **2. 网络重试** | HTTP超时自动重试 | 幂等键去重 |
| **3. 银行回调重试** | 银行发送多次回调 | 回调幂等性检查 |
| **4. 对账修正** | 对账任务重复执行 | 状态终态检查 |
| **5. MQ消息重复消费** | 消息队列至少一次投递 | 消费幂等性 |

### 实现1️⃣：支付发起幂等

#### 代码实现
```java
// PaymentService.java
@Transactional(rollbackFor = Exception.class)
public PaymentInitResponse initiatePayment(PaymentInitRequest request) {
    // 1. 生成幂等键
    String idempotencyKey = generateIdempotencyKey(request);
    
    // 2. 检查交易是否已存在（数据库唯一索引保证）
    PaymentTransaction existing = transactionMapper.selectByIdempotencyKey(idempotencyKey);
    if (existing != null) {
        log.warn("检测到重复支付请求，幂等键：{}", idempotencyKey);
        return buildResponseFromTransaction(existing); // 返回原交易结果
    }
    
    // 3. 创建新交易...
}

private String generateIdempotencyKey(PaymentInitRequest request) {
    // 订单号 + 金额 + 时间戳 = 唯一标识
    return request.getOrderReference() + "_" +
           request.getAmount().toString() + "_" +
           System.currentTimeMillis();
}
```

#### 数据库保障
```sql
-- PAYMENT_TRANSACTION 表
CREATE TABLE PAYMENT_TRANSACTION (
    -- ...
    IDEMPOTENCY_KEY VARCHAR2(200) NOT NULL UNIQUE,  -- 唯一约束
    -- ...
);

-- 唯一索引防止并发插入
CREATE UNIQUE INDEX UK_PT_IDEMPOTENCY ON PAYMENT_TRANSACTION(IDEMPOTENCY_KEY);
```

#### 工作原理

```
请求1: 订单CSP001 + 金额176 + 时间1
    ↓
生成幂等键: CSP001_176_1709876543210
    ↓
插入数据库 ✅ 成功
    ↓
返回交易ID: TXN001


请求2: 订单CSP001 + 金额176 + 时间2（用户重复点击）
    ↓
生成幂等键: CSP001_176_1709876543999
    ↓
查询已存在交易（订单号匹配）
    ↓
返回已存在的交易ID: TXN001 ✅ 防止重复扣款
```

### 实现2️⃣：回调处理幂等

#### 代码实现
```java
// PaymentCallbackService.java
@Async("callbackExecutor")
@Transactional(rollbackFor = Exception.class, 
               isolation = Isolation.READ_COMMITTED,
               propagation = Propagation.REQUIRES_NEW)
public void processCallback(String rawBody, Map<String, String> headers, 
                             String clientIp, PaymentChannel channel) {
    try {
        // 1. 解析回调数据
        PaymentCallbackData callbackData = gateway.parseCallbackData(rawBody);
        
        // 2. 查找交易
        PaymentTransaction transaction = transactionMapper.selectByGatewayTransactionId(
            callbackData.getGatewayTransactionId()
        );
        
        // 3. ⭐ 幂等性检查：检查是否已处于终态
        if (isTerminalStatus(transaction.getPaymentStatus())) {
            log.warn("[{}] 交易已处于终态，交易ID：{}，状态：{}",
                channel.getName(), transaction.getTransactionId(), 
                transaction.getPaymentStatus());
            callbackLog.setProcessingStatus("DUPLICATE"); // 标记为重复
            return; // 直接返回，不再处理
        }
        
        // 4. ⭐ 使用乐观锁更新（防止并发）
        boolean updated = updateTransactionStatus(transaction, callbackData);
        
        if (!updated) {
            // 版本冲突，说明有其他线程已更新
            throw new OptimisticLockException("版本冲突");
        }
        
        // 5. 触发下游动作
        if (PaymentStatus.SUCCESS.name().equals(callbackData.getPaymentStatus())) {
            notifyOrderSystem(transaction);
        }
        
    } finally {
        // 6. ⭐ 总是记录回调日志（审计追踪）
        callbackLogMapper.insert(callbackLog);
    }
}

private boolean isTerminalStatus(String status) {
    // 终态：SUCCESS, FAILED, REFUNDED
    // 一旦进入终态，永不再变
    return PaymentStatus.SUCCESS.name().equals(status) ||
           PaymentStatus.FAILED.name().equals(status) ||
           PaymentStatus.REFUNDED.name().equals(status);
}

private boolean updateTransactionStatus(PaymentTransaction transaction, 
                                         PaymentCallbackData callbackData) {
    PaymentTransaction update = new PaymentTransaction();
    update.setId(transaction.getId());
    update.setVersion(transaction.getVersion()); // ⭐ 乐观锁关键
    update.setPaymentStatus(callbackData.getPaymentStatus());
    // ...
    
    int rows = transactionMapper.updateById(update); // MyBatis-Plus自动检查version
    return rows > 0; // 更新失败返回false
}
```

#### 乐观锁原理

```sql
-- MyBatis-Plus生成的SQL（简化版）
UPDATE PAYMENT_TRANSACTION 
SET PAYMENT_STATUS = 'SUCCESS',
    VERSION = VERSION + 1  -- 自动递增版本号
WHERE ID = 123 
  AND VERSION = 5;         -- ⭐ 只有版本号匹配才更新

-- 如果VERSION已被其他事务改为6，则更新失败（返回0行）
```

### 实现3️⃣：回调日志去重

```java
// 所有回调都记录到 PAYMENT_CALLBACK_LOG
CREATE TABLE PAYMENT_CALLBACK_LOG (
    ID NUMBER(19) PRIMARY KEY,
    TRANSACTION_ID VARCHAR2(100) NOT NULL,
    CALLBACK_TIME DATE NOT NULL,
    PROCESSING_STATUS VARCHAR2(20),  -- SUCCESS, FAILED, DUPLICATE
    -- ...
);

-- 查询某交易收到了几次回调
SELECT COUNT(*) AS callback_count,
       SUM(CASE WHEN PROCESSING_STATUS = 'SUCCESS' THEN 1 ELSE 0 END) AS success_count,
       SUM(CASE WHEN PROCESSING_STATUS = 'DUPLICATE' THEN 1 ELSE 0 END) AS duplicate_count
FROM PAYMENT_CALLBACK_LOG
WHERE TRANSACTION_ID = 'TXN20260203200150001';
```

## 实际案例分析

### 案例1：网络抖动导致的重复支付请求

#### 问题
```
14:30:00.123 - 客户点击"支付"
14:30:00.125 - 请求1发送
14:30:05.000 - 网络超时，前端未收到响应
14:30:05.001 - 客户再次点击
14:30:05.005 - 请求2发送
14:30:06.000 - 请求1到达服务器
14:30:06.100 - 请求2到达服务器
```

#### 系统行为（有幂等性）
```java
// 请求1
idempotencyKey = "CSP001_176_1709876400123"
查询数据库 → 未找到 → 创建交易TXN001 → 返回

// 请求2（100ms后）
idempotencyKey = "CSP001_176_1709876405001"
查询数据库 → 找到订单CSP001的交易TXN001 → 返回已存在交易
```

**结果**：✅ 只创建1笔交易，客户只扣款1次

#### 如果没有幂等性
```
请求1 → 创建交易TXN001 → 扣款176元
请求2 → 创建交易TXN002 → 再扣款176元
客户损失：352元（本应176元）
```

### 案例2：银行回调风暴

#### 问题
```
20:01:30 - 支付成功
20:01:31 - 银行发送回调1
20:01:32 - 系统处理中，数据库锁超时
20:01:35 - 银行未收到响应（超时3秒）
20:06:31 - 银行重试，发送回调2
20:06:32 - 系统处理成功
20:11:31 - 银行再次重试，发送回调3
20:16:31 - 银行再次重试，发送回调4
```

#### 系统行为（有幂等性）
```
回调1: PENDING → 更新失败 → 记录日志(FAILED)
回调2: PENDING → SUCCESS ✅ → 通知订单 → 记录日志(SUCCESS)
回调3: SUCCESS → 检测终态 → 直接返回 → 记录日志(DUPLICATE)
回调4: SUCCESS → 检测终态 → 直接返回 → 记录日志(DUPLICATE)
```

**结果**：✅ 订单只确认1次，库存只扣减1次

#### 如果没有幂等性
```
回调1: 失败
回调2: 订单确认，库存-1
回调3: 订单再次确认，库存-1 ❌
回调4: 订单再次确认，库存-1 ❌
库存超卖：原库存10，应扣1，实际扣3，剩余7（错误）
```

#### 问题
```

对账任务意外崩溃，3:05重启
再次查询TIMEOUT交易 → 还是100笔
再次处理...
```

#### 系统行为（有状态机保护）
```java
// 第一次对账
for (transaction : timeoutList) {
    gatewayStatus = bank.query(transaction.gatewayId);
    if (gatewayStatus == SUCCESS && transaction.status == TIMEOUT) {
        transaction.status = SUCCESS;  // TIMEOUT → SUCCESS
        transaction.reconciliationStatus = "MISMATCH";
        update(transaction);  // ✅ 更新成功
        notifyOrder(transaction);  // ✅ 通知1次
    }
}

// 第二次对账（重启后）
for (transaction : timeoutList) {
    // 这10笔已经是SUCCESS状态了
    gatewayStatus = bank.query(transaction.gatewayId);
    if (gatewayStatus == SUCCESS && transaction.status == SUCCESS) {
        // ✅ 状态已匹配，跳过
        transaction.reconciliationStatus = "MATCHED";
        continue;
    }
}
```

**结果**：✅ 订单只通知1次，不会重复处理

### 当前系统的不足

#### ❌ 问题1：幂等键生成依赖时间戳
```java

    return request.getOrderReference() + "_" +
           request.getAmount().toString() + "_" +
           System.currentTimeMillis();  // ⚠️ 时间戳每次不同
```

**问题**：同一订单的两次请求会生成不同的幂等键

**场景**：

```
              → 检查发现订单CSP001已有交易 → ✅ 返回TXN001（依赖订单号查询）
```

**风险**：如果不同金额（如修改了订单），会创建新交易

#### ❌ 问题2：缺少分布式锁

```java
private String generateIdempotencyKey(PaymentInitRequest request) {
    // 方案1：使用订单号 + 支付类型（适合一订单多支付场景）
    return request.getOrderReference() + "_" + 
           request.getPaymentMethod();
    
    // 方案2：前端生成UUID传入（最推荐）
    if (StringUtils.isNotBlank(request.getIdempotencyToken())) {
        return request.getIdempotencyToken();
    }
    
    // 方案3：哈希方式
    return DigestUtils.md5Hex(
        request.getOrderReference() +
        request.getUserId() +
        request.getAmount()
    );
}
```

#### 改进2：添加分布式锁

```java
// 添加Redis分布式锁
@Autowired
private RedissonClient redissonClient;

@Transactional(rollbackFor = Exception.class)
public PaymentInitResponse initiatePayment(PaymentInitRequest request) {
    String idempotencyKey = generateIdempotencyKey(request);
    
    // 获取分布式锁
    RLock lock = redissonClient.getLock("payment:lock:" + idempotencyKey);
    
    try {
        // 尝试获取锁，最多等待5秒，锁定30秒
        boolean locked = lock.tryLock(5, 30, TimeUnit.SECONDS);
        if (!locked) {
            throw new PaymentException("系统繁忙，请稍后重试");
        }
        
        // 再次检查（双重检查）
        PaymentTransaction existing = transactionMapper.selectByIdempotencyKey(idempotencyKey);
        if (existing != null) {
            return buildResponseFromTransaction(existing);
        }
        
        // 创建交易...
        
    } catch (InterruptedException e) {
        throw new PaymentException("获取锁失败", e);
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

## 总结

### 幂等性在支付系统中的地位

```
┌────────────────────────────────────────┐
│         支付系统核心四大支柱             │
├────────────────────────────────────────┤
│  1. 安全性（签名验证、加密）             │
│  2. 幂等性（防重复）        ⭐ 本文重点  │
│  3. 一致性（最终一致、对账）             │
│  4. 可用性（降级、限流）                 │
└────────────────────────────────────────┘
```

### 本系统幂等性评分

| 维度 | 评分 | 说明 |
|-----|------|------|
| 支付发起幂等 | ⭐⭐⭐⭐☆ | 有幂等键+数据库约束，建议加分布式锁 |
| 回调处理幂等 | ⭐⭐⭐⭐⭐ | 终态检查+乐观锁+日志，设计完善 |
| 对账修正幂等 | ⭐⭐⭐⭐☆ | 状态机保护，建议增加执行锁 |
| 并发安全 | ⭐⭐⭐☆☆ | 乐观锁可靠，但分布式场景需Redis锁 |

### 是否需要修改？

#### 🟢 如果是单机部署
**当前实现已足够**，无需修改：
- 数据库唯一约束保证幂等性
- 乐观锁防止并发问题
- 终态检查防止重复处理

#### 🟡 如果是分布式部署（推荐改进）
建议添加：
1. Redis分布式锁（高优先级）
2. 前端传递幂等Token（中优先级）
3. Redis回调去重（可选）

#### 🔴 如果有以下场景（必须改进）
- 预期高并发（TPS > 1000）
- 多实例部署
- 对重复零容忍的业务
