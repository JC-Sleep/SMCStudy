```bash

  datasource:

curl -X POST http://localhost:8080/api/payment/initiate \
  -H "Content-Type: application/json" \
```

```sql
-- 1. 每日交易统计
SELECT 
    TO_CHAR(CREATE_TIME, 'YYYY-MM-DD') AS day,
```

### 故障排查

#### 问题1：重复扣款

**检查步骤**：
```sql
-- 查找重复交易
SELECT * FROM PAYMENT_TRANSACTION 
WHERE ORDER_REFERENCE IN (
    SELECT ORDER_REFERENCE 
    FROM PAYMENT_TRANSACTION
    GROUP BY ORDER_REFERENCE 
    HAVING COUNT(*) > 1
)
ORDER BY ORDER_REFERENCE, CREATE_TIME;

-- 查看幂等键
SELECT IDEMPOTENCY_KEY, COUNT(*) 
FROM PAYMENT_TRANSACTION
GROUP BY IDEMPOTENCY_KEY
HAVING COUNT(*) > 1;  -- 不应该有重复

-- 如果有重复，检查是否是增强版
SELECT 
    TRANSACTION_ID,
    IDEMPOTENCY_KEY,
    CREATE_TIME,
    GATEWAY_NAME
FROM PAYMENT_TRANSACTION
WHERE ORDER_REFERENCE = 'CSP001xxxxx'
ORDER BY CREATE_TIME;
```

**解决方案**：
- 如果是标准版 → 升级到增强版
- 如果是增强版 → 检查Redis是否正常

#### 问题2：Redis连接失败

**检查**：
```bash
# 测试连接
redis-cli -h redis-host -p 6379 -a password PING

# 查看日志
tail -f logs/payment-system.log | grep Redis
```

**降级方案**：
```java
// 增强版会自动降级
if (redissonClient != null) {
    return doPaymentWithLock(request, gateway, idempotencyKey);
} else {
    log.warn("Redis不可用，降级为标准模式");
    return doPaymentWithoutLock(request, gateway, idempotencyKey);
}
```

#### 问题3：TPS下降

**排查清单**：
```
□ 数据库连接池是否耗尽？

□ 网络延迟是否增加？
□ 是否有慢SQL？
```

## 💡 核心技术要点

### 1. 幂等性设计

```
标准版：
数据库唯一索引 + 乐观锁 + 终态检查
适合：单机、TPS<500

增强版：
Redis分布式锁 + 前端Token + 数据库约束
适合：分布式、TPS>1000
```

### 2. 回调处理

```
关键点：
✅ 异步处理（立即返回200）
✅ 签名验证（防伪造）
✅ 幂等性保护（防重复）
✅ 完整日志（可追溯）
✅ 终态保护（不可逆）
```

### 3. 对账机制

```
定时对账：
- 30分钟对账一次（TIMEOUT交易）
- 凌晨3点全量对账（所有交易）

自动修正：
- 发现不一致 → 查询银行 → 更新状态
- 标记对账结果 → 人工复核
```

### 4. 多渠道扩展

```
新增渠道只需3步：
1. 实现PaymentGateway接口
2. 添加渠道枚举
3. 配置渠道参数
→ 无需修改业务代码
```

## 📞 技术支持

### 常见问题

**Q1: 标准版和增强版可以共存吗？**
A: 可以。两个Service可以同时存在，按需选择使用。

**Q2: 如果Redis宕机怎么办？**
A: 增强版会自动降级到标准版模式，依赖数据库唯一索引保护。

**Q3: 如何添加新的支付渠道？**
A: 参考 `多渠道支付扩展设计方案.md`，实现PaymentGateway接口即可。

**Q4: 幂等性如何测试？**
A: 并发发送100个相同请求，检查只创建1笔交易。

**Q5: 生产环境推荐哪个版本？**
A: 
- 单机 → 标准版
- 分布式/高并发 → 增强版
