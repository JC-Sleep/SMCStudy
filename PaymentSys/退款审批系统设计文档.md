# 退款审批系统设计文档

> 版本 v1.0 · 2026-05-08

---

## 一、为什么退款要走审批流？

| 风险场景 | 没有审批流的后果 |
|---------|----------------|
| 任何员工都能调退款 API | 内部人员误操作/恶意退款，公司资金损失 |
| 用户自己发起退款 | 恶意刷退款攻击 |
| 没有审计记录 | 出事无法追溯是谁操作，法律纠纷无证据 |
| 退款次数无限制 | 超额退款（总退款 > 原始金额） |

审批流把权力关进笼子：**用户只能申请，财务才能批准，钱才会退出去。**

## 二、同步退款 vs 异步退款——大厂怎么做的？

```
同步退款（❌不推荐）：
  财务点"批准" → 等待银行 API → HTTP 超时（渣打最多 10s）
  → 财务不知道退没退成 → 可能点第二次 → 双重退款！

异步退款（✅ 支付宝/微信/渣打标准做法）：
  财务点"批准" → 立即返回"处理中" → 后台异步调银行 API
  → 财务刷新列表查状态 → COMPLETED/FAILED
```

**本系统采用异步退款**，原因：
1. 银行 API 不稳定，超时 5~10 秒属正常
2. HTTP 线程不阻塞，体验好
3. 异步执行中状态为 `EXECUTING`，天然防止重复执行
4. 失败了有明确的 `FAILED` 状态 + 审计日志，不是静默丢失

## 四、权限验证原理（GroupId 体系）

```
每个用户属于一个 Group，Group 有父 Group。
登录后 JWT 里携带：userId, groupId, parentGroupId

parentGroupId = 345  →  普通财务
                           - 可查看/审批退款申请
                           - 同一笔交易最多批准 3 次部分退款

parentGroupId = 59   →  财务经理
                           - 所有财务权限
                           - 退款次数无限制

其他 parentGroupId   →  无财务权限，访问 /api/finance/** 返回 403
```

### 安全三层防护

| 层级 | 实现 | 拦截时机 |
|------|------|---------|
| Servlet Filter | `AuthFilter` | 所有请求，解析 JWT |
| MVC Interceptor | `FinanceAuthInterceptor` | `/api/finance/**` 路径级验证 |
| AOP | `FinanceAuthAspect` + `@RequireFinance` | Service 方法级验证（防绕过 HTTP 层） |

## 六、多次部分退款设计

```
原始金额：HKD 300.00

第 1 次退款申请：100.00
  → 财务批准 → 执行成功
  → PAYMENT_TRANSACTION.TOTAL_REFUNDED_AMOUNT = 100
  → PAYMENT_STATUS = PARTIALLY_REFUNDED（可继续退款）

第 2 次退款申请：100.00
  → 财务批准 → 执行成功
  → TOTAL_REFUNDED_AMOUNT = 200
  → PAYMENT_STATUS = PARTIALLY_REFUNDED

第 3 次退款申请：100.00（普通财务已达上限，需经理审批）
  → 普通财务尝试批准 → ❌ 已达3次上限，拒绝
  → 经理（parentGroupId=59）批准 → ✅
  → TOTAL_REFUNDED_AMOUNT = 300 = 原始金额
  → PAYMENT_STATUS = REFUNDED（终态，不可继续退款）
```

### 防超额三重校验

| 时机 | 校验内容 | 代码位置 |
|------|---------|---------|
| 用户提交申请 | 已有效申请金额之和 + 本次 ≤ 原始金额 | `RefundApplicationService.applyRefund()` |
| 财务批准时 | 普通财务：有效申请次数 ≤ 3 | `RefundApprovalService.approveApplication()` |
| 执行退款时 | PaymentTransaction 累计金额校验 | `PaymentServiceEnhanced.refund()` |

## 七、数据库设计

### REFUND_APPLICATION（退款申请主表）

| 字段 | 类型 | 说明 |
|------|------|------|
| ID | NUMBER(19) | 主键（Snowflake ID） |
| TRANSACTION_ID | VARCHAR2(100) | 关联支付交易 |
| REFUND_AMOUNT | NUMBER(10,2) | 本次申请退款金额 |
| ORIGINAL_AMOUNT | NUMBER(10,2) | 原始金额（冗余，校验用） |
| APPLICANT_USER_ID | VARCHAR2(100) | 申请人 |
| STATUS | VARCHAR2(20) | 状态（见枚举） |
| REVIEWED_BY | VARCHAR2(100) | 财务审批人 |
| REVIEWED_AT | DATE | 审批时间 |
| REFUND_NO | VARCHAR2(100) | 银行退款单号 |
| VERSION | NUMBER(10) | **乐观锁**（防并发重复审批/执行） |

### REFUND_AUDIT_LOG（审计日志，只增不改）

| 字段 | 类型 | 说明 |
|------|------|------|
| ACTION | VARCHAR2(30) | APPLY/APPROVE/REJECT/EXECUTE_SUCCESS/EXECUTE_FAILED |
| OPERATOR_USER_ID | VARCHAR2(100) | 操作人 |
| OPERATOR_PARENT_GROUP_ID | NUMBER(10) | 操作人角色（345/59/其他） |
| OPERATOR_IP | VARCHAR2(50) | **客户端 IP，防抵赖** |
| CREATE_TIME | DATE | 操作时间（无 UPDATE_TIME） |

### PAYMENT_TRANSACTION 新增字段

| 字段 | 类型 | 说明 |
|------|------|------|
| TOTAL_REFUNDED_AMOUNT | NUMBER(10,2) | 累计已退款金额 |
| PAYMENT_STATUS | +PARTIALLY_REFUNDED | 新增状态：已部分退款 |

## 九、JWT 使用指南

### 登录获取 Token
> 你们现有登录接口成功后调用 `JwtUtil.generateToken()` 生成 token，  
> 存入 Session 或返回给前端。

```java
// 示例：登录成功后生成 JWT
String token = jwtUtil.generateToken(
    "U001",   // userId
    "张三",   // username
    12,       // groupId
    345       // parentGroupId（345=普通财务, 59=经理）
);

// Web 登录：存入 Session
session.setAttribute(AuthFilter.SESSION_JWT_KEY, token);

// API 调用：前端存储，每次请求带在 Header
// Authorization: Bearer <token>
```

### JWT Payload 内容
```json
{
  "userId": "U001",
  "username": "张三",
  "groupId": 12,
  "parentGroupId": 345,
  "exp": 1748686800
}
```

### 配置项
```yaml
jwt:
  secret: ${JWT_SECRET:your-secret-key}  # 生产必须用环境变量！
  expire-hours: 24
```

## 十、安全清单

| 安全点 | 实现方式 | 文件 |
|--------|---------|------|
| JWT 签名防伪造 | HS256 + 服务端 secret | `JwtUtil.java` |
| Token 过期 | exp claim 验证 | `JwtUtil.parseToken()` |
| ThreadLocal 泄漏 | filter finally 块 clear() | `AuthFilter.java` |
| 财务路径防未授权 | `FinanceAuthInterceptor` | `WebMvcConfig.java` |
| 财务方法防绕过 | `@RequireFinance` + AOP | `FinanceAuthAspect.java` |
| 并发重复审批 | 乐观锁 version 字段 | `RefundApprovalService` |
| 并发重复执行 | EXECUTING 中间态 + 乐观锁 | `RefundExecutionService` |
| 重复申请 | 进行中申请计数检查 | `RefundApplicationService` |
| 超额退款 | 金额累计校验 | `RefundApplicationService` |
| 操作人 IP 记录 | REFUND_AUDIT_LOG.OPERATOR_IP | 每次写日志时记录 |
| 审计不可篡改 | 只 INSERT，无 UPDATE/DELETE 接口 | `RefundAuditLogMapper` |
| 普通财务次数上限 | count check（上限=3） | `RefundApprovalService` |
| 经理无限制 | `UserContext.isManager()` 跳过检查 | `RefundApprovalService` |

## 十一、FAILED 状态处理指南

当退款状态变为 `FAILED`，说明银行 API 调用失败：

```
1. 查询审计日志：GET /api/finance/refund/audit-log/{applicationId}
   → 查看 EXECUTE_FAILED 记录的 remark（失败原因）

2. 常见原因及处理：
   - 银行 API 超时：等 1~2 小时后查银行后台，确认是否已实际退款
   - PAYMENT_TRANSACTION 状态异常：联系开发查 REFUNDING 卡住的记录
   - 网络问题：重新在银行后台手动操作退款

3. 严禁直接修改 DB 状态绕过流程！
   如需重新触发退款，应重新提交申请流程。
```

## 十二、支付路由修复说明

同时修复了 `PaymentGatewayRouter.java` 的 Bug：

**原 Bug：**
```java
// 只注册 available 的网关，如配置缺失时被跳过
if (gateway.isAvailable()) {
    gatewayMap.put(channel, gateway);  ← 配置缺失 → 不注册
}
// 补全配置后必须重启才能路由到该渠道
```

**修复后：**
```java
// 始终注册所有网关，isAvailable() 在请求时实时检查
gatewayMap.put(channel, gateway);  // 无论是否配置
log.info("注册: {} available={}", channel.getCode(), gateway.isAvailable());

// 路由失败时输出诊断信息
if (availableGateways.isEmpty()) {
    // 输出所有渠道的可用状态，快速定位哪个渠道没配 API Key
    log.error("没有支持 [{}] 的网关。{各渠道状态}", paymentMethod, diagnosis);
}
```

同时删除了从未被填充和使用的 `paymentMethodGatewayMap`（死代码）。
