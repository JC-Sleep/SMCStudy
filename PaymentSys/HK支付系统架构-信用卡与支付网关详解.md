# HK 支付系统架构详解 —— 信用卡、支付网关与多银行接入

---

## 一、你的疑问解答

> "输入卡号和月份直接扣款，这些都是信用卡吗？是直接对接银行还是通过第三方？"

### 1.1 信用卡 vs 借记卡

| 卡类型 | 说明 | 典型卡头 |
|--------|------|---------|
| **信用卡 (Credit Card)** | Visa/Mastercard/Amex 发行，先消费后还款 | 4xxx / 5xxx / 3xxx |
| **借记卡 (Debit Card)** | 绑定银行账户，直接扣款，但也可走 Visa/MC 网络 | 同上，但带 Debit 标记 |
| **预付卡 (Prepaid Card)** | 充值后使用，同样走 Visa/MC 网络 | 同上 |

> **关键点**：只要卡面印有 Visa / Mastercard / Amex Logo，无论是信用卡还是借记卡，**都走同一套卡组织网络**，商户端的接入方式完全相同。

## 二、核心概念：卡组织 vs 发卡行 vs 收单行

```
┌─────────────┐     ①持卡人刷卡     ┌─────────────────┐
│   持卡人      │ ─────────────────► │  商户 (Merchant) │
│ (Cardholder) │                    └────────┬────────┘
└─────────────┘                             │ ②支付请求
       ▲                                    ▼
       │                          ┌─────────────────┐
       │  ⑥资金到账               │  支付网关/收单机构  │
       │                          │ (Payment Gateway │
┌──────┴──────┐                   │  / Acquirer)     │
│  发卡行      │                   └────────┬────────┘
│ (Issuing    │ ◄────────────────────────── │ ③转发授权请求
│  Bank)      │  ⑤授权响应                  ▼
│ (持卡人的银行)│          ┌─────────────────────────┐
└─────────────┘          │  卡组织网络               │
                         │  (Visa / Mastercard /    │
                         │   Amex / UnionPay)       │
                         └─────────────────────────┘
                                    ④路由授权
```

### 角色说明

| 角色 | 说明 | HK 常见例子 |
|------|------|------------|
| **卡组织 (Card Network/Scheme)** | 制定规则、负责在发卡行与收单行之间路由交易 | Visa、Mastercard、Amex、銀聯(UnionPay) |
| **发卡行 (Issuing Bank)** | 给持卡人发卡的银行 | HSBC、Hang Seng、BOC、Citibank |
| **收单行 (Acquiring Bank)** | 给商户提供收款服务的银行 | HSBC Merchant Services、渣打 |
| **支付网关 (Payment Gateway)** | 商户对接的技术中间层，处理加密、路由、协议转换 | Stripe、Adyen、PayMe、FPS、2C2P |

## 三、为什么不需要对接每家银行的 SDK？（核心解答）

### 3.1 商户只需对接「支付网关」或「收单行」

```
商户系统
   │
   │  只对接一个 SDK / API
   ▼
支付网关 (如 Stripe / Adyen / 2C2P)
   │
   │  网关内部处理以下路由：
   ├──► Visa Network ──► 持卡人的发卡行（无论是哪家银行）
   ├──► Mastercard Network ──► 持卡人的发卡行
   ├──► Amex Network ──► Amex 自己（发卡+清算一体）
   └──► UnionPay Network ──► 持卡人的发卡行
```

**商户完全不需要知道持卡人用的是哪家银行！**  
卡组织会根据卡号前缀（BIN/IIN）自动路由到对应的发卡行。

### 3.2 卡号 BIN 路由原理

```
卡号: 4532 1234 5678 9010
      ↑
      4 开头 = Visa 卡
      
卡号: 5412 7512 3456 7890  
      ↑
      51-55 开头 = Mastercard
      
卡号: 3714 496353 98431
      ↑
      34/37 开头 = Amex
```

支付网关拿到卡号后，自动判断走哪个卡组织网络，商户无感知。

## 四、HK 公司常见支付架构（十几家银行可选的场景）

你提到"有十几家银行可以选择"，这通常有两种情况：

### 场景 A：选择「收单行」（Acquiring Bank）

商户在开通商户账户时，可以选择用哪家银行作为收单行（即资金打到哪个商户账户）。  
这不影响持卡人用什么卡，只影响商户资金结算路径。

```
持卡人 (任意银行信用卡)
    │
    ▼
支付网关
    │
    ├── 若选渣打收单 ──► 资金结算到渣打商户账户
    ├── 若选汇丰收单 ──► 资金结算到汇丰商户账户
    └── 若选中银收单 ──► 资金结算到中银商户账户
```

### 场景 B：支付网关支持多种支付方式（Payment Methods）

如 Adyen / 2C2P 这类网关，支持：

```
客户端选择支付方式：
┌─────────────────────────────────┐
│  请选择支付方式：                  │
│  ○ Visa / Mastercard (信用卡)    │
│  ○ UnionPay (银联)               │
│  ○ FPS (转数快)                  │
│  ○ Alipay HK                    │
│  ○ WeChat Pay HK                │
│  ○ PayMe by HSBC                │
└─────────────────────────────────┘
```

商户只对接一个网关 SDK，网关内部适配所有支付方式。

## 八、为什么不需要每家银行单独 SDK

| 方案 | 工作量 | 维护成本 | 推荐度 |
|------|--------|---------|--------|
| ❌ 对接每家银行 SDK | 极高（十几个 SDK） | 极高 | 不推荐 |
| ✅ 对接支付网关（Stripe/Adyen） | 低（一套 API） | 低 | **强烈推荐** |
| ✅ 对接收单行（如渣打 PayNow） | 中（一套 API + 认证） | 中 | 适合银行直连需求 |

## 九、PCI DSS 合规说明（重要）

> 为什么前端可以直接输入卡号但商户不需要存卡号？

```
持卡人浏览器
     │
     │ ① 卡号直接发送到支付网关（不经过商户服务器）
     │    使用网关提供的 JS SDK / iframe / Hosted Fields
     ▼
支付网关（PCI DSS 认证）
     │
     │ ② 返回 Token（如 tok_xxx）
     ▼
商户后端
     │
     │ ③ 用 Token 发起扣款请求（商户服务器不接触真实卡号）
     ▼
支付网关处理真实扣款
```

**好处**：商户不需要自己做 PCI DSS 认证（成本极高），卡号由网关保管。

## 十、总结

1. **信用卡/借记卡** 只要有 Visa/MC/Amex Logo，商户接入方式一样，走卡组织网络
2. **商户不直接对接每家银行**，只对接一个「支付网关」或「收单行」
3. **支付网关** 内部帮你路由到正确的发卡行，商户无感知
4. **"十几家银行可选"** = 选择收单行/结算账户，或网关支持多种支付方式
5. **银联 (UnionPay)** 是另一个卡组织，和 Visa/MC 平行，支付网关通常也支持
6. **PCI DSS**：通过网关 Tokenization，商户不碰真实卡号，合规成本极低

## 十一、补充解答 —— Visa/MC/Amex 是什么组织？配哪个 SDK？

### 11.1 Visa / Mastercard / Amex 是「卡组织」，不是银行

```
常见误解：以为 Visa 是一家银行
正确理解：Visa 是一个"支付网络运营商"，类似高速公路公司

高速公路比喻：
  Visa/MC/Amex  =  高速公路（负责连通各地）
  发卡行 (HSBC)  =  你的出发地（钱从这里来）
  收单行         =  商户的目的地（钱到这里去）
  持卡人         =  开车的人
  商户           =  终点收费站
```

| 卡组织 | 全称 | 总部 | 特点 |
|--------|------|------|------|
| **Visa** | Visa Inc. | 美国 | 全球最大，4开头卡号 |
| **Mastercard** | Mastercard Inc. | 美国 | 全球第二，5开头卡号 |
| **Amex** | American Express | 美国 | 既是卡组织又是发卡行，3开头卡号 |
| **UnionPay** | 中国银联 | 中国 | 亚洲最大，62开头卡号 |

> **它们本身不存你的钱，只负责在银行之间传递授权指令和清算资金。**

### 11.2 完整流程一句话理解

```
你刷 HSBC 发的 Visa 信用卡
         │
         │ 卡面上有 Visa Logo → 走 Visa 网络
         ▼
    Visa 网络路由授权请求
         │
         │ 根据卡号 BIN 找到 HSBC（发卡行）
         ▼
    HSBC 批准 → 扣你信用额度 → 资金走 Visa 网络 → 到商户收单行
```

**所以答案是：Yes！配一个 SDK（支付网关），由卡组织去找对应银行扣钱。**

### 11.3 到底配哪个 SDK？

```
商户系统
    │
    │ 只需对接其中一个：
    │
    ├── Stripe SDK          ← 国际化首选，文档好，HK 支持 FPS
    ├── Adyen SDK           ← 企业级，支持最全（Visa/MC/Amex/UP/FPS/Octopus）
    ├── 2C2P SDK            ← 亚洲本地化强，HK/SG/TH 常用
    └── 渣打 PayNow SDK     ← 如果公司是渣打商户，直接用渣打网关
```

你们的系统是渣打银行对接，所以对接的是**渣打的支付网关 API**，  
渣打内部再帮你对接 Visa/MC/Amex 网络，你不用管。

## 十二、信用卡强制退款 —— Chargeback（拒付）机制

### 12.1 什么是 Chargeback？

> 这是信用卡的一个**消费者保护机制**，持卡人可以向发卡行申请强制退款，**不需要商户同意**。

```
正常退款（Refund）：
  持卡人 ──请求退款──► 商户 ──同意──► 支付网关 ──退款──► 持卡人
  （商户有主动权，可以拒绝）

强制退款 Chargeback：
  持卡人 ──投诉──► 发卡行（HSBC/Citi等）
                      │
                      │ 发卡行直接从商户账户扣回款项
                      ▼
                  商户收到 Chargeback 通知
                      │
                  ┌───┴────────────┐
                  │ 商户可以提出抗议 │
                  │ 需提供证据：     │
                  │ - 订单记录      │
                  │ - 物流/服务证明 │
                  └───┬────────────┘
                      │
              发卡行裁决（商户常输）
```

### 12.2 持卡人可以 Chargeback 的常见理由

| 理由 | 说明 | 商户能防吗？ |
|------|------|------------|
| **未授权交易** | 信用卡被盗刷 | 难防，需做 3DS 验证 |
| **商品未收到** | 物流问题 | 保留物流证明可抗议 |
| **商品与描述不符** | 货不对板 | 保留商品描述截图 |
| **重复扣款** | 系统 bug 扣了两次 | 系统幂等性设计可防 |
| **商户已关闭** | 商户跑路 | - |

### 12.3 Chargeback 对商户的影响

```
┌─────────────────────────────────────────────────┐
│  Chargeback 发生时，商户损失：                      │
│                                                 │
│  1. 被扣回原交易金额                               │
│  2. 额外被收 Chargeback 手续费（约 USD 15-50）     │
│  3. Chargeback 率 > 1% 可能被支付网关停号          │
└─────────────────────────────────────────────────┘
```

### 12.4 商户如何防止恶意 Chargeback？

```
技术手段：
  ✅ 3DS 2.0 验证（持卡人手机收短信/App确认）
       → 一旦持卡人通过3DS，Chargeback 责任转移到发卡行，商户免责！
  ✅ 幂等性设计（防重复扣款，你们系统已实现）
  ✅ 保留完整交易日志
  ✅ 发送交易确认邮件给持卡人

法律手段：
  ✅ 保留服务/商品交付证明
  ✅ 在规定时间内提交抗议文件（通常30-45天）
```

### 12.5 3DS 验证流程（防 Chargeback 核心手段）

```
持卡人        商户前端       支付网关        卡组织/发卡行      持卡人手机
  │              │              │                │               │
  │─ 输入卡号 ───►│              │                │               │
  │              │─ 发起3DS ────►│                │               │
  │              │              │─ 3DS请求 ───────►│               │
  │              │              │                │─ 推送验证码 ────►│
  │              │◄─ 弹出验证页 ─│                │               │
  │◄─ 输入验证码 ─│              │                │               │
  │─ 提交验证码 ──►│              │                │               │
  │              │─ 提交验证 ────►│                │               │
  │              │              │─ 验证请求 ───────►│               │
  │              │              │◄─ 验证通过 ───────│               │
  │              │◄─ 授权成功 ───│                │               │
  │◄─ 支付成功 ───│              │                │               │

  关键：通过3DS后，若持卡人 Chargeback，
        责任由「商户」转移到「发卡行」，商户免责！
```
