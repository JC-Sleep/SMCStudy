# Oracle统计信息过期影响详解 - 完整时序分析

## 🎯 核心回答

### 你的问题：
> 如果统计信息一直不更新，执行计划是否失效？查询是否还能查到数据？

### 答案：
1. ✅ **查询功能正常**：仍然能查到数据，不会报错
2. ❌ **性能严重下降**：执行计划基于错误的统计信息，可能慢10-100倍
3. ⚠️ **执行计划不会失效**：优化器仍会生成执行计划，但可能是错误的计划
4. 🔥 **后果**：数据库性能下降、用户投诉、系统超时

## 🔍 CRUD操作与统计信息自动更新机制详解

### 10%阈值是如何工作的？

```
Oracle的DML监控机制：
┌──────────────────────────────────────────────────────────────┐
│  1. DML操作 (INSERT/UPDATE/DELETE)                           │
│     ↓                                                        │
│  2. Oracle在内部字典表中记录变化量                           │
│     - SYS.MON_MODS_ALL$（监控表修改）                        │
│     - 记录：表名、修改行数、修改时间                         │
│     ↓                                                        │
│  3. 累计变化量达到阈值？                                     │
│     - 计算：修改行数 / 表总行数 > 10%                       │
│     - 例如：表有100万行，修改>10万行 → 触发                 │
│     ↓                                                        │
│  4. 标记表统计信息为"过期"                                   │
│     - USER_TAB_STATISTICS.STALE_STATS = 'YES'               │
│     ↓                                                        │
│  5. 等待维护窗口（自动任务）                                 │
│     - 默认时间：晚上22:00-凌晨06:00                         │
│     - 任务名：GATHER_STATS_JOB                              │
│     ↓                                                        │
│  6. 自动收集统计信息                                         │
│     - 执行：DBMS_STATS.GATHER_DATABASE_STATS_JOB_PROC       │
│     - 只收集STALE_STATS='YES'的表                           │
│     ↓                                                        │
│  7. 更新统计信息，标记STALE_STATS='NO'                       │
└──────────────────────────────────────────────────────────────┘
```

## ⚠️ 为什么不立即收集？Oracle的设计考虑

### 原因1：性能开销
```
统计信息收集的成本：
- 大表（100万行）：10-30秒
- 超大表（1000万行）：2-5分钟
- 巨型表（1亿行）：10-30分钟

如果每次DML操作都立即收集：
→ 频繁的统计信息收集
→ 占用大量CPU/IO资源
→ 影响正常业务操作
```

### 原因2：批量处理更高效
```
维护窗口批量收集的优势：
1. 业务低峰期（晚上）执行，不影响白天业务
2. 批量收集多个表，资源利用更高效
3. 可以使用更高的并行度和采样率
```

### 原因3：避免"抖动"
```
如果立即收集，可能出现：
10:00 - INSERT触发收集，耗时15秒
10:30 - UPDATE触发收集，耗时15秒
11:00 - DELETE触发收集，耗时15秒
...
→ 频繁收集，资源浪费
→ 统计信息频繁变化，执行计划不稳定
```

## 🔥 特殊场景：TRUNCATE和SQL*Loader为什么不触发？

### 场景对比：INSERT vs TRUNCATE + SQL*Loader

```
┌──────────────────────────────────────────────────────────────┐
│  场景1：常规INSERT（会触发DML监控）                          │
├──────────────────────────────────────────────────────────────┤
│  操作流程：                                                  │
│  INSERT → SQL引擎解析 → Buffer Cache → Redo Log → 数据文件  │
│              ↓                                               │
│         DML监控记录（SYS.MON_MODS_ALL$）✅                   │
│              ↓                                               │
│         累计变化>10% → 标记STALE_STATS='YES' ✅             │
│              ↓                                               │
│         维护窗口自动收集 ✅                                  │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│  场景2：TRUNCATE（不触发DML监控）                            │
├──────────────────────────────────────────────────────────────┤
│  操作流程：                                                  │
│  TRUNCATE → 直接释放表段（HWM重置）→ 数据文件              │
│              ↓                                               │
│         不经过DML监控 ❌                                     │
│         不记录Redo Log（最小化）                             │
│              ↓                                               │
│         统计信息保持旧值 ❌                                  │
│         STALE_STATS='NO'（未标记）❌                         │
│              ↓                                               │
│         永远不会自动收集 ❌                                  │
│                                                              │
│  原因：TRUNCATE是DDL操作，不是DML操作                        │
│        设计目标是快速清空表（快60倍），不关注统计信息        │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│  场景3：SQL*Loader直接路径（不触发DML监控）                  │
├──────────────────────────────────────────────────────────────┤
│  操作流程：                                                  │
│  SQL*Loader → 直接写入数据文件（绕过SQL引擎）               │
│              ↓                                               │
│         不经过DML监控 ❌                                     │
│         最小化Redo Log                                       │
│              ↓                                               │
│         统计信息保持旧值 ❌                                  │
│         STALE_STATS='NO'（未标记）❌                         │
│              ↓                                               │
│         永远不会自动收集 ❌                                  │
│                                                              │
│  原因：直接路径加载绕过SQL引擎，为了性能（快10倍）           │
│        不触发任何SQL层面的监控机制                           │
└──────────────────────────────────────────────────────────────┘
```

## 📋 统计信息过期检测与预防

### 检测脚本

```sql
-- ============================================================================
-- 检测统计信息过期风险
-- ============================================================================

-- 1. 检查统计信息与实际数据差异
SELECT 
    t.table_name,
    s.num_rows as stats_rows,
    (SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp) as actual_rows,
    ABS(s.num_rows - (SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp)) as difference,
    ROUND(ABS(s.num_rows - (SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp)) / 
          GREATEST((SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp), 1) * 100, 2) as diff_percent,
    s.last_analyzed,
    s.stale_stats,
    CASE 
        WHEN s.stale_stats = 'YES' THEN '❌ 已过期'
        WHEN s.last_analyzed IS NULL THEN '❌ 从未收集'
        WHEN ABS(s.num_rows - (SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp)) / 
             GREATEST((SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp), 1) > 0.2 THEN 
             '❌ 差异>20%（严重）'
        WHEN ABS(s.num_rows - (SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp)) / 
             GREATEST((SELECT COUNT(*) FROM FES.zz_pos_item_diff_temp), 1) > 0.1 THEN 
             '⚠️ 差异>10%（警告）'
        ELSE '✅ 正常'
    END as risk_level
```

## 📊 统计信息过期监控与告警方案

### 方案1：每日健康检查脚本

```sql
-- ============================================================================
-- 统计信息健康检查与告警脚本
-- 建议：每天上午执行，检查昨晚维护窗口是否成功收集
-- ============================================================================

SET SERVEROUTPUT ON SIZE UNLIMITED;

DECLARE
    v_table_count NUMBER := 0;
    v_stale_count NUMBER := 0;
    v_never_analyzed NUMBER := 0;
    v_old_stats NUMBER := 0;
    v_alert_msg VARCHAR2(4000) := '';
BEGIN
    DBMS_OUTPUT.PUT_LINE('======== 统计信息健康检查报告 ========');
    DBMS_OUTPUT.PUT_LINE('执行时间：' || TO_CHAR(SYSDATE, 'YYYY-MM-DD HH24:MI:SS'));
    DBMS_OUTPUT.PUT_LINE('');
    
    -- 检查1：统计信息过期的表
    SELECT COUNT(*)
    INTO v_stale_count
    FROM user_tab_statistics
    WHERE stale_stats = 'YES';
    
    DBMS_OUTPUT.PUT_LINE('【检查1】过期统计信息：' || v_stale_count || ' 个表');
    
    IF v_stale_count > 0 THEN
        DBMS_OUTPUT.PUT_LINE('  警告：以下表的统计信息已过期（STALE_STATS=YES）：');
        FOR rec IN (
            SELECT table_name, num_rows, last_analyzed
            FROM user_tab_statistics
            WHERE stale_stats = 'YES'
            ORDER BY num_rows DESC
            FETCH FIRST 10 ROWS ONLY
        ) LOOP
            DBMS_OUTPUT.PUT_LINE('  - ' || rec.table_name || 
                               ' (行数:' || rec.num_rows || 
                               ', 最后分析:' || TO_CHAR(rec.last_analyzed, 'YYYY-MM-DD') || ')');
        END LOOP;
        
        v_alert_msg := v_alert_msg || '【告警】' || v_stale_count || '个表统计信息过期；';
    END IF;
    
    DBMS_OUTPUT.PUT_LINE('');
    
    -- 检查2：从未收集统计信息的表
    SELECT COUNT(*)
    INTO v_never_analyzed
    FROM user_tab_statistics
    WHERE last_analyzed IS NULL
      AND num_rows > 0;
    
    DBMS_OUTPUT.PUT_LINE('【检查2】从未收集统计信息：' || v_never_analyzed || ' 个表');
    
    IF v_never_analyzed > 0 THEN
        DBMS_OUTPUT.PUT_LINE('  警告：以下表从未收集统计信息：');
        FOR rec IN (
            SELECT table_name
            FROM user_tab_statistics
            WHERE last_analyzed IS NULL
              AND num_rows > 0
            ORDER BY table_name
            FETCH FIRST 10 ROWS ONLY
        ) LOOP
            DBMS_OUTPUT.PUT_LINE('  - ' || rec.table_name);
        END LOOP;
        
        v_alert_msg := v_alert_msg || '【告警】' || v_never_analyzed || '个表从未收集统计信息；';
    END IF;
    
    DBMS_OUTPUT.PUT_LINE('');
    
    -- 检查3：统计信息超过7天未更新的表
    SELECT COUNT(*)
    INTO v_old_stats
    FROM user_tab_statistics
    WHERE last_analyzed < SYSDATE - 7
      AND num_rows > 10000;  -- 只检查大表
    
    DBMS_OUTPUT.PUT_LINE('【检查3】统计信息超过7天未更新（大表）：' || v_old_stats || ' 个表');
    
    IF v_old_stats > 0 THEN
        DBMS_OUTPUT.PUT_LINE('  警告：以下大表统计信息超过7天未更新：');
        FOR rec IN (
            SELECT table_name, num_rows, 
                   ROUND((SYSDATE - last_analyzed), 1) as days_old,
                   TO_CHAR(last_analyzed, 'YYYY-MM-DD') as last_date
            FROM user_tab_statistics
            WHERE last_analyzed < SYSDATE - 7
              AND num_rows > 10000
            ORDER BY num_rows DESC
            FETCH FIRST 10 ROWS ONLY
        ) LOOP
            DBMS_OUTPUT.PUT_LINE('  - ' || rec.table_name || 
                               ' (行数:' || rec.num_rows || 
                               ', ' || rec.days_old || '天前, ' || rec.last_date || ')');
        END LOOP;
        
        v_alert_msg := v_alert_msg || '【警告】' || v_old_stats || '个大表统计信息>7天；';
    END IF;
    
    DBMS_OUTPUT.PUT_LINE('');
    
    -- 检查4：对比实际行数与统计信息差异（针对关键表）
    DBMS_OUTPUT.PUT_LINE('【检查4】关键表统计信息准确性：');
    
    FOR rec IN (
        SELECT 
            'ZZ_POS_ITEM_DIFF_TEMP' as table_name,
            (SELECT num_rows FROM user_tables WHERE table_name = 'ZZ_POS_ITEM_DIFF_TEMP') as stats_rows,
            (SELECT COUNT(*) FROM zz_pos_item_diff_temp) as actual_rows
        FROM dual
    ) LOOP
        DECLARE
            v_diff_percent NUMBER;
        BEGIN
            IF rec.actual_rows > 0 THEN
                v_diff_percent := ABS(rec.stats_rows - rec.actual_rows) / rec.actual_rows * 100;
                
                DBMS_OUTPUT.PUT_LINE('  表：' || rec.table_name);
                DBMS_OUTPUT.PUT_LINE('    统计信息行数：' || rec.stats_rows);
                DBMS_OUTPUT.PUT_LINE('    实际行数：' || rec.actual_rows);
                DBMS_OUTPUT.PUT_LINE('    差异：' || ROUND(v_diff_percent, 2) || '%');
                
                IF v_diff_percent > 20 THEN
                    DBMS_OUTPUT.PUT_LINE('    状态：❌ 严重不准确（>20%）');
                    v_alert_msg := v_alert_msg || '【严重】' || rec.table_name || '统计信息差异' || ROUND(v_diff_percent, 1) || '%；';
                ELSIF v_diff_percent > 10 THEN
                    DBMS_OUTPUT.PUT_LINE('    状态：⚠️ 不够准确（>10%）');
                    v_alert_msg := v_alert_msg || '【警告】' || rec.table_name || '统计信息差异' || ROUND(v_diff_percent, 1) || '%；';
                ELSE
                    DBMS_OUTPUT.PUT_LINE('    状态：✅ 正常');
                END IF;
            END IF;
        END;
    END LOOP;
    
    DBMS_OUTPUT.PUT_LINE('');
    DBMS_OUTPUT.PUT_LINE('======== 检查完成 ========');
    
    -- 如果有告警，输出告警摘要
    IF v_alert_msg IS NOT NULL THEN
        DBMS_OUTPUT.PUT_LINE('');
        DBMS_OUTPUT.PUT_LINE('⚠️⚠️⚠️ 告警摘要 ⚠️⚠️⚠️');
        DBMS_OUTPUT.PUT_LINE(v_alert_msg);
        DBMS_OUTPUT.PUT_LINE('');
        DBMS_OUTPUT.PUT_LINE('建议执行：');
        DBMS_OUTPUT.PUT_LINE('  @gather_statistics.sql');
        DBMS_OUTPUT.PUT_LINE('或手动收集：');
        DBMS_OUTPUT.PUT_LINE('  EXEC DBMS_STATS.GATHER_SCHEMA_STATS(''FES'', cascade=>TRUE);');
    ELSE
        DBMS_OUTPUT.PUT_LINE('');
        DBMS_OUTPUT.PUT_LINE('✅ 所有统计信息正常！');
    END IF;
END;
/
```

### 方案2：自动告警并修复

```sql
-- ============================================================================
-- 自动检测并修复统计信息过期问题
-- 建议：配置为定时任务，每天上午执行
-- ============================================================================

CREATE OR REPLACE PROCEDURE auto_fix_stale_stats AS
    v_stale_count NUMBER;
    v_table_name VARCHAR2(128);
BEGIN
    -- 检查是否有过期的统计信息
    SELECT COUNT(*)
    INTO v_stale_count
    FROM user_tab_statistics
    WHERE stale_stats = 'YES';
    
    IF v_stale_count > 0 THEN
        DBMS_OUTPUT.PUT_LINE('发现 ' || v_stale_count || ' 个表统计信息过期，开始自动修复...');
        
        -- 逐个收集过期表的统计信息
        FOR rec IN (
            SELECT table_name, num_rows
            FROM user_tab_statistics
            WHERE stale_stats = 'YES'
            ORDER BY num_rows DESC  -- 先收集大表
        ) LOOP
            BEGIN
                DBMS_OUTPUT.PUT_LINE('正在收集：' || rec.table_name || ' (行数:' || rec.num_rows || ')');
                
                DBMS_STATS.GATHER_TABLE_STATS(
                    ownname => USER,
                    tabname => rec.table_name,
                    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
                    cascade => TRUE,
                    degree => 4,
                    no_invalidate => FALSE
                );
                
                DBMS_OUTPUT.PUT_LINE('  ✅ 完成');
            EXCEPTION
                WHEN OTHERS THEN
                    DBMS_OUTPUT.PUT_LINE('  ❌ 失败：' || SQLERRM);
            END;
        END LOOP;
        
        DBMS_OUTPUT.PUT_LINE('自动修复完成');
    ELSE
        DBMS_OUTPUT.PUT_LINE('✅ 所有表统计信息正常，无需修复');
    END IF;
END;
/

-- 创建定时任务（每天上午8点执行）
BEGIN
    DBMS_SCHEDULER.CREATE_JOB(
        job_name        => 'AUTO_FIX_STALE_STATS_JOB',
        job_type        => 'PLSQL_BLOCK',
        job_action      => 'BEGIN auto_fix_stale_stats; END;',
        start_date      => SYSTIMESTAMP,
        repeat_interval => 'FREQ=DAILY; BYHOUR=8; BYMINUTE=0',
        enabled         => TRUE,
        comments        => '自动修复过期统计信息'
    );
END;
/
```

## ✅ 关键记忆点（补充）

### CRUD与统计信息更新机制：
1. **INSERT/UPDATE/DELETE** → 触发DML监控 → 累计>10% → 标记过期 → 等维护窗口收集
2. **TRUNCATE** → 不触发DML监控 → 统计信息保持旧值 → 永远不自动收集 ❌
3. **SQL*Loader** → 不触发DML监控 → 统计信息保持旧值 → 永远不自动收集 ❌

### 时间延迟：
- **触发标记**：变化>10%后立即标记（STALE_STATS='YES'）
- **实际收集**：等到维护窗口（默认晚上22:00）
- **延迟时间**：可能长达**12-20小时** ⚠️

### 执行计划影响：
- **不会失效**：执行计划仍然存在
- **但会错误**：基于过期统计信息生成错误计划
- **查询仍成功**：能返回正确数据
- **性能下降**：可能慢10-100倍 ❌
