# Oracle统计信息快速参考卡片

## 🎯 3秒理解统计信息

```
统计信息 = 数据的"说明书"（不是索引本身）
作用 = 帮助优化器选择最快的查询路径
重要性 = 缺失/过期会导致查询慢10倍以上！
```

---

## ⚡ 最常用命令

### 收集统计信息（最重要！）
```sql
-- 收集单表统计信息（推荐）
EXEC DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'FES',
    tabname => 'ZZ_POS_ITEM_DIFF_TEMP',
    cascade => TRUE,
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    degree => 4
);
```

### 验证统计信息
```sql
SELECT 
    table_name, 
    TO_CHAR(num_rows, '999,999,999') as num_rows,
    TO_CHAR(last_analyzed, 'YYYY-MM-DD HH24:MI:SS') as last_analyzed,
    stale_stats as 是否过期
FROM user_tab_statistics
WHERE table_name = 'ZZ_POS_ITEM_DIFF_TEMP';

-- last_analyzed应为最近时间
-- stale_stats应为'NO'
```

---

## 🔥 关键场景

### ❌ 不会自动收集统计信息的操作（必须手动收集！）

| 操作 | 为什么不自动收集 | 解决方案 |
|------|------------------|----------|
| **TRUNCATE** | DDL操作，不记录DML | 立即执行`GATHER_TABLE_STATS` |
| **SQL*Loader (DIRECT=TRUE)** | 绕过SQL引擎，不触发监控 | 导入后执行`GATHER_TABLE_STATS` |
| **TRUNCATE + SQL*Loader** | 两者都不触发 | **最危险！**必须手动收集 |
| **DROP + CREATE INDEX** | 新索引无统计信息 | 创建后执行`GATHER_INDEX_STATS` |
| **IMPDP** | 默认不导入统计信息 | 导入后执行`GATHER_SCHEMA_STATS` |

## ✅ 每日工作流程（TRUNCATE + SQL*Loader）

```bash
# 步骤1: 清空表
sqlplus FES/password << EOF
TRUNCATE TABLE zz_pos_item_diff_temp;
EXIT;
EOF

# 步骤2: 导入数据
sqlldr userid=FES/password control=load.ctl direct=true

# 步骤3: 收集统计信息（关键！）
sqlplus FES/password << EOF
EXEC DBMS_STATS.GATHER_TABLE_STATS('FES', 'ZZ_POS_ITEM_DIFF_TEMP', cascade=>TRUE);
EXIT;
EOF

# 步骤4: 验证统计信息
sqlplus FES/password << EOF
SELECT table_name, num_rows, last_analyzed FROM user_tables WHERE table_name='ZZ_POS_ITEM_DIFF_TEMP';
EXIT;
EOF
```

## 🚨 故障排查

### 问题1：查询很慢
```sql
-- 检查统计信息
SELECT table_name, num_rows, last_analyzed, stale_stats
FROM user_tab_statistics
WHERE table_name = 'ZZ_POS_ITEM_DIFF_TEMP';

-- 如果last_analyzed为NULL或很久以前 → 立即收集
EXEC DBMS_STATS.GATHER_TABLE_STATS('FES', 'ZZ_POS_ITEM_DIFF_TEMP', cascade=>TRUE);
```

### 问题2：统计信息显示0行，实际有数据
```sql
-- 典型场景：TRUNCATE + SQL*Loader后忘记收集
-- 对比实际行数
SELECT COUNT(*) FROM zz_pos_item_diff_temp;  -- 实际：1,000,000
SELECT num_rows FROM user_tables WHERE table_name='ZZ_POS_ITEM_DIFF_TEMP';  -- 统计：0

-- 立即修复
EXEC DBMS_STATS.GATHER_TABLE_STATS('FES', 'ZZ_POS_ITEM_DIFF_TEMP', cascade=>TRUE);
```

### 问题3：收集了统计信息，执行计划还是不对
```sql
-- 原因：游标缓存未失效
-- 解决1：收集时立即失效游标
EXEC DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'FES',
    tabname => 'ZZ_POS_ITEM_DIFF_TEMP',
    cascade => TRUE,
    no_invalidate => FALSE  -- 关键参数
);

-- 解决2：清空共享池（慎用）
ALTER SYSTEM FLUSH SHARED_POOL;
```

## 📊 性能对比

### 场景：查询100万行表，返回50万行数据

| 统计信息状态 | 优化器选择 | 执行时间 | 差异 |
|-------------|-----------|---------|------|
| ❌ 缺失/过期 | 索引扫描+50万次回表 | 15秒 | 基准 |
| ✅ 准确 | 全表扫描（顺序IO） | 1.5秒 | **10倍** |

**结论：** 统计信息缺失/过期会导致**10倍性能差异**！

## 🔧 高级选项

### 加速统计信息收集

```sql
-- 方式1：降低采样率（快3倍，稍不准确）
EXEC DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'FES',
    tabname => 'ZZ_POS_ITEM_DIFF_TEMP',
    estimate_percent => 10,  -- 只采样10%
    cascade => TRUE
);

-- 方式2：增加并行度（快5-8倍，需要多核CPU）
EXEC DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'FES',
    tabname => 'ZZ_POS_ITEM_DIFF_TEMP',
    degree => 8,  -- 并行度8
    cascade => TRUE
);
```

### 检查哪些表需要收集统计信息

```sql
-- 查找统计信息过期的表
SELECT 
    table_name,
    num_rows,
    last_analyzed,
    stale_stats
FROM user_tab_statistics
WHERE stale_stats = 'YES'  -- 标记为过期
   OR last_analyzed IS NULL  -- 从未收集
   OR last_analyzed < SYSDATE - 7  -- 超过7天
ORDER BY last_analyzed NULLS FIRST;
```

## 📞 常见问题速查

| 问题 | 答案 |
|------|------|
| **TRUNCATE会自动收集统计信息吗？** | ❌ 不会！必须手动收集 |
| **SQL*Loader会自动收集吗？** | ❌ 不会！必须手动收集 |
| **INSERT会自动收集吗？** | ⚠️ 变化>10%后，等维护窗口自动收集（有延迟） |
| **收集统计信息需要多久？** | 百万数据约10-30秒 |
| **不收集统计信息会怎样？** | 查询可能慢10倍以上！ |
| **统计信息过期，查询会失败吗？** | ❌ 不会！查询仍成功，但性能下降10-100倍 |
| **执行计划会失效吗？** | ❌ 不会失效，但会基于错误统计信息生成错误计划 |
| **统计信息会占用多少空间？** | 很小，只有几KB（存在数据字典） |
| **统计信息就是索引吗？** | ❌ 不是！统计信息是数据的"说明书" |

## ⚠️ 统计信息过期后果（简化版）

### 完整过程（时序）：
```
Day 1 早上：TRUNCATE表
    ↓
统计信息：num_rows=100万（旧值）❌
实际数据：0行
    ↓
Day 1 上午：SQL*Loader导入120万行
    ↓
统计信息：num_rows=100万（仍是旧值）❌
实际数据：120万行
    ↓
用户查询：WHERE compare_status='1'
    ↓
优化器推理：
"统计信息显示100万行，返回50%约50万行"
"索引扫描+50万次回表，成本低"
选择：索引扫描 ❌（错误决策）
    ↓
实际执行：
索引扫描+72万次回表（实际是60%）
随机IO：72万次
耗时：35秒 ❌❌❌
    ↓
用户投诉：页面超时
    ↓
DBA收集统计信息
    ↓
统计信息：num_rows=120万 ✅
    ↓
再次查询：
优化器选择：全表扫描 ✅（正确决策）
耗时：1.8秒 ✅
```

### 关键结论：
- ✅ **查询仍成功**：能返回正确数据，不会报错
- ❌ **性能严重下降**：从1.8秒变成35秒（慢20倍）
- ⚠️ **执行计划错误**：不是失效，是基于错误信息的错误计划
- 🔧 **解决方案**：立即收集统计信息
