# 索引优化与统计信息完整指南

## 📁 脚本清单

| 脚本文件 | 用途 | 何时使用 |
|---------|------|---------|
| `rename_indexes.sql` | 重命名索引（不改结构） | 仅需改名称，保留统计信息 |
| `optimize_indexes_with_stats.sql` | **索引结构优化+统计信息收集** | 需要优化索引以支持复杂查询 |
| `rollback_optimize_indexes.sql` | 回滚索引优化 | 优化后发现性能问题 |
| `gather_statistics.sql` | 日常统计信息收集 | 定期维护（建议每周） |
| `rollback_rename_indexes.sql` | 回滚索引重命名 | 需要恢复原索引名 |

```
是否需要改变索引？
├─ 仅改名称

```

## 📊 统计信息更新规则总结

### ✅ 需要收集统计信息的场景

| 操作 | 原因 | 收集方式 |
|------|------|---------|
| **DROP + CREATE索引** | 新索引无统计信息 | `DBMS_STATS.GATHER_INDEX_STATS` |
| **大量数据变更（>10%）** | 统计信息过期 | `DBMS_STATS.GATHER_TABLE_STATS` |
| **ALTER INDEX REBUILD** | 索引重建后建议更新 | `DBMS_STATS.GATHER_INDEX_STATS` |
| **TRUNCATE表后重新导入** | 统计信息完全失效 | `DBMS_STATS.GATHER_TABLE_STATS` |

### ❌ 不需要收集统计信息的场景

| 操作 | 原因 |
|------|------|
| **ALTER INDEX RENAME** | 统计信息自动保留 |
| **少量数据变更（<5%）** | 统计信息仍然准确 |
| **仅查询操作** | 统计信息不变 |

## 🔍 故障排查

### 问题1：执行计划仍然使用旧索引
**原因：** 
- 游标缓存未失效
- 统计信息未生效

**解决：**
```sql
-- 清空共享池（慎用，会影响所有会话）
ALTER SYSTEM FLUSH SHARED_POOL;

-- 或重新收集统计信息（推荐）
EXEC DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'FES',
    tabname => 'ZZ_POS_ITEM_DIFF_TEMP',
    cascade => TRUE,
    no_invalidate => FALSE  -- 立即失效游标
);
```

### 问题2：统计信息收集失败
**错误：** `ORA-20000: object statistics are locked`

**解决：**
```sql
-- 解锁统计信息
EXEC DBMS_STATS.UNLOCK_TABLE_STATS('FES', 'ZZ_POS_ITEM_DIFF_TEMP');

-- 重新收集
@gather_statistics.sql
```

### 问题3：索引创建失败
**错误：** `ORA-01408: such column list already indexed`

**解决：**
```sql
-- 查找重复索引
SELECT index_name, column_name, column_position
FROM user_ind_columns
WHERE table_name = 'ZZ_POS_ITEM_DIFF_TEMP'
ORDER BY index_name, column_position;

-- 删除重复索引
DROP INDEX duplicate_index_name;
```

## 📞 关键SQL参考

### 查看当前正在使用的索引
```sql
SELECT i.index_name, i.table_name, s.value as index_scans
FROM user_indexes i
LEFT JOIN v$segment_statistics s ON i.index_name = s.object_name
WHERE i.table_name IN ('ZZ_POS_ITEM_DIFF_TEMP', 'ZZ_PSUB_REF_DIFF_TEMP')
  AND s.statistic_name = 'logical reads'
ORDER BY s.value DESC NULLS LAST;
```

### 查看索引效率
```sql
SELECT 
    i.index_name,
    i.num_rows,
    i.clustering_factor,
    t.num_rows as table_rows,
    ROUND(i.clustering_factor / t.num_rows * 100, 2) as efficiency_pct
FROM user_indexes i
JOIN user_tables t ON i.table_name = t.table_name
WHERE i.table_name IN ('ZZ_POS_ITEM_DIFF_TEMP', 'ZZ_PSUB_REF_DIFF_TEMP')
ORDER BY efficiency_pct DESC;
```
**说明：** efficiency_pct越接近100%，索引效率越高

## ⚠️ 重要提醒

**如果你的表使用TRUNCATE + SQL*Loader导入数据：**

1. ✅ **必须**在导入后手动收集统计信息
2. ✅ 推荐使用 `import_with_statistics.bat/sh` 一体化脚本
3. ✅ 详细原理请阅读：`Oracle统计信息完整解析.md`

**原因：** TRUNCATE和SQL*Loader都不会自动触发统计信息收集，导致优化器使用过期统计信息，查询性能可能慢**10倍以上**！
