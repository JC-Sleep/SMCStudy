## 场景1: 违反最左前缀原则

### 📋 场景说明

复合索引必须从第一列开始，连续使用。跳过或缺少第一列会导致索引失效。

### 🔧 索引定义

```sql
CREATE INDEX idx_user ON orders(user_id, order_date, status, amount);
```

### ❌ 失效案例

#### 案例1.1: 缺少第一列

```sql
-- ❌ 索引失效
SELECT * FROM orders WHERE order_date = '2026-04-02';
```

**过程图：**
```
查询: WHERE order_date = '2026-04-02'

索引结构:
┌─────────────┬──────────────┬────────┬────────┐
│  user_id    │ order_date   │ status │ amount │
├─────────────┼──────────────┼────────┼────────┤
│ 100         │ 2026-04-01   │ PAID   │ 1000   │
│ 100         │ 2026-04-02   │ PAID   │ 2000   │ ← 目标数据
│ 101         │ 2026-04-01   │ PAID   │ 1500   │
│ 101         │ 2026-04-02   │ PAID   │ 2500   │ ← 目标数据
│ 102         │ 2026-04-02   │ PENDING│ 3000   │ ← 目标数据
└─────────────┴──────────────┴────────┴────────┘

问题: order_date 分散在所有 user_id 中
      无法直接定位到 order_date = '2026-04-02'
      
决策: 放弃使用索引 → 全表扫描
```

**执行计划：**
```
┌────────────────────────────────────────┐
│ Operation: TABLE ACCESS FULL           │
│ Object: orders                         │
│ Cost: 1000                             │
│ Rows: 10000                            │
└────────────────────────────────────────┘
       ↓
┌────────────────────────────────────────┐
│ 扫描所有行                              │
│ 过滤: order_date = '2026-04-02'        │
└────────────────────────────────────────┘
```

#### 案例1.2: 跳过中间列

```sql
-- ⚠️ 部分使用索引（只用第1列）
SELECT * FROM orders WHERE user_id = 100 AND status = 'PAID';
```

**过程图：**
```
查询: WHERE user_id = 100 AND status = 'PAID'

索引查找过程:
    [根节点]
       ↓
    [user_id = 100] ✅ 能定位
       ↓
    ┌─────────────────────────────────┐
    │ user_id=100 的所有数据           │
    │ - order_date: 2026-04-01, ...   │
    │ - order_date: 2026-04-02, ...   │
    │ - order_date: 2026-04-03, ...   │
    └─────────────────────────────────┘
       ↓
    ⚠️ 跳过了 order_date（第2列）
       ↓
    ❌ 无法使用 status（第3列）
       ↓
    需要扫描 user_id=100 的所有行
    然后过滤 status='PAID'
```

**执行计划：**
```
┌────────────────────────────────────────┐
│ Operation: INDEX RANGE SCAN            │
│ Index: idx_user                        │
│ Access: user_id = 100                  │
│ Cost: 10                               │
└────────────────────────────────────────┘
       ↓
┌────────────────────────────────────────┐
│ TABLE ACCESS BY INDEX ROWID            │
│ Filter: status = 'PAID'                │
│ Cost: 20                               │
└────────────────────────────────────────┘

效率: 中等（使用了索引，但需要回表过滤）
```

## 场景2: 在索引列上使用函数

### 📋 场景说明

在索引列上使用函数会导致索引失效，除非创建了对应的函数索引。

### 🔧 索引定义

```sql
CREATE INDEX idx_create_date ON orders(create_date);
```

### ❌ 失效案例

#### 案例2.1: 使用TRUNC函数

```sql
-- ❌ 索引失效
SELECT * FROM orders WHERE TRUNC(create_date) = TRUNC(SYSDATE);
```

**过程图：**
```
索引存储内容:
┌──────────────────────────┐
│ create_date              │
├──────────────────────────┤
│ 2026-04-02 09:30:15      │
│ 2026-04-02 14:20:00      │
│ 2026-04-02 18:45:30      │
│ 2026-04-03 08:00:00      │
└──────────────────────────┘

查询: WHERE TRUNC(create_date) = TRUNC(SYSDATE)
需要的值: 2026-04-02 (日期部分)

问题分析:
┌──────────────────────────────────────────┐
│ 索引存储: 精确时间戳                      │
│ 查询需要: TRUNC后的日期                   │
│                                          │
│ 无法匹配!                                │
│                                          │
│ Oracle必须:                              │
│ 1. 扫描整个表                            │
│ 2. 对每行执行 TRUNC(create_date)         │
│ 3. 与 TRUNC(SYSDATE) 比较                │
└──────────────────────────────────────────┘
```

**执行计划对比：**
```
❌ 未使用函数索引:
┌────────────────────────────────────────┐
│ Operation: TABLE ACCESS FULL           │
│ Filter: TRUNC(create_date)=TRUNC(...)  │
│ Cost: 1000                             │
│ Rows Processed: 100000                 │
└────────────────────────────────────────┘

✅ 使用函数索引:
CREATE INDEX idx_trunc_date ON orders(TRUNC(create_date));

┌────────────────────────────────────────┐
│ Operation: INDEX RANGE SCAN            │
│ Index: idx_trunc_date                  │
│ Access: TRUNC(create_date)=TRUNC(...)  │
│ Cost: 10                               │
│ Rows Processed: 100                    │
└────────────────────────────────────────┘

性能提升: 1000倍
```

#### 案例2.2: 使用其他函数

```sql
-- ❌ 索引失效
SELECT * FROM orders WHERE UPPER(customer_name) = 'JOHN';
SELECT * FROM orders WHERE SUBSTR(phone, 1, 3) = '138';
SELECT * FROM orders WHERE TO_CHAR(create_date, 'YYYYMM') = '202604';
```

**可视化对比：**
```
普通索引 vs 函数索引

情况1: 普通索引 + 函数查询
┌─────────────────────────────────────────────────────┐
│ 索引: CREATE INDEX idx ON t(name)                  │
│ 查询: WHERE UPPER(name) = 'JOHN'                   │
│                                                     │
│ 索引内容: 'john', 'John', 'JOHN', 'alice', ...     │
│ 需要匹配: UPPER('john')='JOHN'? UPPER('John')=... │
│                                                     │
│ 结果: 无法直接使用索引 ❌                           │
│ 必须全表扫描，对每行执行UPPER函数                   │
└─────────────────────────────────────────────────────┘

情况2: 函数索引
┌─────────────────────────────────────────────────────┐
│ 索引: CREATE INDEX idx ON t(UPPER(name))           │
│ 查询: WHERE UPPER(name) = 'JOHN'                   │
│                                                     │
│ 索引内容: 'JOHN', 'JOHN', 'JOHN', 'ALICE', ...     │
│ 直接匹配: 'JOHN' = 'JOHN' ✅                        │
│                                                     │
│ 结果: 直接使用索引 ✅                               │
└─────────────────────────────────────────────────────┘
```

## 场景3: 隐式类型转换

### 📋 场景说明

当查询条件的数据类型与索引列类型不一致时，Oracle会进行隐式转换，可能导致索引失效。

### 🔧 索引定义

```sql
CREATE INDEX idx_phone ON customers(phone_number);  -- VARCHAR2类型
```

### ❌ 失效案例

#### 案例3.1: 字符串列用数字查询

```sql
-- ❌ 索引失效
SELECT * FROM customers WHERE phone_number = 13812345678;
```

**隐式转换过程：**
```
原始查询:
WHERE phone_number = 13812345678

Oracle隐式转换为:
WHERE TO_NUMBER(phone_number) = 13812345678
           ↑
        对索引列使用了函数！

等价于:
┌──────────────────────────────────────────┐
│ 对每一行执行:                             │
│ TO_NUMBER('13812345678') = 13812345678   │
│ TO_NUMBER('13898765432') = 13812345678   │
│ TO_NUMBER('13587654321') = 13812345678   │
│ ...                                      │
└──────────────────────────────────────────┘

结果: 索引失效，全表扫描
```

**类型转换规则可视化：**
```
Oracle隐式转换规则:

数字 vs 字符串:
┌──────────────────────────────────────────────────────┐
│ 列类型: VARCHAR2        查询值: NUMBER               │
│ WHERE col = 123                                      │
│         ↓                                            │
│ TO_NUMBER(col) = 123  ❌ 对列使用函数，索引失效       │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│ 列类型: NUMBER          查询值: VARCHAR2             │
│ WHERE col = '123'                                    │
│         ↓                                            │
│ col = TO_NUMBER('123')  ✅ 对值转换，索引有效        │
└──────────────────────────────────────────────────────┘

结论: 在"值"上做转换OK，在"列"上做转换会失效！
```

#### 案例3.2: 日期类型转换

```sql
-- ❌ 可能失效
SELECT * FROM orders WHERE create_date = '2026-04-02';
```

**转换过程：**
```
假设 create_date 是 DATE 类型

Oracle转换:
WHERE create_date = TO_DATE('2026-04-02', 'YYYY-MM-DD')
                          ↑
                    对值转换，索引有效 ✅

但如果是:
WHERE TO_CHAR(create_date, 'YYYY-MM-DD') = '2026-04-02'
      ↑
    对列使用函数，索引失效 ❌
```

## 场景4: 使用OR连接不同列

### 📋 场景说明

使用OR连接不同列时，即使每列都有索引，也可能无法有效使用。

### 🔧 索引定义

```sql
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status ON orders(status);
```

### ❌ 失效案例

```sql
-- ⚠️ 索引使用效率低
SELECT * FROM orders WHERE user_id = 100 OR status = 'PAID';
```

**查询过程可视化：**
```
查询: WHERE user_id = 100 OR status = 'PAID'

Oracle可能的执行方案:

方案1: 使用两个索引（效率低）
┌────────────────────────────────────────┐
│ 步骤1: INDEX RANGE SCAN (idx_user_id) │
│        找到 user_id = 100 的行         │
│        → RowID: [R1, R2, R3, ...]     │
└────────────────────────────────────────┘
       ↓
┌────────────────────────────────────────┐
│ 步骤2: INDEX RANGE SCAN (idx_status)  │
│        找到 status = 'PAID' 的行       │
│        → RowID: [R5, R6, R7, ...]     │
└────────────────────────────────────────┘
       ↓
┌────────────────────────────────────────┐
│ 步骤3: OR 合并                         │
│        RowID: [R1,R2,R3,R5,R6,R7,...]│
│        需要去重                        │
└────────────────────────────────────────┘

方案2: 全表扫描（可能更快）
┌────────────────────────────────────────┐
│ TABLE ACCESS FULL                      │
│ 直接扫描，对每行判断条件               │
└────────────────────────────────────────┘

Oracle选择: 通常选择方案2（全表扫描）
```

**✅ 优化方案：使用UNION**
```sql
SELECT * FROM orders WHERE user_id = 100
UNION ALL
SELECT * FROM orders WHERE status = 'PAID' AND user_id != 100;
```

**优化后的过程：**
```
┌────────────────────────────────────────┐
│ 查询1: INDEX RANGE SCAN (idx_user_id) │
│        WHERE user_id = 100             │
│        → 快速精准定位                  │
└────────────────────────────────────────┘
       ↓ UNION ALL
┌────────────────────────────────────────┐
│ 查询2: INDEX RANGE SCAN (idx_status)  │
│        WHERE status='PAID' AND user!=100│
│        → 快速精准定位                  │
└────────────────────────────────────────┘
       ↓
┌────────────────────────────────────────┐
│ 结果合并（无需去重）                   │
└────────────────────────────────────────┘

效率: 高 ✅
```

## 场景5: LIKE前导通配符

### 📋 场景说明

LIKE查询如果以通配符开头，无法使用索引。

### 🔧 索引定义

```sql
CREATE INDEX idx_name ON customers(customer_name);
```

### ❌ 失效案例

```sql
-- ❌ 索引失效（前导通配符）
SELECT * FROM customers WHERE customer_name LIKE '%John%';
SELECT * FROM customers WHERE customer_name LIKE '%Smith';

-- ✅ 可以使用索引
SELECT * FROM customers WHERE customer_name LIKE 'John%';
```

**LIKE匹配过程可视化：**
```
索引结构（B-Tree，按字母顺序）:
┌─────────────────────────────────┐
│ 'Alice Johnson'                 │
│ 'Bob Smith'                     │
│ 'Charlie Brown'                 │
│ 'David Johnson'                 │
│ 'John Doe'                      │
│ 'John Smith'                    │
│ 'Johnny Walker'                 │
│ 'Smith Anderson'                │
└─────────────────────────────────┘

场景1: LIKE 'John%' ✅
┌─────────────────────────────────┐
│ 可以快速定位到 'John' 开头的位置│
│         ↓                       │
│ 'John Doe'      ← 起点          │
│ 'John Smith'                    │
│ 'Johnny Walker' ← 终点          │
│         ↓                       │
│ 只扫描 3 行                     │
└─────────────────────────────────┘
效率: 高 ✅

场景2: LIKE '%John%' ❌
┌─────────────────────────────────┐
│ 无法定位起始位置                 │
│         ↓                       │
│ 必须扫描所有行:                  │
│ 'Alice Johnson'  ✅ 包含 John   │
│ 'Bob Smith'      ❌             │
│ 'Charlie Brown'  ❌             │
│ 'David Johnson'  ✅ 包含 John   │
│ 'John Doe'       ✅ 包含 John   │
│ 'John Smith'     ✅ 包含 John   │
│ 'Johnny Walker'  ✅ 包含 John   │
│ 'Smith Anderson' ❌             │
│         ↓                       │
│ 扫描全部 8 行                   │
└─────────────────────────────────┘
效率: 低 ❌（全表扫描）

场景3: LIKE '%John' ❌
┌─────────────────────────────────┐
│ 无法定位起始位置                 │
│ 必须扫描所有行检查后缀           │
│ 扫描全部 8 行                   │
└─────────────────────────────────┘
效率: 低 ❌（全表扫描）
```

**解决方案：**
```
如果必须使用 LIKE '%keyword%':

方案1: 全文索引（Oracle Text）
CREATE INDEX idx_name_fulltext ON customers(customer_name) 
INDEXTYPE IS CTXSYS.CONTEXT;

SELECT * FROM customers 
WHERE CONTAINS(customer_name, 'John') > 0;

方案2: 反向索引（适用于后缀匹配）
CREATE INDEX idx_name_reverse ON customers(REVERSE(customer_name));

WHERE REVERSE(customer_name) LIKE REVERSE('%John');
→ WHERE REVERSE(customer_name) LIKE 'nhoJ%';

方案3: 使用搜索引擎（Elasticsearch等）
```

## 场景6: 使用不等于操作符

### 📋 场景说明

使用 `!=` 或 `<>` 操作符时，Oracle通常选择全表扫描。

### 🔧 索引定义

```sql
CREATE INDEX idx_status ON orders(status);
```

### ❌ 失效案例

```sql
-- ⚠️ 可能不使用索引
SELECT * FROM orders WHERE status != 'CANCELLED';
SELECT * FROM orders WHERE status <> 'CANCELLED';
```

**查询过程分析：**
```
假设状态分布:
┌──────────────────────────────────┐
│ CANCELLED:  1,000  (1%)          │
│ PAID:      40,000 (40%)          │
│ PENDING:   30,000 (30%)          │
│ SHIPPED:   29,000 (29%)          │
│ 总计:     100,000                │
└──────────────────────────────────┘

查询: WHERE status != 'CANCELLED'
返回: 99,000 行（99%的数据）

Oracle分析:
┌────────────────────────────────────────┐
│ 方案1: 使用索引                        │
│ - INDEX RANGE SCAN                     │
│ - 需要读取 99,000 个索引条目           │
│ - 需要 99,000 次回表                   │
│ - 成本: 99,000 * 2 = 198,000         │
└────────────────────────────────────────┘

┌────────────────────────────────────────┐
│ 方案2: 全表扫描                        │
│ - TABLE ACCESS FULL                    │
│ - 顺序读取 100,000 行                  │
│ - 成本: 100,000                        │
└────────────────────────────────────────┘

决策: 选择全表扫描（成本更低）❌
```

**可视化对比：**
```
选择性高（返回少量数据）:
WHERE status = 'CANCELLED' (返回1%)
┌────────────────────────┐
│ INDEX RANGE SCAN ✅    │
│ 读取 1,000 索引条目    │
│ 回表 1,000 次          │
│ 成本: 2,000            │
└────────────────────────┘

选择性低（返回大量数据）:
WHERE status != 'CANCELLED' (返回99%)
┌────────────────────────┐
│ TABLE ACCESS FULL ✅   │
│ 顺序扫描全表           │
│ 成本: 100,000          │
└────────────────────────┘
```

**✅ 优化方案：**
```sql
-- 改用 IN 列举所有需要的值
SELECT * FROM orders 
WHERE status IN ('PAID', 'PENDING', 'SHIPPED');
```

## 场景7: IS NULL/IS NOT NULL

### 📋 场景说明

对于可空列，`IS NULL` 或 `IS NOT NULL` 可能无法有效使用索引。

### 🔧 索引定义

```sql
CREATE INDEX idx_email ON customers(email);
```

### ❌ 失效案例

```sql
-- ⚠️ 可能不使用索引
SELECT * FROM customers WHERE email IS NULL;
SELECT * FROM customers WHERE email IS NOT NULL;
```

**NULL值在索引中的存储：**
```
Oracle索引特性:
┌──────────────────────────────────────────┐
│ 普通B-Tree索引不存储NULL值               │
│                                          │
│ 索引内容:                                │
│ 'alice@example.com'                      │
│ 'bob@example.com'                        │
│ 'charlie@example.com'                    │
│ ...                                      │
│                                          │
│ NULL值: 不在索引中 ❌                    │
└──────────────────────────────────────────┘

查询: WHERE email IS NULL
┌──────────────────────────────────────────┐
│ 索引中没有NULL值                         │
│         ↓                                │
│ 必须全表扫描查找NULL                     │
└──────────────────────────────────────────┘

查询: WHERE email IS NOT NULL
┌──────────────────────────────────────────┐
│ 如果大部分数据都有email（99%）           │
│         ↓                                │
│ 全表扫描更快                             │
└──────────────────────────────────────────┘
```

**✅ 解决方案：**
```sql
-- 方案1: 复合索引（包含一个不为NULL的列）
CREATE INDEX idx_email_comp ON customers(email, 1);
-- 或
CREATE INDEX idx_email_comp ON customers(email, user_id);

-- 方案2: 函数索引
CREATE INDEX idx_email_nvl ON customers(NVL(email, 'NULL'));

-- 方案3: 位图索引（仅适用于低基数列）
CREATE BITMAP INDEX idx_email_bitmap ON customers(email);
```

## 场景8: 对索引列进行运算

### 📋 场景说明

对索引列进行数学运算或拼接会导致索引失效。

### 🔧 索引定义

```sql
CREATE INDEX idx_salary ON employees(salary);
CREATE INDEX idx_first_name ON employees(first_name);
```

### ❌ 失效案例

#### 案例8.1: 数学运算

```sql
-- ❌ 索引失效
SELECT * FROM employees WHERE salary * 12 > 100000;
SELECT * FROM employees WHERE salary + bonus > 50000;
SELECT * FROM employees WHERE salary / 2 < 3000;
```

**运算导致失效的过程：**
```
原始查询:
WHERE salary * 12 > 100000

等价于:
WHERE f(salary) > 100000
      ↑
    对列使用了函数！

索引无法使用:
┌──────────────────────────────────────────┐
│ 索引存储: salary 的原始值                │
│ 5000, 6000, 7000, 8000, ...              │
│                                          │
│ 查询需要: salary * 12 的计算结果         │
│ 60000, 72000, 84000, 96000, ...          │
│                                          │
│ 不匹配! 必须全表扫描                     │
└──────────────────────────────────────────┘
```

**✅ 正确写法：**
```sql
-- 把运算移到查询值一侧
SELECT * FROM employees WHERE salary > 100000 / 12;
SELECT * FROM employees WHERE salary > 50000 - bonus;
```

**对比可视化：**
```
❌ 错误: WHERE salary * 12 > 100000
┌─────────────────────────────────────────┐
│ 对每一行:                               │
│ salary=5000 → 5000*12=60000 > 100000? ❌│
│ salary=6000 → 6000*12=72000 > 100000? ❌│
│ salary=7000 → 7000*12=84000 > 100000? ❌│
│ salary=8000 → 8000*12=96000 > 100000? ❌│
│ salary=9000 → 9000*12=108000 > 100000?✅│
│ ...                                     │
│ 全表扫描: 100,000行                     │
└─────────────────────────────────────────┘

✅ 正确: WHERE salary > 8333.33
┌─────────────────────────────────────────┐
│ 索引查找:                               │
│ 定位到 salary > 8333.33 的位置          │
│         ↓                               │
│ salary=8400 ✅                          │
│ salary=8500 ✅                          │
│ salary=9000 ✅                          │
│ ...                                     │
│ 索引扫描: 只读取符合条件的行            │
└─────────────────────────────────────────┘
```

#### 案例8.2: 字符串拼接

```sql
-- ❌ 索引失效
SELECT * FROM employees 
WHERE first_name || ' ' || last_name = 'John Smith';
```

**✅ 优化方案：**
```sql
-- 方案1: 分别查询
WHERE first_name = 'John' AND last_name = 'Smith'

-- 方案2: 创建虚拟列索引
ALTER TABLE employees ADD (
    full_name AS (first_name || ' ' || last_name)
);
CREATE INDEX idx_full_name ON employees(full_name);
```

## 场景9: 优化器认为全表扫描更快

### 📋 场景说明

即使有索引，如果Oracle优化器计算后认为全表扫描成本更低，也会放弃使用索引。

### 🔧 索引定义

```sql
CREATE INDEX idx_status ON orders(status);
```

### ⚠️ 案例

```sql
-- 返回大量数据时，可能不使用索引
SELECT * FROM orders WHERE status = 'PAID';
```

**优化器决策过程：**
```
数据分布分析:

└──────────────────────────────────────┘

查询: WHERE status = 'PAID'

成本计算:
┌────────────────────────────────────────┐
│ 方案1: 使用索引                        │
│ ┌────────────────────────────────────┐ │
│ │ 1. INDEX RANGE SCAN                │ │

│ │    I/O: 900个索引块                │ │
│ │                                    │ │
│ │ 2. TABLE ACCESS BY INDEX ROWID     │ │
│ │    回表90,000次                    │ │

│ │                                    │ │
```

## 🎯 索引失效总结表

| 场景 | 示例 | 失效原因 | 解决方案 |
|------|------|----------|----------|
| **违反最左原则** | `WHERE col2=?` (索引:col1,col2) | 缺少第一列 | 添加第一列条件或调整索引顺序 |
| **函数应用** | `WHERE TRUNC(date)=?` | 对索引列使用函数 | 创建函数索引 |
| **隐式转换** | `WHERE varchar_col=123` | 自动类型转换 | 使用正确类型 '123' |
| **OR连接** | `WHERE col1=? OR col2=?` | 多列OR | 改用UNION |
| **前导通配符** | `WHERE name LIKE '%abc%'` | 无法定位起始位置 | 使用全文索引或改为'abc%' |
| **不等于** | `WHERE status != 'X'` | 返回数据量大 | 改用IN列举需要的值 |
| **NULL判断** | `WHERE col IS NULL` | NULL不存储在B-Tree索引 | 使用复合索引或位图索引 |
| **列运算** | `WHERE salary*12>?` | 对列使用运算 | 把运算移到值一侧 |
| **选择性差** | `WHERE status='COMMON'` (90%数据) | 全表扫描成本更低 | 优化查询条件或使用分区 |

## 💡 最佳实践建议

### 1. 索引设计原则

```
✅ DO:
- 在WHERE、JOIN、ORDER BY列上创建索引
- 高选择性列放在复合索引前面
- 定期更新统计信息
- 为常用查询创建专用索引

❌ DON'T:
- 在小表上创建过多索引
- 在更新频繁的列上创建索引
- 创建冗余索引
- 忽略索引维护
```

### 2. 查询优化原则

```
✅ DO:
- 避免在索引列上使用函数
- 使用正确的数据类型
- 遵循最左前缀原则
- 尽量使用等值查询

❌ DON'T:
- 不要使用 SELECT *
- 避免复杂的子查询
- 少用OR，多用UNION
- 避免隐式类型转换
```
