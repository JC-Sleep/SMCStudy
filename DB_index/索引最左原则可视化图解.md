# 索引最左原则可视化图解
## 🎯 核心概念：一张图看懂最左原则
```
索引: idx_example(A, B, C, D)
物理存储结构（B-Tree）:
                    [根节点]
                       |
        ┌──────────────┼──────────────┐
        |              |              |
    [A=1分支]      [A=2分支]      [A=3分支]
        |              |              |
    ┌───┴───┐      ┌───┴───┐      ┌───┴───┐
 [B=1] [B=2]    [B=1] [B=2]    [B=1] [B=2]
    |     |        |     |        |     |
  [叶子] [叶子]  [叶子] [叶子]  [叶子] [叶子]
   (C,D) (C,D)   (C,D) (C,D)   (C,D) (C,D)
```
### 查询路径分析
```
查询1: WHERE A=1                    ✅ 使用索引
路径: 根 → A=1分支 → [找到所有数据]
查询2: WHERE A=1 AND B=2            ✅ 使用索引
路径: 根 → A=1分支 → B=2分支 → [找到所有数据]
查询3: WHERE B=2                    ❌ 不使用索引
路径: 根 → ??? (不知道走哪个A分支) → 只能全表扫描
查询4: WHERE A=1 AND C=3            ⚠️ 部分使用
路径: 根 → A=1分支 → [扫描所有B分支] → 过滤C=3
```

### 你的查询1路径分析
```sql
SELECT * FROM zz_pos_item_diff_temp 
WHERE TRUNC(CREATE_DATE) = TRUNC(SYSDATE)    -- ⚠️ 不在此索引中
  and invoice_no_nb='MVNOs invo'             -- ❌ 第4层（无法直达）
  and compare_status = '2'                   -- ✅ 第1层
  and inv_type_nb ='02'                      -- ⚠️ 不在此索引中
ORDER BY compare_status, subscriber, cellular;
```
**查询路径:**
```
1. 根节点 → compare_status='2' 分支  ✅ 可以使用
2. 到达 compare_status='2' 后...
   - 需要找 subscriber_nb=??? → ❌ WHERE中没有，必须扫描所有分支
   - 需要找 cellular_nb=??? → ❌ WHERE中没有，必须扫描所有分支
   - 需要找 invoice_no_nb='MVNOs invo' → ⚠️ 即使有，也无法利用（跳过了前面的层）
结论: 只使用了第1层 compare_status，后续层级无法利用
```
**实际执行:**
```
INDEX RANGE SCAN | idx_pos_diff_new_data (如果选择此索引)
  ├─ 第1步: 定位到 compare_status='2' 的索引块 ✅
  ├─ 第2步: 扫描该块下的所有 subscriber_nb ❌ (全扫描)
  ├─ 第3步: 扫描所有 cellular_nb ❌ (全扫描)
  ├─ 第4步: 过滤 invoice_no_nb='MVNOs invo' ⚠️ (索引过滤)
  └─ 第5步: 回表获取其他列并过滤 inv_type_nb='02' ❌ (回表过滤)
效率: 低（大量索引扫描）
```
**更好的选择: idx_pos_diff_export_main**
```
INDEX RANGE SCAN | idx_pos_diff_export_main
  ├─ 第1步: 定位到 TRUNC(create_date)=今天 的索引块 ✅
  ├─ 第2步: 定位到 compare_status='2' 的子块 ✅
  └─ 第3步: 回表过滤其他条件 ⚠️
效率: 高（精准定位）
```

## 🔍 最左原则失效案例可视化
### 案例：为什么跳过中间列会失效？
索引: (A, B, C, D)  
查询: WHERE A=1 AND C=3
```
索引物理存储:
A=1, B=1, C=1, D=1  →  RowID_001
A=1, B=1, C=2, D=1  →  RowID_002
A=1, B=1, C=3, D=1  →  RowID_003  ✅ 符合条件
A=1, B=2, C=1, D=1  →  RowID_004
A=1, B=2, C=3, D=1  →  RowID_005  ✅ 符合条件
A=1, B=3, C=2, D=1  →  RowID_006
A=1, B=3, C=3, D=1  →  RowID_007  ✅ 符合条件
A=2, B=1, C=1, D=1  →  RowID_008
问题: 索引按 (A, B, C) 排序
- A=1 的数据聚集在一起 ✅
- 但 C=3 的数据分散在不同的 B 值中 ❌
- 必须扫描 A=1 下的所有 B 值，然后过滤 C=3
效率: 中等（使用了A，但B和C效率低）
```
---
### 案例：为什么缺少第一列会完全失效？
索引: (A, B, C)  
查询: WHERE B=2 AND C=3
```
索引物理存储:
A=1, B=1, C=1  →  RowID_001
A=1, B=2, C=3  →  RowID_002  ✅ 符合条件（但不知道）
A=2, B=1, C=1  →  RowID_003
A=2, B=2, C=3  →  RowID_004  ✅ 符合条件（但不知道）
A=3, B=2, C=3  →  RowID_005  ✅ 符合条件（但不知道）
问题: B=2 的数据分散在所有 A 值中
- 无法直接定位到 B=2 的位置
- 必须扫描整个索引（或全表扫描）
效率: 极低（等同于全表扫描）
Oracle决策: 放弃使用此索引，直接全表扫描
```

## 💡 最佳实践：如何设计索引
### 规则1: 高选择性列放前面
```sql
-- 假设数据分布:
-- user_id: 100万个唯一值 (选择性高)
-- status: 只有3个值 ('ACTIVE', 'PENDING', 'CLOSED') (选择性低)
-- ✅ 好的设计
CREATE INDEX idx ON orders(user_id, status);
-- 查询: WHERE user_id=100 AND status='ACTIVE'
-- 效率: 先定位到user_id=100的少量数据，再过滤status
-- ❌ 不好的设计
CREATE INDEX idx ON orders(status, user_id);
-- 查询: WHERE user_id=100 AND status='ACTIVE'
-- 效率: 先定位到status='ACTIVE'的大量数据，再过滤user_id
```
---
### 规则2: 等值查询列放在范围查询列前面
```sql
-- 查询: WHERE user_id=100 AND create_date >= '2026-01-01'
-- ✅ 好的设计
CREATE INDEX idx ON orders(user_id, create_date);
-- 效率: 定位user_id=100 → 扫描该用户的日期范围
-- ❌ 不好的设计
CREATE INDEX idx ON orders(create_date, user_id);
-- 效率: 扫描日期范围 → 过滤user_id（效率低）
```
---
### 规则3: WHERE列放在ORDER BY列前面
```sql
-- 查询: WHERE user_id=100 ORDER BY create_date
-- ✅ 好的设计
CREATE INDEX idx ON orders(user_id, create_date);
-- 效率: 定位user_id=100 → 数据已按create_date排序
-- ❌ 不好的设计
CREATE INDEX idx ON orders(create_date, user_id);
-- 效率: 无法直接定位user_id=100，需要额外排序
```

### 当前情况分析
```
你的查询1:
WHERE TRUNC(create_date)=today 
  AND invoice_no_nb='x' 
  AND compare_status='2' 
  AND inv_type_nb='02'
ORDER BY compare_status, subscriber, cellular
现有索引:
1. idx_pos_diff_export_main(TRUNC(date), compare_status, subscriber, cellular, invoice_no)
2. idx_pos_diff_new_data(compare_status, subscriber_nb, cellular_nb, invoice_no_nb)

- 索引1: 缺少 invoice_no_nb, inv_type_nb
- 索引2: 缺少 TRUNC(create_date)，且ORDER BY不匹配

```
---
### 优化方案：创建专用索引
```sql
-- 专用索引设计
CREATE INDEX idx_pos_diff_custom_query ON zz_pos_item_diff_temp(
    TRUNC(create_date),      -- WHERE条件1（高选择性）
    compare_status,          -- WHERE条件2（低选择性）
    invoice_no_nb,           -- WHERE条件3（中选择性）
    inv_type_nb,            -- WHERE条件4（中选择性）
    subscriber,              -- ORDER BY 列1
    cellular                 -- ORDER BY 列2
);
```
### 优化后的查询路径
```

┌────────────────────────────────────────────────────┐
│ 步骤1: 定位 TRUNC(create_date)=today               │

│        └→ 进一步缩小范围 ✅                         │
├────────────────────────────────────────────────────┤
│ 步骤3: 在步骤2结果中，定位 invoice_no_nb='x'       │
│        └→ 再次缩小范围 ✅                           │
├────────────────────────────────────────────────────┤
│ 步骤4: 在步骤3结果中，定位 inv_type_nb='02'        │
│        └→ 精准定位目标数据 ✅                       │
├────────────────────────────────────────────────────┤
│ 步骤5: 读取索引中的 subscriber, cellular            │
│        └→ 数据已按此顺序排序，无需额外排序 ✅       │

结果: 完美的Index Range Scan，性能最优！
```

## 🎓 学习总结
### 最左原则的本质
**就像查字典:**
1. 必须先知道**第一个字母**才能找到对应页码
2. 如果你只知道**第二个字母**，无法直接翻到目标页
3. 如果你知道**第一和第三个字母**，可以找到第一个字母的页，但要在该页中逐个查找第三个字母
### TRUNC函数索引的本质
**普通索引 vs 函数索引:**
```
普通索引: CREATE INDEX idx ON t(create_date);
存储内容: 2026-04-02 09:30:15, 2026-04-02 14:20:00, 2026-04-03 08:00:00
查询: WHERE TRUNC(create_date) = TRUNC(SYSDATE)
问题: 索引存储的是精确时间，查询需要的是日期
结果: 无法使用索引 ❌
---
函数索引: CREATE INDEX idx ON t(TRUNC(create_date));
存储内容: 2026-04-02, 2026-04-02, 2026-04-03
查询: WHERE TRUNC(create_date) = TRUNC(SYSDATE)
问题: 索引存储的就是TRUNC后的日期
结果: 完美匹配，使用索引 ✅
```
### MySQL中的TRUNC
**MySQL没有TRUNC函数，使用DATE函数:**
```sql
-- Oracle
WHERE TRUNC(create_date) = TRUNC(SYSDATE)
-- MySQL等价
WHERE DATE(create_date) = CURDATE()
-- MySQL 8.0+函数索引
CREATE INDEX idx ON t((DATE(create_date)));
-- MySQL 5.7替代方案（虚拟列）
ALTER TABLE t ADD date_only DATE AS (DATE(create_date)) STORED;
CREATE INDEX idx ON t(date_only);
```
