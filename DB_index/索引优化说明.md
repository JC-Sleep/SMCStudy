# 索引优化说明文档

## 📋 问题总结

### 原始问题
1. **Oracle不支持PostgreSQL风格的WHERE子句部分索引**
   ```sql
   -- ❌ Oracle不支持
   CREATE INDEX idx ON table(col1) WHERE col2 = 'value';
   ```

2. **CASE表达式索引在Oracle中语法错误**
   ```sql
   -- ❌ Oracle不支持普通B-tree索引中使用CASE
   CREATE INDEX idx ON table(CASE WHEN status = '1' THEN col1 END);
   ```

3. **冗余索引设计**
   - 多个索引覆盖相同查询模式
   - 缺少对实际查询的优化

## 🎯 关键优化技术

### 1. **函数索引 (Function-Based Index)**
```sql
CREATE INDEX idx_diff_temp_trunc_date ON ZZ_PSUB_REF_diff_temp(TRUNC(create_date));
```
- **原因**: `WHERE TRUNC(CREATE_DATE) = TRUNC(SYSDATE)` 使用了TRUNC函数
- **效果**: 避免全表扫描，直接使用索引

### 2. **覆盖索引 (Covering Index)**
```sql
CREATE INDEX idx_diff_temp_export_main ON ZZ_PSUB_REF_diff_temp(
    TRUNC(create_date), 
    compare_status,
    Subscriber,
    Cellular
);
```
- **原因**: 包含WHERE条件和ORDER BY列
- **效果**: Index Only Scan，不需要回表查询

### 3. **索引压缩 (Index Compression)**
```sql
CREATE INDEX idx_diff_temp_old_data ON ZZ_PSUB_REF_diff_temp(
    compare_status, 
    Subscriber, 
    Cellular
) COMPRESS 1;
```
- **原因**: `compare_status` 只有两个值 ('1', '2')，重复率极高
- **效果**: 减少索引空间占用50%以上

### 4. **避免部分索引模拟**
❌ **不要使用以下方式模拟部分索引：**
```sql
-- 方式1: WHERE子句（Oracle不支持）
CREATE INDEX idx ON table(col1) WHERE status = '1';

-- 方式2: CASE表达式（性能差）
CREATE INDEX idx ON table(CASE WHEN status = '1' THEN col1 END);

-- 方式3: 虚拟列（过于复杂）
ALTER TABLE t ADD (col1_filtered AS (CASE WHEN status='1' THEN col1 END));
CREATE INDEX idx ON t(col1_filtered);
```

✅ **改用复合索引 + COMPRESS：**
```sql
CREATE INDEX idx ON table(status, col1) COMPRESS 1;
-- Oracle会自动优化 WHERE status = '1' AND col1 = ?
```

## 📝 注意事项

### 1. **函数索引的限制**
- 使用函数索引后，查询必须使用完全相同的函数表达式
- ✅ 匹配: `WHERE TRUNC(create_date) = TRUNC(SYSDATE)`
- ❌ 不匹配: `WHERE TO_CHAR(create_date, 'YYYYMMDD') = TO_CHAR(SYSDATE, 'YYYYMMDD')`

### 2. **索引维护**
```sql
-- 定期重建索引（建议每月）
ALTER INDEX idx_diff_temp_export_main REBUILD ONLINE;
ALTER INDEX idx_pos_diff_export_main REBUILD ONLINE;

-- 或者合并索引（更快）
ALTER INDEX idx_diff_temp_export_main COALESCE;
```

### 3. **空间管理**
```sql
-- 查看索引占用空间
SELECT segment_name, segment_type, bytes/1024/1024 AS size_mb
FROM user_segments
WHERE segment_type = 'INDEX'
AND segment_name LIKE '%DIFF%'
ORDER BY bytes DESC;
```

### 4. **性能监控**
```sql
-- 查看索引使用情况（需要开启监控）
ALTER INDEX idx_diff_temp_export_main MONITORING USAGE;

-- 查询使用统计
SELECT * FROM V$OBJECT_USAGE 
WHERE index_name LIKE '%DIFF%';
```

## ❓ 常见问题

### Q1: 为什么不使用位图索引（Bitmap Index）？
**A:** 
- ❌ 位图索引适合**只读**或**低并发DML**场景
- ✅ 该表是**临时表**，频繁INSERT/DELETE
- ✅ B-tree索引更适合OLTP场景

### Q2: COMPRESS 1 是什么意思？
**A:**
- `COMPRESS 1` 表示压缩第一列的重复值
- 例如: `(compare_status, subscriber)` 中的 `compare_status` 只有 '1' 和 '2'
- 压缩后索引大小可减少50-70%

### Q3: 为什么不使用全局临时表（Global Temporary Table）？
**A:**
- 当前设计使用普通表 + `create_date` 区分批次
- 如果改用GTT，需要修改Java代码逻辑
- 当前方案可保留历史数据用于审计

### Q4: 索引太多会影响INSERT性能吗？
**A:**
- 是的，每个索引都会增加INSERT/UPDATE/DELETE开销
- 当前设计已经精简到5个索引（从原来8个）
- 主要是批量INSERT，影响较小（约10-15%）

### 2. **自动清理历史数据**
```sql
-- 创建定时任务，每天清理7天前数据
BEGIN
  DBMS_SCHEDULER.CREATE_JOB (
    job_name        => 'CLEAN_DIFF_TEMP_DATA',
    job_type        => 'PLSQL_BLOCK',
    job_action      => '
      BEGIN
        DELETE FROM ZZ_PSUB_REF_diff_temp WHERE create_date < SYSDATE - 7;
        DELETE FROM zz_pos_item_diff_temp WHERE create_date < SYSDATE - 7;
        COMMIT;
      END;',
    start_date      => SYSTIMESTAMP,
    repeat_interval => 'FREQ=DAILY; BYHOUR=2',
    enabled         => TRUE
  );
END;
/
```
