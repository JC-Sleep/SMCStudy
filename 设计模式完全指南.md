# 设计模式完全指南 🏗️

> 结合支付系统 & 优惠券系统，从项目角度理解设计模式

---

## 一、设计模式是什么？用什么实现的？

设计模式的实现基础是面向对象的三大核心概念：

| 概念 | 作用 | 设计模式怎么用 |
|------|------|---------------|
| **接口（Interface）** | 定义行为契约 | 策略、观察者、工厂等大量用接口定义"能做什么" |
| **抽象类（Abstract Class）** | 定义骨架+部分实现 | 模板方法模式的核心，定义流程，留空给子类填 |
| **继承（extends）** | 复用代码 | 模板方法、装饰器等会用到 |
| **组合/聚合** | has-a关系 | 策略、装饰器、代理、责任链都是"持有另一个对象" |

### 接口 vs 抽象类 —— 选哪个？

```
接口（Interface）：
  ✅ 只定义行为，不含实现
  ✅ 可多实现（一个类 implements 多个接口）
  ✅ 策略模式、观察者模式首选
  例：PaymentStrategy、CouponHandler

抽象类（Abstract Class）：
  ✅ 可以有默认实现（模板方法放在这里）
  ✅ 只能单继承
  ✅ 模板方法模式首选
  例：AbstractPaymentProcessor、AbstractCouponValidator
```

## 二、23种设计模式全览

### 创建型（5种）—— 怎么造对象

| 模式 | 核心思想 | 实现方式 | 常见场景 |
|------|---------|---------|---------|
| **单例** | 全局唯一实例 | 私有构造+静态方法 | Spring Bean、线程池、配置类 |
| **工厂方法** | 子类决定造哪种对象 | 抽象类/接口定义工厂方法 | Spring BeanFactory |
| **抽象工厂** | 造一族相关对象 | 接口定义一组工厂方法 | 跨数据库DAO、换主题UI |
| **建造者** | 分步骤构建复杂对象 | 链式调用Builder类 | Lombok @Builder、Request构建 |
| **原型** | 克隆已有对象 | 实现Cloneable接口 | 深拷贝对象、缓存对象复用 |

---

### 结构型（7种）—— 怎么组织类/对象

| 模式 | 核心思想 | 实现方式 | 常见场景 |
|------|---------|---------|---------|
| **代理** | 控制对真实对象的访问 | 实现同一接口/继承，持有真实对象 | AOP、@Transactional、@Cacheable |
| **装饰器** | 动态叠加功能 | 实现同一接口，持有被装饰对象 | Java IO流、缓存增强 |
| **适配器** | 让不兼容的接口协同工作 | 包装旧接口，暴露新接口 | 第三方SDK对接、旧系统改造 |
| **门面（外观）** | 简化复杂系统调用 | 提供一个统一入口类 | Service层封装多个DAO |
| **桥接** | 抽象与实现分离 | 接口持有另一个接口 | 跨平台、多维度变化 |
| **组合** | 树形结构统一处理 | 节点和叶子实现同一接口 | 菜单树、组织架构、文件目录 |
| **享元** | 共享细粒度对象节省内存 | 对象池+工厂 | String常量池、Integer缓存 |

---

### 行为型（11种）—— 怎么分配职责/通信

| 模式 | 核心思想 | 实现方式 | 常见场景 |
|------|---------|---------|---------|
| **策略** | 算法族可互换 | 接口+多实现+Context持有 | 多渠道支付、促销折扣、多种登录 |
| **模板方法** | 固定流程+可变步骤 | 抽象类定义流程，子类实现步骤 | 数据导入、支付流程骨架 |
| **观察者** | 一对多事件通知 | 接口/Spring Event | 订单通知、MQ消息、Spring事件 |
| **责任链** | 请求沿链传递处理 | 链表/数组串联Handler | 过滤器链、审批流、参数校验 |
| **命令** | 把请求封装成对象 | Command接口+execute() | 操作日志、撤销/重做、消息队列 |
| **状态** | 状态变化驱动行为变化 | 接口+多状态实现 | 订单状态机、工单流转 |
| **迭代器** | 统一遍历不同集合 | Iterator接口 | Java for-each、集合框架 |
| **中介者** | 对象间通过中介通信 | 中介类持有所有参与者 | 聊天室、MQ作为中介 |
| **备忘录** | 保存/恢复对象状态 | 快照对象 | 草稿箱、撤销操作、游戏存档 |
| **解释器** | 定义语言的文法规则 | 抽象语法树 | SQL解析、正则表达式引擎 |
| **访问者** | 不改类结构添加操作 | Visitor接口+accept() | 编译器AST遍历、报表生成 |

## 三、最常用的8个模式详解（带代码）

### 1. 策略模式 ⭐ —— 你支付系统的核心

**用什么实现：接口 + 多实现类 + Map路由**

```java
// 第一步：定义策略接口
public interface PaymentStrategy {
    PaymentResult pay(PaymentRequest request);
    String getChannel(); // 用于路由的key
}

// 第二步：每个渠道一个实现类
@Component
public class StandardCharteredStrategy implements PaymentStrategy {
    public PaymentResult pay(PaymentRequest request) {
        // 渣打银行API调用逻辑
    }
    public String getChannel() { return "SC"; }
}

@Component
public class AlipayStrategy implements PaymentStrategy {
    public PaymentResult pay(PaymentRequest request) {
        // 支付宝逻辑
    }
    public String getChannel() { return "ALIPAY"; }
}

// 第三步：Context用Map路由（Spring自动注入所有实现）
@Service
public class PaymentService {
    private final Map<String, PaymentStrategy> strategyMap;
    
    // Spring会把所有PaymentStrategy实现注入进来
    public PaymentService(List<PaymentStrategy> strategies) {
        this.strategyMap = strategies.stream()
            .collect(Collectors.toMap(PaymentStrategy::getChannel, s -> s));
    }
    
    public PaymentResult pay(PaymentRequest request) {
        PaymentStrategy strategy = strategyMap.get(request.getChannel());
        if (strategy == null) throw new BusinessException("不支持的支付渠道");
        return strategy.pay(request);
    }
}
```

**新增渠道只需新增一个类，主流程零修改 = 开闭原则**

### 2. 模板方法模式 ⭐ —— 流程固定细节不同

**用什么实现：抽象类（Abstract Class）**

```java
// 抽象类定义骨架
public abstract class AbstractPaymentProcessor {
    
    // final：子类不能覆盖流程顺序！
    public final PaymentResult process(PaymentRequest request) {
        validateParams(request);           // 公共：参数校验
        checkDuplicate(request);           // 公共：幂等检查
        PaymentOrder order = createOrder(request); // 公共：创建订单
        ChannelResult result = callChannel(request); // 抽象：各渠道自己实现
        return buildResult(order, result); // 公共：构建结果
    }
    
    // 公共步骤：抽象类直接实现
    private void validateParams(PaymentRequest request) { ... }
    private void checkDuplicate(PaymentRequest request) { ... }
    
    // 变化步骤：子类必须实现
    protected abstract ChannelResult callChannel(PaymentRequest request);
    
    // 钩子方法：子类可选择性覆盖
    protected boolean needSign() { return true; }
}

// 渣打银行：只需实现自己特有的部分
public class SCPaymentProcessor extends AbstractPaymentProcessor {
    @Override
    protected ChannelResult callChannel(PaymentRequest request) {
        // 只写渣打银行特有逻辑
        return scApiClient.charge(buildSCRequest(request));
    }
}
```

### 4. 责任链模式 ⭐ —— 校验/过滤链

**用什么实现：接口或抽象类 + 链式调用**

```java
// 优惠券兑换前的校验链
public abstract class CouponValidator {
    private CouponValidator next;
    
    public CouponValidator setNext(CouponValidator next) {
        this.next = next;
        return next; // 支持链式设置
    }
    
    public final void validate(ExchangeContext ctx) {
        doValidate(ctx); // 当前节点校验
        if (next != null) next.validate(ctx); // 传递给下一个
    }
    
    protected abstract void doValidate(ExchangeContext ctx);
}

// 各个校验节点
public class ActivityExistValidator extends CouponValidator {
    protected void doValidate(ExchangeContext ctx) {
        if (!activityExists(ctx.getActivityId())) 
            throw new BusinessException("活动不存在");
    }
}

public class StockValidator extends CouponValidator {
    protected void doValidate(ExchangeContext ctx) {
        if (getStock(ctx.getActivityId()) <= 0) 
            throw new BusinessException("库存不足");
    }
}

// 组装链
@Bean
public CouponValidator validatorChain() {
    CouponValidator chain = new ActivityExistValidator();
    chain.setNext(new UserQualificationValidator())
         .setNext(new StockValidator())
         .setNext(new FrequencyLimitValidator());
    return chain;
}
```

### 5. 代理模式 ⭐ —— 面试超高频

**用什么实现：接口（JDK动态代理）或 继承（CGLIB）**

```java
// Spring AOP就是代理模式的工业级应用

// 你每天写的这些注解，背后全是代理：
@Transactional  // 事务代理：方法前开事务，后提交/回滚
@Cacheable      // 缓存代理：先查缓存，没有再执行方法
@PreAuthorize   // 权限代理：方法前检查权限

// JDK动态代理原理（面试手写）
public class LoggingProxy implements InvocationHandler {
    private final Object target;
    
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        log.info("调用方法: {}", method.getName());
        long start = System.currentTimeMillis();
        Object result = method.invoke(target, args); // 调用真实方法
        log.info("耗时: {}ms", System.currentTimeMillis() - start);
        return result;
    }
}

// JDK代理 vs CGLIB代理
// JDK：目标类必须有接口，代理实现同一接口
// CGLIB：目标类可以没接口，代理继承目标类
// Spring Boot 2.x默认：全部用CGLIB
```

### 6. 单例模式

**用什么实现：私有构造 + 静态变量 + 静态方法**

```java
// 方式一：枚举单例（最推荐，线程安全，防反序列化破坏）
public enum PaymentConfig {
    INSTANCE;
    private String apiKey = "xxx";
    public String getApiKey() { return apiKey; }
}

// 方式二：双重检查锁（面试必考）
public class TokenCache {
    // volatile防止指令重排序
    private static volatile TokenCache instance;
    
    private TokenCache() {} // 私有构造
    
    public static TokenCache getInstance() {
        if (instance == null) {          // 第一次检查：不加锁，提高性能
            synchronized (TokenCache.class) {
                if (instance == null) {  // 第二次检查：防并发重复创建
                    instance = new TokenCache();
                }
            }
        }
        return instance;
    }
}

// 方式三：Spring Bean（实际项目最常用）
@Service // 默认scope=singleton，Spring帮你管理单例
public class CouponService { ... }
```

### 7. 工厂模式

**用什么实现：接口 + 工厂类（简单工厂）或 抽象类（工厂方法）**

```java
// 简单工厂（最常用）
public class CouponFactory {
    public static Coupon create(String type) {
        return switch (type) {
            case "DISCOUNT" -> new DiscountCoupon();
            case "CASH"     -> new CashCoupon();
            case "GIFT"     -> new GiftCoupon();
            default -> throw new IllegalArgumentException("未知优惠券类型: " + type);
        };
    }
}

// 工厂方法（可扩展，每种产品有自己的工厂）
public interface CouponFactory {
    Coupon create(CreateCouponRequest request);
}

@Component("DISCOUNT")
public class DiscountCouponFactory implements CouponFactory {
    public Coupon create(CreateCouponRequest request) {
        DiscountCoupon coupon = new DiscountCoupon();
        coupon.setDiscount(request.getDiscount());
        return coupon;
    }
}

// Spring的BeanFactory就是工厂模式的完整实现
```

### 8. 建造者模式

**用什么实现：静态内部Builder类 + 链式方法**

```java
// 手写建造者
public class PaymentRequest {
    private final String orderId;
    private final BigDecimal amount;
    private final String currency;
    private final String channel;
    
    private PaymentRequest(Builder builder) {
        this.orderId = builder.orderId;
        this.amount = builder.amount;
        this.currency = builder.currency;
        this.channel = builder.channel;
    }
    
    public static class Builder {
        private String orderId;
        private BigDecimal amount;
        private String currency = "HKD"; // 默认值
        private String channel;
        
        public Builder orderId(String orderId) { this.orderId = orderId; return this; }
        public Builder amount(BigDecimal amount) { this.amount = amount; return this; }
        public Builder channel(String channel) { this.channel = channel; return this; }
        public PaymentRequest build() {
            // 构建前校验
            Objects.requireNonNull(orderId, "orderId不能为空");
            return new PaymentRequest(this);
        }
    }
}

// 使用（Lombok @Builder帮你生成上面的代码）
PaymentRequest request = PaymentRequest.builder()
    .orderId("ORDER-001")
    .amount(new BigDecimal("100.00"))
    .channel("SC")
    .build();
```

## 四、适配器模式 —— 对接第三方必用

**用什么实现：接口 + 适配器类（持有旧对象）**

你支付系统对接渣打银行就是典型适配器场景：

```java
// 你系统内部统一接口
public interface PaymentGateway {
    PaymentResult charge(PaymentRequest request);
    RefundResult refund(RefundRequest request);
}

// 渣打银行SDK（第三方，你不能改）
public class SCBankSDK {
    public SCResponse doPayment(SCPaymentParam param) { ... }
    public SCResponse doRefund(SCRefundParam param) { ... }
}

// 适配器：把渣打SDK适配成你的接口
@Component
public class SCBankAdapter implements PaymentGateway {
    @Autowired
    private SCBankSDK scSdk; // 持有旧的/第三方对象
    
    public PaymentResult charge(PaymentRequest request) {
        // 转换参数
        SCPaymentParam scParam = convertToSCParam(request);
        // 调用旧接口
        SCResponse response = scSdk.doPayment(scParam);
        // 转换结果
        return convertToResult(response);
    }
}
```

## 五、状态模式 —— 订单状态机必用

**用什么实现：接口 + 每种状态一个实现类**

```java
// 支付订单状态：待支付→支付中→已支付→已退款

public interface OrderState {
    void pay(PaymentOrder order);      // 发起支付
    void confirm(PaymentOrder order);  // 确认支付
    void refund(PaymentOrder order);   // 退款
}

// 待支付状态
public class PendingState implements OrderState {
    public void pay(PaymentOrder order) {
        // 发起支付，切换状态
        order.setState(new ProcessingState());
    }
    public void confirm(PaymentOrder order) {
        throw new IllegalStateException("待支付状态不能直接确认");
    }
    public void refund(PaymentOrder order) {
        throw new IllegalStateException("待支付状态不能退款");
    }
}

// 已支付状态
public class PaidState implements OrderState {
    public void pay(PaymentOrder order) {
        throw new IllegalStateException("已支付不能重复支付");
    }
    public void refund(PaymentOrder order) {
        // 允许退款，切换到退款中
        order.setState(new RefundingState());
    }
}
```

## 六、模式组合使用（真实项目常见）

```
支付系统实际用了多个模式组合：

┌─────────────────────────────────────────┐
│  PaymentController (门面模式)            │
│  统一入口，屏蔽内部复杂性                 │

       └── 更新统计
```

## 七、面试必问问题汇总

### Q1：策略模式和if-else有什么区别，为什么要用？
> if-else违反开闭原则，每次新增渠道都要改核心代码，风险大。
> 策略模式新增渠道只需新增一个类，核心代码零修改。

### Q2：代理模式和装饰器模式区别？
> **代理**：控制访问，代理类主动决定是否调用真实对象（可以不调）
> **装饰器**：增强功能，一定会调用真实对象，只是在前后加料

### Q3：Spring AOP底层用的什么代理？
> - 目标类有接口 → JDK动态代理（实现同一接口）
> - 目标类没接口 → CGLIB（生成子类覆盖方法）
> - Spring Boot 2.x 默认全部用 CGLIB

### Q4：单例模式为什么要volatile？
> 防止**指令重排序**。`new`操作分3步：①分配内存②初始化对象③赋值给变量
> JVM可能重排成①③②，另一个线程拿到了未初始化完的对象
> volatile禁止重排序，保证拿到的一定是完整初始化的对象

### Q5：工厂方法和抽象工厂区别？
> **工厂方法**：一个工厂造一种产品，子类决定造哪种（纵向扩展）
> **抽象工厂**：一个工厂造一族相关产品，比如MySQL工厂造MySQLConnection+MySQLStatement（横向扩展）

### Q6：你项目里用了哪些设计模式？（黄金回答）

> "支付系统中，多渠道支付用了**策略模式**，每个渠道实现PaymentStrategy接口，通过Map按渠道码路由，新增渠道零修改主流程。
> 对接渣打银行SDK用了**适配器模式**，把第三方SDK接口转换成系统内部统一接口。
> 支付流程用了**模板方法模式**，抽象类定义了校验→幂等→创建订单→调渠道→返回结果的固定流程，各渠道只需实现调用那一步。
> 
> 优惠券系统中，兑换前多步校验用了**责任链模式**，兑换成功后的通知（短信/日志/统计）用了**观察者模式**通过Spring ApplicationEvent解耦。
> 
> 两个系统都大量用了**代理模式**（Spring AOP的@Transactional保证事务），对象构建用了**建造者模式**（Lombok @Builder）。"

## 八、一句话速记

| 模式 | 一句话 | 实现基础 |
|------|--------|---------|
| 策略 | 同一件事多种做法，运行时切换 | **接口** + Map |
| 模板方法 | 流程我定，细节你填 | **抽象类** |
| 观察者 | 有事我广播，你们自己处理 | **接口** / Spring Event |
| 责任链 | 一个接一个，谁能处理谁来 | **抽象类** + 链表 |
| 代理 | 我帮你挡着，进门前我加个检查 | **接口**（JDK）/ **继承**（CGLIB）|
| 装饰器 | 叠叠乐，套娃加功能 | **接口** + 组合 |
| 适配器 | 转接头，让不兼容的能用 | **接口** + 包装旧对象 |
| 工厂 | 我要一个XX，怎么造不管 | **接口** / **抽象类** |
| 单例 | 全球只此一家 | 私有构造 + static |
| 建造者 | 一步步拼装复杂对象 | Builder内部类 |
| 状态 | 状态不同，行为不同 | **接口** + 多状态类 |
| 命令 | 把操作打包成对象 | **接口** + execute() |
| 门面 | 一个入口搞定一切 | 普通类封装 |
